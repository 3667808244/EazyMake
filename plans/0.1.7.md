# EazyMake 0.1.7 — 基本国际化（i18n）

---

## 1 设计方案

采用 **编译期字符串索引 + JSON 资源文件** 方案：

- 所有用户可见字符串用 `I18nKey` 枚举值标识（如 `I18nKey::build_success`）
- 每种语言一个 JSON 文件：`locale/en.json`、`locale/zh.json`
- 编译时将 JSON 嵌入二进制（或运行时按需加载），通过 `I18nKey` → 字符串查找
- 语言选择优先级：环境变量 `EZMK_LANG` > 系统语言检测 > 默认 `en`

### 方案对比

| 方案                        | 优点                 | 缺点                                          | 结论            |
| --------------------------- | -------------------- | --------------------------------------------- | --------------- |
| 编译期 JSON 嵌入            | 零文件依赖，分发简单 | 无法热切换语言更新                            | ✅ 采用          |
| 运行时加载 JSON 文件        | 可独立更新翻译       | 需要管理 locale/ 路径，文件缺失风险           | 保留为 fallback |
| gettext (.po/.mo)           | 生态成熟             | 额外工具链（xgettext/msgfmt），Windows 集成差 | ❌ 不采用        |
| 硬编码 `std::unordered_map` | 零开销               | 不可维护，无法由非开发者贡献翻译              | ❌ 不采用        |

**格式约定**：
- key 使用 `snake_case`，与 `I18nKey` 枚举名一致
- 占位符使用 `{name}` 格式，由 `i18n::fmt()` 替换
- `meta.version` 用于兼容性检查：JSON 文件版本与代码期望版本不匹配时 warn 并使用英文 fallback
- 后续可添加更多语言（`locale/de.json`、`locale/ja.json` 等），无需修改 C++ 代码

### 3.2 语言检测逻辑

`i18n::detect_language()` 实现：

1. 检查环境变量 `EZMK_LANG` → 若为 `"zh"` / `"en"` 等合法值则直接返回
2. 在 Windows 上调用 `GetUserDefaultLocaleName()` 获取系统语言
3. 在 Linux/macOS 上检查 `$LANG` / `$LC_ALL` 环境变量
4. 解析出语言代码（`zh-CN` → `zh`、`en-US` → `en`）
5. 检查对应的 `locale/<lang>.json` 是否存在 → 存在则返回，不存在则 fallback `"en"`
6. 全部失败 → 返回 `"en"`

### 3.3 JSON 嵌入方式

两种加载方式的实现：

**编译期嵌入（首选）**：
- 用 Python 脚本将 `locale/*.json` 转为 C++ 字符串常量
- 在 `build.sh` 中添加步骤：`python scripts/embed_locale.py locale/ > src/locale_data.cpp`
- `src/locale_data.cpp` 生成 `std::map<std::string, std::string> embedded_locales` 映射（语言代码 → JSON 文本）
- 优点：二进制自包含，无需外部文件

**运行时加载（fallback）**：
- 在 `get_exe_dir() / "../locale/"` 下查找 `<lang>.json`
- 若嵌入数据和外部文件同时存在，以外部文件为准（允许用户覆盖翻译）
- 加载失败时 fallback 到嵌入的 `en.json`

### 3.4 集成到现有日志系统

修改 `util::info/warn/error/fatal` 使其接受 `I18nKey`：

```cpp
// 新增重载（在 util.hpp 中）
void info(I18nKey key, const std::map<std::string, std::string>& args = {});
void warn(I18nKey key, const std::map<std::string, std::string>& args = {});
void error(I18nKey key, const std::map<std::string, std::string>& args = {});
[[noreturn]] void fatal(I18nKey key, const std::map<std::string, std::string>& args = {});
```

实现层调用 `i18n::fmt(key, args)` 获取本地化字符串后输出。现有的
`util::info(std::string_view)` 重载保留不变，用于非用户可见的调试日志。

**与颜色系统的交互**：
- `util::info(I18nKey)` 内部先调用 `i18n::fmt()` 获取字符串，再应用 `color_msg()` 和前缀
- `cache_hit` 在 verbose 模式下使用了 `util::color_msg(util::color::cyan, ...)` — i18n 集成后保持颜色不变，仅翻译文本
- `util::fatal(I18nKey)` 内部使用红色输出，保持现有的 `[ezmk fatal]` 前缀风格

## 5 注意事项

- JSON 文件中的 key 使用 `snake_case`，与枚举名一致
- 格式化函数使用 `{name}` 命名占位符，避免索引混乱；小型实现基于 `string::find` + `replace`，不引入 fmt 库依赖
- 编译后 `locale/` 目录数据嵌入二进制，外部文件为可选覆盖；加载失败时 fallback 到英文硬编码字符串
- 翻译文件版本通过 `meta.version` 管理：代码期望版本与 JSON 声明的版本不一致时输出警告并使用英文 fallback
- 语言检测仅在 `main()` 中执行一次，结果缓存在全局变量中
- 错误信息（如编译器参数错误、链接失败）中的**技术内容**（如退出码、命令行、编译器输出）不翻译，仅翻译描述性文本
- 安装指引（不同平台的 `pacman` / `apt` / `brew` 命令）保持原样不翻译
- 未来添加新语言仅需创建新的 `locale/<lang>.json`，无需修改 C++ 代码
- **颜色与 i18n 交互**：颜色代码（ANSI escape codes）在 i18n 格式化**之前**包裹文本，不在翻译内容中加入颜色控制字符
- `util::color_msg()` + `util::info(I18nKey)` 组合：`info()` 内部先翻译文本再着色，确保翻译内容与技术格式分离
- `main.cpp` 中的 `"EazyMake 0.1.5"` 应变为 `i18n::fmt(I18nKey::version_output, {{"version", EZMK_VERSION}})`，其中 `EZMK_VERSION` 是编译期宏定义
- 硬编码的 `"[y/N]"` 等交互提示文本不需要 i18n（不同键盘布局下保持统一）
- 纯技术日志（如 `std::to_string(exit_code)`）不翻译，直接拼接到格式化字符串中
