# EazyMake 0.2.3 — 开发者体验提升

---

## 1 背景

0.2.2 完成了编译控制的精细化（可选依赖、语义宏、多源目录），0.2.1 实现了 MSVC 支持，0.2.0 引入了 Lua 工具链。当前 ezmk 已经具备了跨平台、跨编译器的基础构建能力，但在**日常开发体验**上仍有明显短板：

- 编译是**单线程**的——多文件项目构建缓慢
- 没有**debug/release 配置切换**——改优化级别需手动编辑 `ezmk.toml`
- 没有**文件监听/自动重编译**——每次修改后需手动运行 `ezmk project build`
- 没有**编译前后的钩子**——无法执行代码生成、资源拷贝等辅助任务
- 没有**`pkg list` 和 `pkg update`**——包管理缺少基本的查看和更新功能
- **i18n 覆盖率仅 ~60%**——帮助文本、`pkg info`、`repo list` 等高频输出仍为硬编码英文
- 存在若干**bug 和错误处理缺陷**——`list_sources()` 不跟随 `src_dirs`、`cc.lua` 硬编码 `g++`、裸 `catch(...)` 静默吞没错误

本版本聚焦于补齐以上短板，提升日常开发体验和代码质量。

## 2 并行编译（`-j` / `--jobs`）

### 2.1 现状

`build_project()` 中源文件编译是按 `for` 循环顺序逐个执行的。对于 10+ 源文件的项目，编译时间等于各文件编译时间之和，无法利用多核 CPU。

### 2.2 目标行为

新增 `-j <N>` / `--jobs <N>` 标志：

```
ezmk project build -j 8          # 最多 8 个编译任务并行
ezmk project build -j 0           # 0 = 自动检测 CPU 核心数（默认）
ezmk project build                # 不传 -j 时默认 -j 0（自动）
```

**实现方案**：使用 `std::thread` 线程池：
1. 收集需要编译的源文件列表（缓存命中判断可并行）
2. 将源文件编译任务分发给线程池
3. 主线程等待所有编译任务完成
4. 任何编译失败 → 终止剩余的编译任务 → 报错

**行为细节**：
- 编译输出（stdout/stderr）需要加锁或按文件缓冲，避免交错输出
- `--verbose` 模式下，每行编译命令前打印 `[<N>/<total>]` 索引和线程 ID
- 链接步骤始终在主线程执行（在所有编译完成后），不受 `-j` 影响
- 缓存 `record.json` 的写入需要原子化——所有编译完成后一次性写入（而非逐个编译完就写）

**线程安全关注点**：
- `record.json` 的读写：编译前一次性读取缓存记录到内存；编译完成后（所有线程结束）一次性写回
- `util::run_command()` 的调用：每个线程独立调用，临时文件使用唯一命名（`<filename>.<thread_id>.tmp.o`）
- 控制台输出：`util::info/warn/error` 内部加 `std::mutex` 保护

**默认值**：

| `-j` 值 | 含义 |
|---------|------|
| 未指定 | 自动检测（`std::thread::hardware_concurrency()`） |
| `-j 0` | 同上 |
| `-j 1` | 单线程（0.2.2 行为） |
| `-j N` | 最多 N 个并行编译任务 |

## 3 构建配置 Profile（debug / release）

### 3.1 现状

用户需手动在 `compile.flags` 中管理 `-O2` / `-g` 等标志。切换 debug/release 意味着手动编辑 `ezmk.toml`，容易出错且不便于版本控制。

### 3.2 目标行为

新增 `--profile <name>` 标志，允许在 `ezmk.toml` 中预定义多组编译选项：

```
ezmk project build --profile debug
ezmk project build --profile release
ezmk project run --profile release
```

**`ezmk.toml` 配置格式**：

```toml
[compile]
flags = ["-Wall", "-Wextra"]
include_dirs = ["include"]

[compile.profile.debug]
flags = ["-g", "-O0", "-DDEBUG"]
macros = { SANITIZE = "address" }

[compile.profile.release]
flags = ["-O3", "-DNDEBUG", "-flto"]

[link.profile.release]
flags = ["-flto"]
```

**合并规则**：
1. 基础 `[compile]` / `[link]` 中的 flags/macros/include_dirs 等始终生效
2. profile 中的 flags **追加**到基础 flags 之后（后者覆盖前者，符合编译器行为）
3. profile 中的 `macros` **合并**到基础 `[compile.macros]`（profile 优先）
4. profile 中未指定的字段继承基础配置

**默认 profile**：

| 未指定 `--profile` | 行为 |
|---|---|
| 无 `[compile.profile.debug]` 定义 | 使用基础配置，不追加任何 profile flags |
| 有 `[compile.profile.debug]` 定义 | **不自动使用 debug**；必须显式传 `--profile debug` |

理由：不隐式应用 profile，避免用户的 release build 意外带上 debug 标志。用户可在 `build.sh` 或 alias 中设置默认。

**profile 名称校验**：
- 仅允许 `[a-zA-Z0-9_-]+`，不允许空名称
- profile 名称大小写敏感（`Debug` ≠ `debug`）

**与现有 flags 的交互**：
- 基础 `flags` + profile `flags` 的拼接顺序：`g++ <base_flags> <profile_flags> ...`
- 翻译后的标志（MSVC 模式）同样适用上述顺序
- profile 的 `macros` 与基础 `macros` 合并后统一经 `macros_to_flags()` 处理

**`compile.profile.<name>` 支持的字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `flags` | string[] | 追加的编译标志 |
| `msvc_flags` | string[] | MSVC 专用编译标志 |
| `macros` | table | 追加的宏定义（与基础配置合并） |

**`link.profile.<name>` 支持的字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `flags` | string[] | 追加的链接标志 |
| `msvc_flags` | string[] | MSVC 专用链接标志 |

## 4 后置构建钩子（Post-Build Hooks）

### 4.1 现状

构建流程是封闭的：编译 → 链接 → 结束。用户无法在构建后自动执行辅助任务（复制资源文件、生成配置文件、运行代码生成器等），必须手动执行额外步骤。

### 4.2 目标行为

在 `ezmk.toml` 中新增 `[hooks]` 节，支持 Lua 脚本钩子：

```toml
[hooks]
pre_build = "scripts/pre_build.lua"    # 编译前执行（相对项目根目录）
post_build = "scripts/post_build.lua"   # 编译后执行（仅链接成功时）
on_failure = "scripts/on_failure.lua"   # 编译/链接失败时执行（可选）
```

**钩子脚本约定**：

每个钩子脚本是一个独立的 Lua 文件，需实现 `run(ctx)` 函数：

```lua
-- scripts/post_build.lua
function run(ctx)
    -- ctx 是一个 table，包含构建上下文信息
    -- ctx.output: 构建产物路径（string）
    -- ctx.project_root: 项目根目录（string）
    -- ctx.profile: 使用的 profile 名称，可能为空字符串

    -- 示例：复制资源文件到 build 目录
    local assets = ezmk.project_root() .. "/assets"
    local dest = ezmk.build_dir() .. "/assets"
    -- ... 复制逻辑 ...

    ezmk.info("post-build hook completed")
    return 0  -- 返回非零值会使构建报告警告（但不失败）
end
```

**执行约定**：
- 钩子脚本在**独立的 sandbox 环境**中执行（与 `ezmk utils` 一致的隔离策略）
- `ctx` table 由 C++ 端构建并传入
- 脚本可以使用完整的 `ezmk.*` API（22 个函数）
- `run()` 返回非零 → `ezmk warn` 提示钩子返回了非零退出码，构建仍视为成功
- `run()` 中调用 `error()` → 构建失败（如果是 `post_build` 或 `on_failure`），或终止构建（如果是 `pre_build`）
- 脚本执行超时？当前版本不设超时限制（与 `ezmk.run()` 行为一致）

**钩子执行流程**：

```
pre_build.lua（若存在）
    │  error() → 终止构建，不执行编译
    │  return 0 → 继续
    ▼
编译 + 链接
    │  失败 → on_failure.lua（若存在）
    │  成功 → post_build.lua（若存在）
    ▼
构建结束
```

**注意事项**：
- 钩子脚本路径相对于项目根目录
- 钩子脚本不需要是已安装包——放在项目目录下即可
- 若钩子脚本文件不存在 → warn 并跳过（不终止构建，避免配置了钩子但忘记提交脚本文件导致构建失败）
- 钩子脚本仅对**用户项目**生效，不适用于包编译（`compile_package()`）

### 4.3 实施步骤

- [x] 在 `include/ezmk/config.hpp` 中新增 `HooksSection`：
  ```cpp

      std::string pre_build;    // path to pre-build Lua script (relative to project root)

  };
  ```

  - 编译后（成功时）：若 `post_build` 非空 → `lua::run_script(L, post_build, ctx)`

## 5 文件监听 / 自动重编译（Watch 模式）

### 5.1 现状

每次修改源码后需手动执行 `ezmk project build`。对于频繁的"修改 → 编译 → 测试"循环，手动操作打断心流。

### 5.2 目标行为

新增 `ezmk project watch` 子命令，监听项目文件变更并自动触发增量编译：

```
ezmk project watch [--profile <name>] [--no-build-on-start]
```

**行为**：
1. 启动时（默认）执行一次全量构建（可跳过：`--no-build-on-start`）
2. 进入文件监听循环，监听以下目录的文件变更：
   - 所有 `src_dirs` 目录
   - 所有 `include_dirs` 目录
   - `ezmk.toml` 文件本身
3. 检测到文件变更 → 防抖（debounce）300ms → 触发增量构建
4. 构建成功 → 打印 `Build succeeded. Watching for changes...`
5. 构建失败 → 打印错误，继续监听（不退出）
6. `Ctrl+C` → 退出 watch 模式

**跨平台文件监听实现**：
- **Windows**：使用 `ReadDirectoryChangesW` API
- **Linux**：使用 `inotify` API
- **macOS**：使用 `kqueue` / `FSEvents` API

封装为平台无关的 `FileWatcher` 类：

```cpp
namespace ezmk::util {
    class FileWatcher {
    public:
        // callback: called with the changed file path
        using Callback = std::function<void(const fs::path&)>;
        
        FileWatcher(Callback cb, int debounce_ms = 300);
        
        // Add a directory to watch (non-recursive by default)
        void add_directory(const fs::path& dir, bool recursive = true);
        
        // Block until stopped (runs the event loop)
        void run();
        
        // Signal stop from another thread
        void stop();
        
    private:
        // platform-specific implementation details
    };
}
```

**防抖策略**：
- 300ms 窗口内同一文件的多次变更合并为一次触发
- 不同文件的变更在窗口结束时一起触发

**限制**：
- 0.2.3 仅支持**源码目录和头文件目录**的监听，不支持任意路径
- 递归监听深度无限制（受 OS 文件句柄数限制）
- 不支持排除模式（如忽略 `.ezmk/` 下的变化）——ezmk 内部目录本就不在监听范围
- `ezmk.toml` 变更触发全量重编译

### 5.3 实施步骤

- [x] 新增 `include/ezmk/file_watcher.hpp` — 声明 `FileWatcher` 类
- [x] 新增 `src/file_watcher.cpp` — 实现跨平台文件监听：
  - Windows: `ReadDirectoryChangesW` + I/O completion port
  - Linux: `inotify_init` + `inotify_add_watch` + `poll`
  - macOS: `kqueue` + `EVFILT_VNODE`
- [x] 新增 `ezmk::cli::Command::ProjectWatch` 命令枚举值

  - 注册信号处理器（`SIGINT` → stop watcher）
  - 进入 `watcher.run()` 循环

## 6 `pkg list` — 列出已安装包

### 6.1 现状

用户可以 `install`、`remove`、`search`、`info` 包，但**无法列出当前已安装了哪些包**。`search` 需要知道包名（模糊匹配），没有一个"一览表"式的命令来查看所有已安装包。

### 6.2 目标行为

新增 `ezmk pkg list` 子命令：

```
ezmk pkg list [-p|-u|-g]
```

**输出格式**（类似于 `repo list`）：

```
Installed packages (project scope):
  fmt            0.1.0    (library, depends on: std)
  ezmk-cc        0.1.0    (utils, tools: cc)

Installed packages (user scope):
  sqlite3        1.2.3    (library, depends on: std)
  nlohmann_json  3.11.2   (library, depends on: std)

Installed packages (global scope):
  (none)
```

**行为**：
- 默认作用域：`-pug`（显示所有作用域）
- 每个作用域内按包名字典序排列
- 展示：包名、版本、类型、依赖数或工具列表
- 若某作用域无包 → 显示 `(none)`

**实现**：扫描 `pkg_install_dir(scope)` 下的所有子目录，每个子目录即一个已安装包。读取其 `ezmk.toml` 获取元信息。

## 7 `pkg update` — 更新已安装包

### 7.1 现状

要"更新"一个包，用户必须先 `pkg remove` 再 `pkg install`。没有内建的更新机制，也没有版本比较或锁定文件。

### 7.2 目标行为

新增 `ezmk pkg update` 子命令（0.2.3 最小可行版本）：

```
ezmk pkg update [-p|-u|-g] <pkg_name>
```

**行为**：
1. 在指定作用域中查找已安装的 `<pkg_name>`
2. 在所有已注册仓库中搜索同名包的最新版本
3. 比较仓库最新版本 vs 已安装版本：
   - 仓库版本 > 已安装版本 → 下载 + 编译 + 替换安装
   - 仓库版本 = 已安装版本 → 提示 "already up-to-date"
   - 仓库中找不到更新的版本 → 提示 "no updates available"
4. 版本比较使用简单的**字符串比较**（不做 SemVer 解析，与仓库索引中"取最高版本号"的逻辑一致）

**限制**（0.2.3）：
- 不支持 `pkg update --all`（更新全部包）
- 不维护 lockfile
- 版本比较不做 SemVer 语义分析（`"2.0.0"` > `"1.10.0"` 在字符串比较中不成立，但在实际使用中通过仓库的 `index.toml` 最新版本通常已是用户想要的版本）

**设计理由**：`pkg update` 的主要价值在于"无需先 remove 再 install"，而非精确的版本管理。版本锁定和 lockfile 是未来版本的关注点。

### 7.3 实施步骤

- [x] 在 `include/ezmk/pkg.hpp` 中新增 `update()` 声明
- [x] 在 `src/pkg.cpp` 中实现 `update()`：
  - 在指定 scope 中查找已安装包 → 读取版本号
  - 调用 `repo::search_package()` 在所有仓库中查找最新版本
  - 比较版本号 → 决定是否更新
  - 下载新版本 → 移除旧版本 → 安装新版本
- [x] 在 `include/ezmk/cli.hpp` 中新增 `Command::PkgUpdate` 枚举值

## 11 特性交互关系

### 11.1 Profile × 并行编译

- `--profile debug -j 8`：使用 debug profile + 8 线程并行
- Profile 不影响 `-j` 参数；两者正交

### 11.2 Profile × Post-Build Hooks

- `ctx.profile` 字段告知钩子脚本当前使用的 profile 名称
- 钩子脚本可以根据 profile 决定行为（如 release 构建时压缩资源）

### 11.3 Watch × Profile

- `ezmk project watch --profile debug`：watch 模式下每次触发均使用 debug profile
- Watch 模式下的构建与手动 `build` 使用相同的缓存逻辑

### 11.4 Watch × 并行编译

- Watch 触发的构建也受益于 `-j` 并行编译
- `ezmk project watch -j 8`：watch + 8 线程

### 11.5 Post-Build Hooks × Lua API

- 钩子脚本有完整的 `ezmk.*` API 访问权限
- 可以在钩子中调用 `ezmk.run()` 执行外部命令（如 `strip`、压缩资源）

## 12 注意事项

- **并行编译的缓存安全**：所有线程完成编译后，主线程一次性写回 `record.json`。若中途某线程失败，已成功编译的目标文件（`.o`）保留（下次可复用），但 `record.json` 不更新（保持上次的一致性状态）
- **并行编译输出**：`--verbose` 模式下，`[<N>/<total>]` 编号在编译开始前分配，完成后输出结果（成功/失败）。输出顺序不保证与源文件顺序一致
- **Profile 合并的覆盖语义**：profile flags 追加到 base flags 之后；profile macros 覆盖 base 中的同名 macro。这与 GCC/Clang 的"后覆盖前"语义一致
- **Watch 模式下的 `ezmk.toml` 变更**：检测到 `ezmk.toml` 修改后，重新解析配置，清空全量缓存（相当于 `--disable-cache` 行为），全量重编译一次
- **Watch 模式的退出**：Windows 上 `Ctrl+C` 正常停止；Linux/macOS 上 `SIGINT` 被捕获后优雅退出
- **Hook 脚本的路径**：相对于项目根目录；若脚本不存在 → warn + 跳过（非致命），考虑到了团队协作中某人配置了钩子但其他人未创建脚本文件的场景
- **Hook 脚本的安全性**：与 `ezmk utils` 共享相同的 sandbox 安全模型（`os.execute`/`io.popen` 不可用，文件写入限制在项目根目录内）
- **跨平台 Watch 实现**：优先使用各平台最优 API（Windows: `ReadDirectoryChangesW`，Linux: `inotify`，macOS: `kqueue`），不引入第三方库
- **Watch 递归监听**：默认为递归监听（`recursive = true`），对深层目录结构友好
- **`--no-build-on-start` 的使用场景**：在 CI 中先 `build` 再 `watch` 是不合理的——watch 仅用于本地开发场景。此 flag 主要用于用户启动 watch 时变更已在构建中
- **向后兼容**：所有新增字段均为可选。不配置 `[hooks]`、`[compile.profile.*]` 的 `ezmk.toml` 行为与 0.2.2 完全一致。`-j` 未指定时默认自动检测核心数
- **`project build` 和 `project run` 新增的 `--profile` 和 `-j` flags**：向后兼容，不加 flags 时行为与 0.2.2 完全一致

## 14 开放问题

1. **并行编译的依赖顺序**：当前设计不保证源文件的编译顺序。如果项目中有文件间的编译依赖（如某文件 `#include` 了另一个源文件生成的代码），应通过 post-build hook 或代码生成工具处理，而非依赖编译顺序。这是否合理？
2. **Watch 模式排除模式**：是否需要支持 `.ezmkignore` 或类似机制排除特定文件/目录？短期可不加——watch 仅监听 src_dirs 和 include_dirs，ezmk 的内部目录不在范围内。
3. **Profile 继承**：是否需要 profile 之间的继承关系（如 `release = base + lto`）？短期采用简单覆盖模型，base + profile 合并。
4. **Watch 的网络文件系统**：NFS/Samba 等网络文件系统可能不支持 inotify。是否需要回退到轮询模式？短期不做——实际使用中项目源码通常在本地磁盘。
