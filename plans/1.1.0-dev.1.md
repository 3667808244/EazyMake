# EazyMake 1.1.0-dev.1 — MSVC 包编译、确定性构建与产物安装

---

## 1 背景

1.0.0 完成了文档整理与翻译补全，EazyMake 作为首个正式版发布。但经过代码审查与使用场景分析，识别出三个影响实际使用的工程缺陷：

### 1.1 MSVC 包编译问题

**严重性：高。** 当前 MSVC 工具链只在用户项目编译层面可用（`cache.cpp` 中 `compile_one_source()` 对 MSVC 有完整分支），但**包编译**（`compile_package()`）完全硬编码 GCC 工具链：

```cpp
// pkg.cpp:167 — 永远输出 .a，无论工具链
fs::path lib = build_dir / ("lib" + name + ".a");

// pkg.cpp:237 — 永远调用 ar，无论平台
ar_cmd << "ar rcs \"" << lib_tmp.string() << "\"";
```

这意味着：
- 使用 MSVC 工具链的用户无法通过 `ezmk pkg install` 从源码编译安装包
- 默认仓库中预编译包仅提供 GCC `.a` 文件，MSVC 项目无法链接
- `index.toml` 的平台二进制映射（`[platform]`）缺少 MSVC 条目
- 随着 Windows 用户更多地使用 MSVC（而非 MSYS2/MinGW），此问题日益突出

此外，0.9.8 计划中提出的 `header_only = true` 支持（`pkg.toml` 字段 + 跳过编译逻辑）尚未实现，这反过来加重了 MSVC 包的编译压力——header-only 库本不需要编译，却没有机制跳过。

### 1.2 编译确定性问题

**严重性：中。** 构建产物不可复现，阻碍了安全审计、缓存共享（CI/CD）、增量构建验证等场景：

- **时间戳嵌入**：`__DATE__` / `__TIME__` 宏每次编译产生不同二进制
- **文件路径嵌入**：调试信息（DWARF / PDB）中包含绝对路径，不同机器产生不同 `.o`
- **编译器版本未追踪**：`record.json` 仅记录 `compiler` 名称（`"g++"` / `"cl.exe"`），不记录版本号。相同源码 + 相同 flags 但编译器版本不同 → 缓存误命中 → 链接失败或运行时错误
- **暂无 `SOURCE_DATE_EPOCH` 支持**：GCC/Clang 的 `SOURCE_DATE_EPOCH` 和 MSVC 的 `/Brepro` 均未启用

### 1.3 构建产物不易安装

**严重性：中。** 当前构建产物仅停留在项目 `build/` 目录下：

- **无 `install` 命令**：没有 `ezmk project install`，用户需手动复制文件
- **无安装路径配置**：没有 `[install]` 配置节，无法声明安装布局（bin/lib/include）
- **手动复制易出错**：遗漏依赖的 DLL、忘记复制头文件、路径拼写错误
- **跨项目复用困难**：用户无法将本地项目作为"库"输出安装到系统/用户目录供其他项目引用
- **与 `pkg install` 割裂**：包系统有完整的安装流程（作用域、钩子、版本管理），但用户自己的项目完全享受不到这些能力

---

## 2 目标

1. **MSVC 包编译** — `compile_package()` 支持 MSVC 工具链（`lib.exe` 生成 `.lib`），`index.toml` 平台映射支持 MSVC 预编译二进制，安装时根据工具链自动选择正确的二进制格式
2. **header-only 包支持** — 实现 0.9.8 遗留的 `header_only = true` 字段，跳过编译步骤（减少 MSVC 包编译压力）
3. **确定性构建** — 支持 `SOURCE_DATE_EPOCH` + GCC `-ffile-prefix-map` + MSVC `/Brepro`，`record.json` 记录编译器版本，`ezmk.lock` 锁定依赖版本与内容哈希
4. **`ezmk project install`** — 新增安装命令，支持 `[install]` 配置节（bin/lib/include 布局），构建产物一键安装到指定前缀

---

## 3 详细设计

### 3.1 MSVC 包编译（`compile_package()` 工具链感知）

#### 3.1.1 现状分析

`compile_package()`（`src/pkg.cpp:154-260`）的编译阶段通过 `cache::compile_sources()` 完成，该函数**已支持 MSVC**（检测 `in.tc.family` 分支，`src/cache.cpp:307-519`）。问题出在**归档阶段**（`pkg.cpp:221-260`），完全硬编码 GCC：

```cpp
// 现状：仅 GCC 路径
fs::path lib = build_dir / ("lib" + name + ".a");          // 永远是 .a
ar_cmd << "ar rcs \"" << lib_tmp.string() << "\"";          // 永远是 ar
for (auto& o : comp_result.objects) {
    ar_cmd << " \"" << o.string() << "\"";
}
auto ar_res = util::run_command(ar_cmd.str());
```

#### 3.1.2 设计方案

修改 `compile_package()` 签名，新增 `Toolchain` 参数，内部根据工具链选择归档器：

```cpp
// 新签名
fs::path compile_package(const fs::path& pkg_dir,
                         const std::vector<fs::path>& dep_includes,
                         const toolchain::Toolchain& tc);   // 新增
```

**编译阶段**（`cache::compile_sources()`）：
- `CompileInput` 已包含 `tc` 字段（`include/ezmk/cache.hpp`），无需额外改动
- MSVC 编译时 `.obj` 后缀、`/showIncludes` 依赖解析、标志翻译均已工作

**归档阶段**（新增 MSVC 分支）：

```cpp
bool is_msvc = (tc.family == toolchain::CompilerFamily::Msvc);
const char* lib_ext = is_msvc ? ".lib" : ".a";
const char* tmp_ext = is_msvc ? ".lib.tmp" : ".a.tmp";

fs::path lib = build_dir / ("lib" + name + lib_ext);
fs::path lib_tmp = build_dir / ("lib" + name + tmp_ext);

if (is_msvc) {
    // MSVC: 使用 lib.exe 创建静态库
    std::ostringstream lib_cmd;
    lib_cmd << "lib.exe /OUT:\"" << lib_tmp.string() << "\"";
    for (auto& o : comp_result.objects) {
        lib_cmd << " \"" << o.string() << "\"";
    }
    auto lib_res = util::run_command(lib_cmd.str());
    // ... 错误处理
} else {
    // GCC/Clang: 使用 ar（现有逻辑）
    std::ostringstream ar_cmd;
    ar_cmd << "ar rcs \"" << lib_tmp.string() << "\"";
    for (auto& o : comp_result.objects) {
        ar_cmd << " \"" << o.string() << "\"";
    }
    auto ar_res = util::run_command(ar_cmd.str());
    // ... 错误处理
}
```

**原子写入**：沿用现有的 temp → rename 模式。

#### 3.1.3 `ezmk.toml` 解析适配

`config::parse_config()` 需要将检测到的工具链传递给 `compile_package()`。当前的调用链：

```
pkg::install() → pkg::resolve_dependencies() → pkg::compile_package()
```

`install()` 函数（`src/pkg.cpp`）需要在早期检测工具链，传递到编译阶段。

#### 3.1.4 `index.toml` 平台映射扩展

当前 `index.toml` 的 `[platform]` 节使用 `os_arch` 格式映射预编译二进制：

```toml
# 现状：仅有 GCC 二进制
[platform]
"windows_x86_64"  = "1.3.1/windows_x86_64/"
"linux_x86_64"    = "1.3.1/linux_x86_64/"
"darwin_x86_64"   = "1.3.1/darwin_x86_64/"
```

扩展为 `os_arch_toolchain` 格式，新增 MSVC 条目：

```toml
# 新格式：os_arch_toolchain
[platform]
"windows_x86_64_gcc"   = "1.3.1/windows_x86_64_gcc/"
"windows_x86_64_msvc"  = "1.3.1/windows_x86_64_msvc/"
"linux_x86_64_gcc"     = "1.3.1/linux_x86_64_gcc/"
"darwin_x86_64_clang"  = "1.3.1/darwin_x86_64_clang/"
```

**向后兼容**：旧格式 `"windows_x86_64"` 映射到 `"windows_x86_64_gcc"`（GCC 为 Windows 默认），兼容现有所有默认仓库包。

**解析逻辑**（`src/repo.cpp` 的 `find_package_archive()` 或等效函数）：
1. 优先匹配精确三元组：`{os}_{arch}_{toolchain}`
2. Fallback：`{os}_{arch}`（旧格式，映射到 GCC）
3. 若都不存在 → 报错提示

**工具链标签**：
| 工具链      | 标签    |
| ----------- | ------- |
| GCC (MinGW) | `gcc`   |
| MSVC        | `msvc`  |
| Clang       | `clang` |

```toml
# pkg.toml

```

#### 3.2.2 安装流程适配（`src/pkg.cpp`）

在 `install()` 或 `compile_package()` 调用之前检查：

```cpp
if (pkg_cfg.header_only) {
```

### 3.3 确定性构建

#### 3.3.1 `[compile]` 新增 `deterministic` 选项

```toml
[compile]
deterministic = true   # 默认 false
```

#### 3.3.2 GCC/Clang 实现

当 `deterministic = true` 时，自动注入以下 flags：

| Flag                              | 作用                                   |
| --------------------------------- | -------------------------------------- |
| `-ffile-prefix-map=<proj_root>=.` | 将绝对路径替换为相对路径（调试信息中） |
| `-frandom-seed=<src_filename>`    | 每次编译同一文件产生相同随机数         |
| 设置 `SOURCE_DATE_EPOCH` 环境变量 | 固化 `__DATE__` / `__TIME__` 宏值      |

`SOURCE_DATE_EPOCH` 取值优先级：
1. 环境变量 `$SOURCE_DATE_EPOCH`（用户显式设置）
2. `ezmk.toml` 的 `compile.source_date_epoch`（项目配置）
3. 当前 HEAD commit 的时间戳（`git log -1 --format=%ct`）
4. `ezmk.toml` 的修改时间（fallback）

```toml
[compile]
deterministic = true
source_date_epoch = 1700000000   # 可选：显式指定 Unix 时间戳
```

#### 3.3.3 MSVC 实现

当 `deterministic = true` 时，自动注入：

| Flag                              | 作用                                   |
| --------------------------------- | -------------------------------------- |
| `/Brepro`                         | 移除二进制中的绝对路径和时间戳         |
| 设置 `SOURCE_DATE_EPOCH` 环境变量 | 同上（MSVC 19.32+/VS 2022 17.2+ 支持） |

这些 flags 在 `cache.cpp` 的 `compile_one_source()` 中注入，通过 `translate_compile_flags()` 的现有机制（MSVC 风格的 `/Brepro` 直接透传，GCC 风格的 `-ffile-prefix-map` 在 MSVC 模式下忽略并 warn）。

#### 3.3.4 `record.json` 记录编译器版本

当前 `record.json` 格式：

```json
{
  "version": 1,
  "compiler": "cl.exe",
  "compile_options_signature": "abc123...",
  "records": { ... }
}
```

扩展为：

```json
{
  "version": 2,
  "compiler": "cl.exe",
  "compiler_version": "19.42.34436 for x64",    ← 新增
  "compile_options_signature": "abc123...",
  "deterministic": false,                         ← 新增
  "records": { ... }
}
```

`compiler_version` 通过 `run_command("g++ --version")` / `run_command("cl 2>&1")` 的首行输出来获取。

**缓存失效规则**：当 `compiler_version` 变化时，清除所有 record 条目（清空 `files` map）。

`compile_options_signature` 的计算**纳入** `deterministic` 标志——开关确定性构建会触发全量重编译（不同 flags → 不同输出）。

#### 3.3.5 Lockfile（`ezmk.lock`）— 依赖版本与内容锁定

> **背景**：`ezmk.toml` 中的 `[depends]` 声明了版本约束（如 `fmt@10.2.1` / `spdlog^1.14`），但依赖解析的结果可能因仓库状态（新增版本、索引更新）而在不同时间/机器上产生不同结果。例如 `spdlog^1.14` 今天解析到 `1.14.1`，下个月可能解析到 `1.15.0`——构建不再可复现。
>
> Lockfile 将依赖解析的**精确结果**固化到文件中，提交到版本控制。与编译器层面的确定性 flags（§3.3.1–§3.3.4）配合，"确定性构建"承诺才能完整兑现：**相同源码 + 相同编译器 + 相同依赖 = 逐字节相同的产物**。

##### 3.3.5.1 文件格式（TOML）

```toml
# ezmk.lock — AUTO-GENERATED by ezmk. Do not edit manually.
# Generated at: 2026-07-17T15:30:00+08:00
# Generated by: ezmk 1.1.0 (MSVC 19.42)

[metadata]
version = 1                              # lockfile 格式版本（用于未来迁移）
generated_by = "ezmk 1.1.0"
generated_at = "2026-07-17T15:30:00+08:00"
toolchain = "msvc"                       # 生成时使用的工具链
toolchain_version = "19.42.34436"

# 每个依赖包（包括传递依赖）
[[packages]]
name = "fmt"
version = "10.2.1"
source = "official"                      # 来自哪个仓库（ezmk repo 名称）
source_url = "https://github.com/ezmk-official/packages"
sha256 = "a1b2c3d4e5f6..."               # 已安装产物的 SHA-256（.a / .lib 或头文件合集）
type = "static"                          # static / shared / header-only
scope = "user"                           # 安装作用域：project / user / global
platform = "windows_x86_64_msvc"         # 选取的预编译平台
dependencies = []                        # 自身依赖（来自 index.toml，传递闭包）

[[packages]]
name = "spdlog"
version = "1.14.1"
source = "official"
source_url = "https://github.com/ezmk-official/packages"
sha256 = "b2c3d4e5f6a1..."
type = "static"
scope = "user"
platform = "windows_x86_64_msvc"
dependencies = ["fmt"]
```

**设计要点**：

- **`sha256` 是安装产物的哈希**：对于 static/shared 包，哈希 `.a`/`.lib`/`.dll` 文件；对于 header-only 包，哈希整个 `include/` 目录的 tar 流。这与确定性构建直接关联——依赖内容变化会触发缓存失效和重编译。
- **`platform` 记录选用的预编译平台**：来自 `index.toml` 的平台映射解析结果。不同平台的 lockfile 内容不同（Mac 选 `darwin_x86_64_clang`，Windows 选 `windows_x86_64_msvc`），因此 lockfile **按平台生成，跨平台项目需各自维护**（或 CI 矩阵中生成）。
- **`dependencies` 是声明的直接依赖**（来自 `index.toml` 的 `[depends].lib`），不重复展开传递闭包（闭包通过所有 `[[packages]]` 条目天然形成）。
- **格式版本 `version`**：未来 lockfile 结构变化时用于自动迁移（类似 `record.json` 的 `version` 字段）。

##### 3.3.5.2 生成与更新时机

| 命令                         | 行为                                                                                                                                              |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ezmk pkg install`           | 解析依赖 → 安装 → 生成/更新 `ezmk.lock`（整个传递闭包）                                                                                           |
| `ezmk pkg install --locked`  | 锁模式：仅使用 lockfile 中的 `version` + `sha256` 安装，不重新解析。若 lockfile 与 `ezmk.toml` 不一致（新增/删除/修改了 dep）则**报错退出**       |
| `ezmk pkg install --no-lock` | 不生成/更新 lockfile（一次性安装，不固化）                                                                                                        |
| `ezmk pkg update <name>`     | 更新指定包及其在 lockfile 中的条目（重新解析版本约束）                                                                                            |
| `ezmk pkg update`            | 全部更新：重新解析所有依赖 → 安装最新兼容版本 → 重写 lockfile                                                                                     |
| `ezmk project build`         | 仅读取 lockfile 做**完整性校验**（不修改）；若 lockfile 存在但某个包的 `sha256` 不匹配 → **warn**（默认）/ **error**（当 `deterministic = true`） |

##### 3.3.5.3 与确定性构建的集成

这是 lockfile 与 §3.3.1–§3.3.4 的耦合点：

1. **`deterministic = true` 时，lockfile 从"建议"变为"强制"**：
   - `ezmk project build` 若发现 `ezmk.lock` 缺失 → **error**（拒绝构建）
   - 若发现任一 `[[packages]]` 的 `sha256` 与实际安装产物不匹配 → **error**
   - 若 lockfile 的 `toolchain`/`toolchain_version` 与当前工具链不同 → **warn**

2. **lockfile 摘要参与编译缓存键**：
   ```cpp
   // cache.cpp: compile_options_signature() 中
   if (deterministic) {
       // 将 lockfile 内容的 SHA-256 纳入编译选项签名
       std::string lock_hash = crypto::sha256_file(proj_root / "ezmk.lock");
       hasher.update(lock_hash);
   }
   ```
   这确保：即使源码和 flags 不变，依赖内容一旦变化（通过 lockfile 的 sha256 字段反映），缓存也会失效触发重编译。

3. **lockfile 记录工具链信息**：`metadata.toolchain` + `metadata.toolchain_version` 提供额外的审计线索——当 CI 和开发者本地产生不同产物时，可快速比对工具链差异。

##### 3.3.5.4 生命周期与工作流

**开发阶段**（lockfile 未提交）：
```
$ ezmk pkg install fmt@10.2.1
[ezmk] Resolving dependencies...
[ezmk] Installing fmt 10.2.1 (official)...
[ezmk] Installed fmt 10.2.1 (static, windows_x86_64_msvc, 450 KB)
[ezmk] Generated ezmk.lock (1 package)

$ git add ezmk.lock && git commit -m "add fmt 10.2.1"
```

**CI / 队友克隆后**（lockfile 已提交）：
```
$ git clone ...
$ ezmk project build
[ezmk] Verifying dependency integrity against ezmk.lock...
[ezmk]   fmt@10.2.1 sha256 OK
[ezmk] All dependencies verified (1/1)
[ezmk] Building myapp 0.1.0...
```

**依赖变更**（有人修改了 `ezmk.toml`）：
```
# ezmk.toml: fmt@10.2.1 → fmt@11.0.0
$ ezmk project build
[ezmk] ezmk.toml depends have changed since ezmk.lock was generated.
       Run 'ezmk pkg install' to update the lockfile.
$ ezmk pkg install
[ezmk] fmt 10.2.1 → 11.0.0 (updated)
[ezmk] Updated ezmk.lock
```

**CI 锁模式**（确保 CI 不意外变更依赖）：
```
$ ezmk pkg install --locked
[ezmk] Installing from ezmk.lock...
[ezmk] fmt@10.2.1 sha256 OK
[ezmk] All packages verified and installed (1/1)
# 若 ezmk.toml 与 lockfile 不一致 → 直接报错退出（非零 exit code）
```

##### 3.3.5.5 文件位置

- 项目根目录：`<project_root>/ezmk.lock`
- `.gitignore` 中**不应**排除 lockfile（对于应用程序项目）；库项目（`type = "static"` / `"shared"`）可自行决定
- 项目创建时（`ezmk project new`）不生成；首次 `ezmk pkg install` 时生成

##### 3.3.5.6 数据结构

```cpp
// include/ezmk/config.hpp 新增

struct LockedPackage {
    std::string name;
    std::string version;
    std::string source;          // repo name
    std::string source_url;
    std::string sha256;          // SHA-256 of the installed artifact
    std::string type;            // "static" / "shared" / "header-only"
    std::string scope;           // "project" / "user" / "global"
    std::string platform;        // e.g. "windows_x86_64_msvc"
    std::vector<std::string> dependencies;  // direct deps (from index.toml)
};

struct Lockfile {
    int version = 1;
    std::string generated_by;
    std::string generated_at;
    std::string toolchain;       // "gcc" / "clang" / "msvc"
    std::string toolchain_version;
    std::vector<LockedPackage> packages;
};
```

##### 3.3.5.7 解析与写入

新增 `src/lockfile.cpp` + `include/ezmk/lockfile.hpp`：

```cpp
namespace ezmk::lockfile {

// 从项目根目录读取 ezmk.lock，不存在返回 std::nullopt
std::optional<Lockfile> load(const fs::path& proj_root);

// 将依赖解析结果写入 ezmk.lock
void save(const fs::path& proj_root, const Lockfile& lf);

// 校验已安装包与 lockfile 的一致性。返回不匹配的包名列表。
// 空列表 = 全部一致。
std::vector<std::string> verify(const fs::path& proj_root,
                                const Lockfile& lf);

// 比较 ezmk.toml 的 [depends] 与 lockfile 的 packages，
// 返回是否有新增/删除/修改的依赖（用于 --locked 模式）
bool depends_changed(const config::EzConfig& cfg,
                     const Lockfile& lf);

} // namespace ezmk::lockfile
```

### 3.4 `ezmk project install` 命令

#### 3.4.1 `[install]` 配置节

```toml
[install]
prefix = "/usr/local"             # 安装根目录（默认：Unix ~/.local, Windows %LOCALAPPDATA%\ezmk）
bindir = "bin"                    # 可执行文件子目录（默认："bin"）
libdir = "lib"                    # 库文件子目录（默认："lib"）
includedir = "include"            # 头文件子目录（默认："include"）
sharedir = "share"                # 数据文件子目录（默认："share"）
# install_headers = ["include/"]   # 待安装的头文件 glob（默认：include_dirs）
# install_data = ["assets/"]       # 待安装的数据文件 glob（默认：空）
```

- `prefix` 支持 `~` 展开（`~/bin` → `/home/user/bin`）
- 各 `*dir` 可以是绝对路径或相对于 `prefix` 的路径
- 默认 `prefix`：Unix `$HOME/.local`，Windows `%LOCALAPPDATA%\ezmk`

#### 3.4.2 CLI 接口

```
ezmk project install [flags] [--prefix <path>]
```

| Flag               | 说明                         |
| ------------------ | ---------------------------- |
| `--prefix <path>`  | 覆盖 `[install].prefix`      |
| `--dry-run`        | 仅显示将安装什么，不实际写入 |
| `--no-headers`     | 跳过安装头文件               |
| `--no-data`        | 跳过安装数据文件             |
| `-v` / `--verbose` | 显示每个安装的文件路径       |

**简写**：`ezmk pi`（project install）。

#### 3.4.3 安装流程

```
1. ezmk project build（若尚未构建或源码有变更）
   ↓
2. 解析 [install] 配置 → 确定目标目录
   ↓
3. 创建目标目录结构
   ↓
4. 复制产物：
   - type="executable" → <bindir>/<name>.exe
   - type="static"     → <libdir>/lib<name>.a
   - type="shared"     → <bindir>/<name>.dll + <libdir>/<name>.lib (import lib)
   - type="utils"      → 不安装（工具通过 ezmk utils 调用）
   ↓
5. 复制头文件：<includedir>/<name>/
   ↓
6. 复制数据文件（如有配置）
   ↓
7. 输出安装摘要
```

#### 3.4.4 安装示例

```toml
# ezmk.toml
[project]
name = "mylib"
type = "static"
version = "1.0.0"

[install]
prefix = "~/.local"
```

```
$ ezmk project install
[ezmk] Building mylib 1.0.0...
[ezmk] [3/3] src/foo.cpp
[ezmk] Build succeeded (0 cached, 3 compiled, 1.2s)
[ezmk] Installing mylib 1.0.0 to ~/.local...
[ezmk]   lib     → ~/.local/lib/libmylib.a
[ezmk]   include → ~/.local/include/mylib/
[ezmk]   include → ~/.local/include/mylib/mylib.h
[ezmk]   include → ~/.local/include/mylib/foo.h
[ezmk] Installed 1 library + 2 headers (3 files, 450 KB)
```

#### 3.4.5 与 `pkg install` 的对比

| 维度     | `pkg install`                       | `project install`                           |
| -------- | ----------------------------------- | ------------------------------------------- |
| 来源     | 远程仓库 / 本地文件 / URL           | 当前项目的 `build/` 产物                    |
| 触发     | 用户手动                            | 用户手动                                    |
| 目标     | `.ezmk/pkg/`（三作用域）            | `[install].prefix`                          |
| 钩子     | `preinstall` / `postinstall`（Lua） | 复用构建钩子 `post_build`（在 link 后执行） |
| 版本管理 | 包版本 + 依赖解析                   | 无（安装即覆盖）                            |

## 5 兼容性矩阵

| 变更                                      | 影响                                              | 处理                                                              |
| ----------------------------------------- | ------------------------------------------------- | ----------------------------------------------------------------- |
| `compile_package()` 新增 `Toolchain` 参数 | 所有调用方需更新                                  | 调用方仅限于 `pkg.cpp` 内部（`install()` / `update()`），影响面小 |

| 新增 `ezmk.lock`                          | 没有 lockfile 的项目行为不变                      | `deterministic = false` 时 lockfile 缺失仅跳过校验，不影响构建    |

| 新增 `[install]` 配置节                   | 不声明 `[install]` 的项目行为不变                 | 纯增量                                                            |
| `header_only = true` 跳过编译             | 旧包（未声明）行为不变                            | 纯增量                                                            |

## 6 跨版本关注点

- **依赖**: 依赖 1.0.0（文档整理完成，正式版发布）。功能上依赖 0.2.1（MSVC 工具链检测与编译基础设施）和 0.9.8（header-only 包设计已在 0.9.8 计划中明确）
- **向后兼容**: 
  - MSVC 包编译：旧 `.a` 包在 MSVC 项目中间接不兼容（使用 GNU 格式），但用户项目行为不变
  - `index.toml` 旧格式继续工作
  - `record.json` v1 自动迁移到 v2
  - 未配置 `[install]` 的项目不受影响
- **SemVer 合规**: 新增功能（`project install`、确定性构建、header-only 支持），符合 1.0.0 → 1.1.0 的次版本升级
- **平台注意事项**:
  - MSVC 包编译仅在 Windows 上可用（`lib.exe` 路径依赖 MSVC 环境）
  - Linux/macOS 上的确定性构建使用 `-ffile-prefix-map`（GCC 8+/Clang 10+）
  - `SOURCE_DATE_EPOCH` 的 git commit 时间戳取值仅在 git 仓库内有效
- **延后项**:
  - MSVC 预编译包的上传到默认仓库（本版本实现工具链能力，包上传是运维操作）
  - 共享库（DLL/SO）的安装后运行时搜索路径（RPATH / `PATH` 配置），留待 1.2.0
  - `ezmk project install` 的卸载命令（`ezmk project uninstall`），留待 1.2.0
  - lockfile 的跨平台合并策略（同一项目在 Windows/Linux/macOS 各有一份 `ezmk.lock`，可考虑 `ezmk.lock.d/` 按平台拆分），留待 1.2.0
  - lockfile 的 `ezmk pkg verify` 独立校验命令（当前在 `project build` 中隐式校验），留待用户反馈后决定
