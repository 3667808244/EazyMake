## 1. 集成 Catch2 并跑通基础单元测试

### 1.1 搭建测试基础设施

- [x] 创建 `test/` 目录，与 `src/` 平级

- [x] 创建 `test/test_main.cpp` — Catch2 入口：

### 1.3 注意事项

- 测试应独立于外部状态：不依赖当前工作目录存在 `ezmk.toml` 或特定项目结构
- 涉及文件系统的测试使用 `std::filesystem::temp_directory_path()` 创建临时目录，并在 tear-down 中清理
- 编译器和 git 相关测试需先检查工具可用性（`git --version` / `g++ --version`），不可用时 skip（使用 Catch2 `SKIP` 宏）
- Catch2 使用 `TEST_CASE` + `SECTION` 风格，目标覆盖率：关键路径 > 80%
- `test/test_util.cpp` 中的 `run_command` 测试需跨平台兼容（`echo` 在不同 shell 行为一致）
- `test/test_cli.cpp` 不实际调用 `main()`，而是直接测试 `cli::parse()` 函数，检查返回的 `CliArgs` 结构体
- `test/test_build.cpp` 中的 `find_compiler()` 测试在 Windows 上验证追加 `.exe`，在 Linux 上验证不追加
- `src/main.cpp`（含 `main()` 函数）不应编译到测试二进制中——通过文件列表排除或使用 `-Dmain=disabled_main` 技巧
- `test/test_pkg.cpp` 和 `test/test_repo.cpp` 中涉及 `compile_package()` 的测试需要 g++ 可用
- 所有测试文件统一使用 `#include "catch2.hpp"` 以及对应的 `ezmk/*.hpp` 头文件

## 2. 补全集成测试场景

在 `test/test_integration.cpp` 中以脚本化方式模拟用户操作流程：

### 2.1 项目生命周期

- [x] `project new`（executable）→ 验证生成的目录结构（`src/main.cpp`、`ezmk.toml`、`.gitignore`、`README.md`）和 toml 内容
- [x] `project new --type static` → 验证 `ezmk.toml` 中 type = "static"
- [x] `project new --type shared` → 验证 `ezmk.toml` 中 type = "shared"
- [x] `project new --disable-git-init` → 验证 `.git/` 目录不存在
- [x] `project new --disable-gitignore` → 验证 `.gitignore` 不存在
- [x] `project build`（executable）→ 验证 `build/<name>.exe` 存在且可执行
- [x] `project build`（static）→ 验证 `build/lib<name>.a` 存在
- [x] `project build`（shared）→ 验证 `build/lib<name>.dll`/`.so` 存在
- [x] `project run` → 验证先构建再执行，exit code == 0
- [x] `project clean` → 验证 `.ezmk/cache/` 和 `.ezmk/temp/` 被清除
- [x] `project build --verbose` → 验证输出包含 `cmd:` 和缓存详情信息

### 2.2 缓存正确性

- [x] 首次构建 → 全部编译（0 cache hits）
- [x] 无修改二次构建 → 全部命中（0 cache miss）
- [x] 修改单个源文件 → 仅该文件重新编译
- [x] 修改公共头文件 → 所有 include 该头的源文件重新编译
- [x] 修改 compile flags（`ezmk.toml`） → 全部重新编译（compile_options_signature 变化）
- [x] `--disable-cache` → 全部重新编译但更新缓存（下次无 --disable-cache 时可命中）
- [x] 损坏 `record.json` → 自动识别、warn、重建缓存
- [x] 清理后重新构建 → 全量编译（无缓存命中）

### 2.3 Package 生命周期

- [x] `pkg install` 本地 `.zip` → 验证解压到 `.ezmk/pkg/`、依赖编译完成、`build/lib<pkg>.a` 存在
- [x] `pkg install` 本地 `.tar.gz` → 同上
- [x] `pkg install` URL → 验证下载 + 安装（使用本地 HTTP 服务器 mock，或测试本地文件路径）
- [x] `pkg install --sha256` → 哈希匹配 → 安装成功
- [x] `pkg install --sha256` → 哈希不匹配 → 报错并拒绝安装
- [x] `pkg install -y` → 跳过所有交互确认（包括全局确认和覆盖确认）
- [x] `pkg install -g` → 验证二次确认提示（`-y` 时跳过）
- [x] `pkg install` 覆盖已有包 → 验证二次确认提示
- [x] `pkg remove` → 验证目录被删除
- [x] `pkg search` → 验证已安装的包被找到（输出包路径）
- [x] `pkg search` 未安装 → 验证输出为空
- [x] `pkg info` → 验证输出包含包名、版本、type、语言、作用域、安装时间、编译 flags、include dirs、依赖等字段
- [x] 循环依赖检测 → 验证报错（`"circular dependency detected"`）而非死循环
- [x] 缺失依赖 → 验证报错提示安装缺失包
- [x] 依赖链安装（A depends on B）→ B 先编译，A 后编译
- [x] preinstall / postinstall hook 脚本 → 验证执行流程（确认 → editor → 执行）

### 2.4 Repo 生命周期

- [x] `repo add` 本地目录（type=local）→ 验证注册到 `list.toml`
- [x] `repo add` git URL → 验证 clone + 注册
- [x] `repo add` 同名 repo → 报错（需先 remove）
- [x] `repo add` 无效 index.toml → 报错并清理 clone
- [x] `repo update` → 验证 git pull / 重新读取
- [x] `repo update` 缓存缺失 → 重新 clone
- [x] `repo list` → 验证输出包含已注册 repo（含 URL、branch、last_update）
- [x] `repo remove` → 验证取消注册 + 缓存删除
- [x] `repo remove` 不删除 `type=local` 的源目录
- [x] 通过 repo 名称 `pkg install` → 验证自动查找 repo 中的包（按版本号取最新）
- [x] repo `index.toml` 中 `sha256` → 安装时自动校验
- [x] 项目/用户/全局三个 scope 的 repo 隔离 → 各自独立注册和查找

### 2.5 CLI 集成测试

- [x] `ezmk` 无参数 → 输出 usage
- [x] `ezmk help` / `ezmk --help` / `ezmk -h` → 输出 usage
- [x] `ezmk version` / `ezmk --version` / `ezmk -V` → 输出版本号
- [x] `ezmk unknown-group` → 报错并提示使用 help
- [x] `ezmk project` 无子命令 → 报错
- [x] Scope flag 组合（`-pug`、`-pu`、`-pg`、`-ug`）正确解析

### 2.6 注意事项

- 集成测试每个 case 在独立的临时目录中运行（`fs::temp_directory_path()` 下创建唯一子目录）
- 不要依赖网络 —— 涉及 URL/download/repo clone 的测试使用本地文件或 mock
  - URL 下载测试：可启动本地 HTTP 服务器（如 Python `http.server`）提供 `.zip` 文件
  - git repo 测试：在临时目录中手动创建 git repo（`git init` + `git add` + `git commit`）用作 clone 源
- 需要 git 的测试先检查 `util::git_available()`，不可用时 skip
- 需要 g++ 的测试先检查编译器可用性
- 测试完成后清理所有临时目录（使用 RAII guard 或在 tear-down 中清理）
- 集成测试中调用 `cli::parse()` 构造参数，然后调用相应的业务函数（如 `project::create_project()`、`build::build_project()`），验证文件系统副作用
- `pkg install` URL 测试注意：当前 `util::download()` 在 Windows 用 WinHTTP、在 Linux 用 curl——集成测试可用本地文件路径绕过网络依赖
