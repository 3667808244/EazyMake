# EazyMake 0.9.9 — 安装钩子 Lua 化

---

## 1 背景

当前 EazyMake 有两套钩子系统，技术栈不统一：

| 钩子类型 | 技术栈 | 生命周期 | 引入版本 |
|---|---|---|---|
| 构建钩子 (`[hooks]`) | Lua (`pre_build`/`post_build`/`on_failure`) | `project build` 流程 | 0.2.3 |
| 安装钩子 (`script/`) | Shell 脚本 (`.sh`/`.ps1`/`.bat`) | `pkg install` 流程 | 0.2.1 |

Shell 脚本方案存在以下问题：

1. **跨平台碎片化**：同一逻辑需为 Linux/macOS（`.sh`）和 Windows（`.ps1` + `.bat` 备选）编写三份脚本，维护成本高且行为可能不一致
2. **功能受限**：Shell 脚本只能调用外部命令，无法使用 `ezmk.*` API（如 `ezmk.file_read()`、`ezmk.info()`、`ezmk.json_decode()` 等）
3. **安全审计困难**：Shell 脚本需打开编辑器供用户审查，但脚本可能调用任意系统命令，审查成本高
4. **与项目整体架构不一致**：构建钩子已于 0.2.3 迁移至 Lua，安装钩子仍停留在旧方案，形成技术债务

本版本将安装钩子迁移至 Lua，遵循 `docs/zh/utils.md` 中确立的 Lua 脚本规范（沙箱安全模型、`ezmk.*` API、跨平台统一），与构建钩子形成一致的技术栈。

> **注意**：构建钩子（`[hooks]`）和安装钩子（`script/`）是两套独立的生命周期系统，本版本仅迁移安装钩子，不改变构建钩子行为。

---

## 2 目标

1. **安装钩子 Lua 化** — `preinstall`/`postinstall` 钩子支持 `.lua` 脚本，优先于旧有的 shell 脚本
2. **统一上下文传递** — Lua 钩子通过 `ctx` 表接收安装上下文（包名、安装路径、作用域等）
3. **向后兼容** — 旧包中的 `.sh`/`.ps1`/`.bat` 脚本继续工作，无需立即迁移
4. **安全性对齐** — Lua 钩子运行在沙箱环境中，复用 `ezmk.*` API 和现有的权限模型

---

## 3 详细设计

### 3.1 Lua 钩子规范

#### 3.1.1 文件位置

沿用现有 `script/` 目录，新增 `.lua` 扩展名：

```
<pkg_root>/
    script/
        preinstall.lua     # 预安装钩子（Lua，跨平台，优先）
        preinstall.sh      # 预安装钩子（Shell，Linux/macOS，向后兼容）
        preinstall.ps1     # 预安装钩子（PowerShell，Windows，向后兼容）
        preinstall.bat     # 预安装钩子（Batch，Windows 备选，向后兼容）
        postinstall.lua    # 安装后钩子（Lua，跨平台，优先）
        postinstall.sh     # 安装后钩子（Shell，Linux/macOS，向后兼容）
        postinstall.ps1    # 安装后钩子（PowerShell，Windows，向后兼容）
        postinstall.bat    # 安装后钩子（Batch，Windows 备选，向后兼容）
```

#### 3.1.2 检测优先级

`detect_install_script()` 修改为：

1. **`.lua`** — 跨平台，最高优先级（若存在则直接使用，跳过平台检测）
2. **平台特定脚本**（作为 fallback）：
   - Windows: `.ps1` → `.bat`
   - Linux/macOS: `.sh`

> 这一优先级确保新包只需提供一个 `.lua` 文件即覆盖所有平台；旧包若未提供 `.lua`，仍走原有的平台脚本逻辑。

#### 3.1.3 Lua 入口约定

每个安装钩子脚本需定义一个 `run(ctx)` 函数：

```lua
-- script/preinstall.lua
-- 在包解压后、安装前执行

-- 入口函数。ctx 为安装上下文表（见 §3.1.4）
-- 返回整数 exit code：0 = 继续安装，非 0 = 失败（用户可选择继续或中止）
function run(ctx)
    ezmk.info("preinstall hook for " .. ctx.pkg_name)

    -- 示例：检查目标路径是否已存在旧版本
    if ezmk.file_exists(ctx.install_path .. "/config.ini") then
        ezmk.warn("found existing config.ini, backing up...")
        local backup = ctx.install_path .. "/config.ini.bak"
        local ok = ezmk.file_write(backup,
            ezmk.file_read(ctx.install_path .. "/config.ini"))
        if not ok then
            ezmk.error("failed to backup config.ini")
            return 1
        end
    end

    return 0
end
```

```lua
-- script/postinstall.lua
-- 在包安装完成后执行

function run(ctx)
    ezmk.info("postinstall hook for " .. ctx.pkg_name)

    -- 示例：生成默认配置文件
    local config_path = ctx.install_path .. "/config.ini"
    if not ezmk.file_exists(config_path) then
        ezmk.file_write(config_path, "# Generated by ezmk\n")
    end

    return 0
end
```

#### 3.1.4 上下文表 `ctx`

`run(ctx)` 接收的上下文表字段：

| 字段 | 类型 | 说明 | 示例值 |
|---|---|---|---|
| `ctx.pkg_name` | string | 正在安装的包名（来自 `ezmk.toml` 的 `[project].name`） | `"fmt"` |
| `ctx.pkg_root` | string | 包解压后的临时根目录（绝对路径） | `"/tmp/ezmk_install_xxx/fmt"` |
| `ctx.install_path` | string | 目标安装路径（绝对路径） | `"~/.local/ezmk/pkg/fmt"` |
| `ctx.scope` | string | 安装作用域 | `"project"` / `"user"` / `"global"` |
| `ctx.pkg_version` | string | 包版本号（来自 `ezmk.toml` 的 `[project].version`） | `"10.2.1"` |
| `ctx.pkg_type` | string | 包类型 | `"static"` / `"shared"` / `"utils"` |

> **设计原则**：`ctx` 表的字段参考了现有构建钩子的 `ctx` 结构（`output`/`project_root`/`profile`，见 `src/lua_api.cpp:1178-1185`），并扩展到安装场景所需的上下文。

#### 3.1.5 `run(ctx)` 契约

- **返回值**：整数 exit code。`0` = 成功，非 `0` = 失败
- **若未定义 `run` 函数**：ezmk 打印警告并跳过（视为成功，继续安装流程）
- **若 `run` 抛出 Lua error**：ezmk 捕获并展示错误信息，用户可选择继续或中止
- **脚本执行超时**：无硬性超时（由用户通过 Ctrl+C 中断）

### 3.2 安装流程调整

当前安装流程（`src/pkg.cpp` 的 `install()` 函数）：

```
1. 解压 → 2. preinstall 脚本 → 3. 覆盖检查 → 4. 编译依赖 → 5. 复制文件 → 6. postinstall 脚本
```

本版本不改变流程顺序，仅在步骤 2 和 6 中调整脚本检测和执行逻辑：

**修改前**（`pkg.cpp:444-453`）：
```cpp
fs::path preinstall_script = detect_install_script(pkg_root, "preinstall");
if (!preinstall_script.empty()) {
    if (!run_install_script(preinstall_script, dest_dir / pkg_name,
                            assume_yes, "preinstall")) {
        // ...
    }
}
```

**修改后**：
```cpp
// detect_install_script 优先返回 .lua（若存在）
fs::path preinstall_script = detect_install_script(pkg_root, "preinstall");
if (!preinstall_script.empty()) {
    bool is_lua = (preinstall_script.extension() == ".lua");
    if (!run_install_script(preinstall_script, dest_dir / pkg_name,
                            assume_yes, "preinstall", is_lua,
                            pkg_name, pkg_root, dest_dir / pkg_name)) {
        // ...
    }
}
```

### 3.3 代码改动

#### 3.3.1 `detect_install_script()` 改造（`src/pkg.cpp`）

```cpp
static fs::path detect_install_script(const fs::path& pkg_root,
                                       std::string_view basename) {
    auto script_dir = pkg_root / "script";
    if (!util::file_exists(script_dir)) return {};

    // 1. .lua 脚本优先（跨平台）
    fs::path lua = script_dir / (std::string(basename) + ".lua");
    if (util::file_exists(lua)) return lua;

    // 2. 平台特定脚本作为 fallback
#ifdef EZMK_WIN
    fs::path ps1 = script_dir / (std::string(basename) + ".ps1");
    if (util::file_exists(ps1)) return ps1;
    fs::path bat = script_dir / (std::string(basename) + ".bat");
    if (util::file_exists(bat)) return bat;
#else
    fs::path sh = script_dir / (std::string(basename) + ".sh");
    if (util::file_exists(sh)) return sh;
#endif
    return {};
}
```

#### 3.3.2 `run_install_script()` 改造（`src/pkg.cpp`）

新增 Lua 执行路径：

```cpp
static bool run_install_script(const fs::path& script, const fs::path& cwd,
                                bool assume_yes, std::string_view label,
                                bool is_lua,
                                const std::string& pkg_name,
                                const fs::path& pkg_root,
                                const fs::path& install_path) {
    std::string desc = std::string(label) + " " + script.filename().string();

    util::info(ezmk::i18n::I18nKey::found_script, {{"label", desc}});

    if (is_lua) {
        // Lua 脚本：不在编辑器中打开（沙箱运行，API 可控）
        if (!confirm(ezmk::i18n::fmt(ezmk::i18n::I18nKey::exec_question,
                                      {{"label", desc}}), assume_yes)) {
            util::info(ezmk::i18n::I18nKey::skipping, {{"label", desc}});
            return true;
        }
        util::info(ezmk::i18n::I18nKey::running_script, {{"label", desc}});
        int rc = lua::run_install_hook_script(lua::state(), script,
                                               pkg_name, pkg_root,
                                               install_path, package_scope);
        if (rc != 0) {
            // ... 失败处理（同现有逻辑：展示错误，询问是否继续）
        }
    } else {
        // Shell 脚本：现有逻辑（打开编辑器 → 确认 → 执行）
        util::open_in_editor(script);
        // ... 现有 shell 执行逻辑
    }
    // ... 共用的确认和错误处理逻辑
}
```

### 3.4 安全模型

Lua 安装钩子复用现有的 ezmk Lua 安全基础设施：

- **Sandbox 环境**：每次调用独立环境表（`lua_api.cpp:1137-1143`），脚本间零污染
- **受限 API**：`os.execute` 和 `io.popen` 编译期移除（`src/vendor/lua/linit.c`）
- **文件写入限制**：`ezmk.file_write()` 拒绝项目根目录外写入
- **外部命令**：必须通过 `ezmk.run()` 执行（受 `[utils.permissions]` 管控，若包声明了权限）
- **无需编辑器审查**：Lua 脚本运行在沙箱中，API 可控，不再需要打开编辑器供用户审查（与 shell 脚本不同）

> **与 Shell 脚本的安全差异**：Shell 脚本需打开编辑器审查 → 用户确认 → 执行；Lua 脚本仅需用户确认执行（不再打开编辑器），因为其 API 边界已由 sandbox 限定。

## 5 兼容性矩阵

| 变更 | 影响 | 处理 |
|---|---|---|
| `.lua` 脚本优先级高于平台脚本 | 若有同时提供 `.lua` 和 `.sh`/`.ps1` 的包，行为可能改变 | `.lua` 与旧脚本不共存是标准做法；若确实共存，`.lua` 优先是合理的升级路径 |
| 不再为 Lua 脚本打开编辑器 | 用户失去审查脚本内容的机会 | Lua 脚本受 sandbox 限制，API 可控；用户确认步骤保留（`[y/N]`） |
| 新增 `run_install_hook_script()` | 无 | 纯增量 |
| `detect_install_script()` 返回值可能为 `.lua` | 调用方需区分文件类型 | `run_install_script()` 内部通过扩展名判断 |
| 旧包（仅有 `.sh`/`.ps1`/`.bat`） | 行为不变 | `detect_install_script()` 在无 `.lua` 时 fallback 到平台脚本 |

## 6 跨版本关注点

- **依赖**: 无硬依赖；Lua 基础设施自 0.2.0 起已就绪（`lua_State*` 全局复用、sandbox 机制、`ezmk.*` API 注册）
- **向后兼容**: 旧包的 shell 脚本继续工作；新包建议只提供 `.lua` 文件
- **与构建钩子的关系**: 构建钩子（`[hooks]`）和安装钩子（`script/`）是两套独立的生命周期系统，本版本不改变构建钩子行为。两者的 Lua 脚本共享相同的 sandbox 机制和 `ezmk.*` API
- **1.0.0 准备**: 本版本消除最后的技术债务（Shell → Lua 迁移完成），为 1.0.0 的"零技术债务"发布标准扫清障碍
- **延后项**: 安装钩子的 `[utils.permissions]` 细粒度权限管控（当前复用 utils 权限模型，安装钩子场景可能需要独立的权限声明机制）
