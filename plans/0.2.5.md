# EazyMake 0.2.5 — 生态与安全

---

## 1 背景

0.2.4 完成了健壮性补全（Bug 修复、代码质量、功能补全、工程规范）。当前 ezmk 已经具备稳定的跨平台构建能力，但在以下方面仍有明显短板：

- **参数解析**：`src/cli.cpp` 的命令行解析是手写的、逐命令特判的实现，不遵循任何统一规范——不支持 `--flag=value`、不支持短选项紧贴取值（`-j4`）、不支持 `--` 终止符、选项与位置参数不能自由混排，作用域 flag（`-pug`）靠字符串查找硬编码。这既影响用户体验，也让 Shell 补全难以对齐
- **Shell 补全**：ezmk 拥有 15+ 子命令和大量 flag，用户每次均需手动输入完整命令，没有 tab 补全支持
- **Utils 安全模型**：当前 Lua sandbox 采用"全有或全无"策略——脚本要么拥有完整的 `ezmk.*` API 访问权限，要么完全不能运行。缺少**细粒度的权限控制**（如限制某脚本只能读特定目录、只能执行白名单命令）
- **仓库系统**：`docs/repo.md` 中描述的部分特性尚未实现或存在粗糙之处——本地仓库验证不完整、缺少 `repo info` 命令、`pkg install` 从仓库安装时不会自动 `repo update`、多仓库同名包的选择策略未明确定义

本版本聚焦于**统一参数解析（GNU 标准）+ 补齐生态工具（Shell 补全）+ 强化安全模型（Utils 权限管理）+ 完善仓库系统**。参数规范化是其余三项的基础：补全脚本需要对齐一套确定的参数语法，权限声明和仓库命令也会引入新 flag。

## 2 参数规范化（GNU 标准）

### 2.1 现状

`src/cli.cpp` 的解析器完全手写、逐子命令特判，存在以下与通用 CLI 约定不一致的地方：

| 问题 | 现状 | 举例 |
|---|---|---|
| 不支持 `--flag=value` | 只认 `--flag value`（空格分隔） | `--profile=debug` 会被当成未知 flag |
| 短选项不支持紧贴取值 | 只认 `-j 4` | `-j4` 报错 |
| 短选项分组是特例 | 仅作用域 flag `-pug` 靠 `find('p')` 硬编码 | 其他短选项无法分组 |
| 无 `--` 终止符 | 无法显式结束选项解析 | `ezmk utils fmt -- --help` 无法把 `--help` 透传给工具 |
| 选项/位置参数不能混排 | 靠固定下标（`i=3` 起）遍历，逻辑脆弱 | `ezmk pkg install foo.zip -g` 可用但纯属巧合 |
| 解析分散、不一致 | 每个子命令各写一套 flag 匹配 | `-v`/`--verbose` 仅 build 支持，作用域走另一套逻辑 |
| hacky 的 argv 改写 | `pkg update --all` 手动 `argv[j]=argv[j+1]` 平移 | 见 `src/cli.cpp` |

**核心问题**：没有统一的"选项语法层"，每加一个 flag 就要在多处手写字符串比较，容易漏、容易不一致，也让 §3 的补全脚本难以对齐。

### 2.2 目标：GNU 参数语法约定

采用 [GNU Program Argument Syntax Conventions](https://www.gnu.org/software/libc/manual/html_node/Argument-Syntax.html)：

1. **长选项** `--word`；带值可写 `--word=value` 或 `--word value`
2. **短选项** `-x`；可分组 `-xyz` 等价于 `-x -y -z`
3. **短选项带值** `-o value` 或紧贴 `-ovalue`；分组时带值选项必须放在末尾（`-vj4` = `-v -j 4`）
4. **`--` 终止选项解析**：其后所有 token 一律视为位置参数——对 `utils` / `project run` 透传下游参数至关重要
5. **选项与位置参数可自由混排**（GNU permutation），`--` 之后除外
6. `--help` / `-h`、`--version` / `-V` 保持标准语义
7. 单独的 `-` 作为 token 视为位置参数（保留给"stdin"惯例），不解释为选项
8. 无法识别的选项 → 统一、可定位的错误消息（`unknown option '--foo' for 'ezmk project build'`）

**明确不做**（超出 GNU 约定或收益低）：
- 长选项前缀缩写（`--verb` 匹配 `--verbose`）——歧义处理复杂，暂不支持
- 选项别名的自动大小写折叠

### 2.3 设计决策

**引入一个轻量选项解析层，而非继续逐命令特判。**

新增 `include/ezmk/argparse.hpp`（header-only 或配 `src/argparse.cpp`），提供：

```cpp
namespace ezmk::cli {

struct OptionSpec {
    char        short_name;   // 短选项字符，'\0' 表示无
    std::string long_name;    // 长选项名（不含 --），空表示无
    bool        takes_arg;    // 是否吃一个值
};

// 解析结果：已识别的选项（按出现顺序）+ 剩余位置参数
struct ParsedOptions {
    // key = long_name（若无 long 则用 string(1,short_name)）
    std::vector<std::pair<std::string, std::string>> options; // (name, value)；无值选项 value 为空
    std::vector<std::string> positionals;
    bool has(std::string_view name) const;
    std::optional<std::string> value(std::string_view name) const;
    int count(std::string_view name) const;   // 支持重复选项（如未来 -v -v）
};

// 按 GNU 规则对 [begin, end) 的 argv 分片做分词。
// 遇到未知选项 / 缺失值 → 抛 CliError（携带命令名 + 具体 token）。
ParsedOptions parse_options(int argc, char** argv, int begin,
                            const std::vector<OptionSpec>& spec,
                            std::string_view cmd_name);

} // namespace ezmk::cli
```

各子命令解析函数改为：**声明自己的 `OptionSpec` 表 → 调用 `parse_options()` → 从 `ParsedOptions` 读值**。作用域 flag `-p`/`-u`/`-g` 退化为三个普通短选项，分组语义由通用分词器免费提供（`-pug` 自然拆成 `-p -u -g`），删除 `parse_scope_flags()` 的硬编码字符查找。

**收益**：
- 新增 flag = 往 spec 表加一行，不再散落多处字符串比较
- `--flag=value`、`-j4`、`--`、混排一次实现，全命令共享
- 补全脚本（§3）可以直接对照各命令的 spec 表，语法一致
- `pkg update --all` 的 argv 平移 hack 消失

### 2.4 各命令的参数语义（规范化后）

| 命令 | 短选项 | 长选项（带值） | 长选项（开关） | 位置参数 |
|---|---|---|---|---|
| `project new` | — | `--type <t>` | `--disable-git-init`、`--disable-gitignore` | `<name>`（必填） |
| `project build` | `-v`、`-j <N>` | `--jobs <N>`、`--profile <name>` | `--disable-cache`、`--verbose`、`--auto-update`(§5) | — |
| `project run` | `-v`、`-j <N>` | `--jobs`、`--profile` | `--disable-cache`、`--verbose` | `--` 之后透传给程序 |
| `project watch` | `-v`、`-j <N>` | `--jobs`、`--profile` | `--disable-cache`、`--verbose`、`--no-build-on-start` | — |
| `pkg install` | `-p/-u/-g`、`-y` | `--sha256 <hash>` | `--yes` | `<pkg>`（必填，单作用域） |
| `pkg remove/search/info` | `-p/-u/-g` | — | — | `<pkg>`（必填，可组合作用域） |
| `pkg list` | `-p/-u/-g` | — | — | — |
| `pkg update` | `-p/-u/-g` | — | `--all` | `[<pkg>]`（`--all` 时可省） |
| `repo add` | `-p/-u/-g` | `--name <n>`、`--branch <b>` | — | `<url_or_path>`（必填，单作用域） |
| `repo remove/update` | `-p/-u/-g` | — | — | `[<name>]` |
| `repo list` | `-p/-u/-g` | — | — | — |
| `utils` | — | — | — | `<tool>` + `--` 之后透传给工具 |

**兼容性**：所有现有写法（`--flag value`、`-j 4`、`-pug`、位置参数在末尾）保持有效；新增的是 `--flag=value`、`-j4`、`--`、以及混排能力。**无破坏性变更。**

### 2.5 边缘情况

- **`--` 语义**：仅第一个 `--` 生效；后续 `--` 作为普通位置参数。`utils` 与 `project run` 必须在 `--` 之后停止解析，把剩余 token 原样存入 `utils_args` / 程序参数。
- **作用域互斥**：`pkg install` / `repo add` 只允许单一作用域。规范化后仍在读值阶段校验：`parse_options` 收集到多个 `-p/-u/-g` 时由调用方判断并报 `accepts only one scope flag`（保留现有错误语义）。
- **`-j` 取值校验**：`--jobs=abc` / `-j -1` 等非法值的错误消息与现有一致（`invalid -j value` / `'-j' value must be >= 0`）。
- **未知短选项分组**：`-xj4` 中 `-x` 未知 → 报 `unknown option '-x'`，而非静默忽略。
- **值以 `-` 开头**：`--sha256 -abc` 中 `-abc` 会被当作 `--sha256` 的值（因为 `--sha256` 声明 `takes_arg`）；如需传真正以 `-` 开头的值仍可 `--sha256=-abc`。

## 3 Shell 命令补全

### 3.1 现状

ezmk 没有任何 shell 补全支持。用户在输入 `ezmk project b<TAB>` 时无法自动补全为 `build`，输入 `ezmk pkg install -<TAB>` 时无法看到可用 flag。

### 3.2 目标行为

为 zsh 提供静态生成的补全脚本 `_ezmk`，支持：

**子命令补全**：
```
ezmk <TAB>
→ build   clean   help   new   pkg   project   repo   run   utils   version   watch
```

**嵌套子命令补全**：
```
ezmk project <TAB>
→ build   clean   new   run   watch

ezmk pkg <TAB>
→ info   install   list   remove   search   update
```

**Flag 补全**：
```
ezmk project build -<TAB>
→ --disable-cache   --verbose   --jobs   --profile   --help

ezmk pkg install -<TAB>
→ -p   -u   -g   --sha256   -y   --help
```

**作用域 flag 补全**（`-p`/`-u`/`-g`）：
```
ezmk repo add -<TAB>
→ -p   -u   -g   --name   --branch   --help
```

**动态补全**（上下文感知）：
- `--profile <TAB>`：读取当前目录 `ezmk.toml` 中已定义的 profile 名称
- `pkg remove/update/info <TAB>`：列出已安装包名
- `repo remove/update <TAB>`：列出已注册仓库名
- `utils <TAB>`：列出已安装的 utils 工具名

### 3.3 设计决策

**为什么从 zsh 开始**：
- zsh 补全系统最强大（支持动态上下文感知补全），bash 补全功能较弱（主要是静态）
- 项目主要开发者在 MSYS2/MinGW 环境工作，zsh 是 MSYS2 下推荐 shell
- zsh 补全脚本可以后续作为 bash/fish 补全的参考模板

**补全语法对齐 §2**：补全脚本按 §2.4 的 spec 表逐命令枚举短/长选项。修改 CLI 参数时，spec 表与补全脚本需同步更新（见 §8 注意事项）。

**静态 vs 动态生成**：
- 补全脚本为**静态文件**（`completions/_ezmk`），安装时复制到 zsh `$fpath` 目录
- 动态部分（如 profile 列表）通过调用 `ezmk` 内部命令或读取文件实现
- 不引入 `ezmk completion generate` 子命令——补全脚本足够简单，直接维护源码即可

**安装方式**：
```bash
# 手动安装
cp completions/_ezmk /usr/share/zsh/site-functions/
# 或
cp completions/_ezmk ~/.zsh/completions/  # 需在 .zshrc 中添加 fpath
```

后续版本可添加 `make install` 目标自动安装补全。

### 3.4 补全覆盖范围

静态 flag 补全项与 §2.4 的各命令参数语义表一一对应（短/长选项直接照搬）。补全脚本额外提供的**动态（上下文感知）补全**：

| 命令 | 动态补全项 |
|---|---|
| `project build/run/watch` | `--profile` — 读取当前目录 `ezmk.toml` 中已定义的 profile 名 |
| `pkg remove/info/update` | 已安装包名 |
| `repo remove/update/info` | 已注册仓库名 |
| `utils` | 已安装工具名 + `--` 之后透传工具自身参数 |

顶层与嵌套子命令补全见 §3.2。

- [ ] 实现顶层命令补全（`_ezmk` 主函数）

## 4 Utils 权限管理

### 4.1 现状

Lua sandbox 当前的安全策略：

| 能力 | 策略 |
|---|---|
| `os.execute` / `io.popen` | 编译期移除（`linit.c`） |
| `ezmk.file_write()` | 拒绝项目根目录外的绝对路径 |
| `ezmk.run()` | 无限制——可执行任意命令 |
| `ezmk.file_read()` | 无限制——可读取任意文件 |
| `ezmk.list_sources()` | 无限制——可列出任意项目源文件 |

**问题**：一个 utils 包（如 `ezmk-cc`）的 Lua 脚本理论上可以通过 `ezmk.file_read()` 读取 `~/.ssh/id_rsa`，或通过 `ezmk.run()` 执行 `rm -rf /`。当前唯一的防线是"信任安装的包"——这在公共仓库场景下是不够的。

### 4.2 目标行为

在 `ezmk.toml` 的 `[utils]` 节中新增可选的权限声明。每类权限（read/write/run）均由**白名单 allow + 黑名单 deny** 两个列表组成：

```toml
[utils]
tools = ["cc", "compile-commands"]

[utils.permissions]
read       = ["src/", "include/", "ezmk.toml"]   # 白名单：允许读取的路径（相对项目根）
read_deny  = ["**/.ssh/", "**/*.key", ".env"]    # 黑名单：即使在白名单内也拒绝
write      = ["build/", ".ezmk/cache/"]
write_deny = []
run        = ["g++", "clang++", "cl.exe", "git"] # 白名单：允许执行的命令
run_deny   = ["rm", "curl", "wget", "sh"]        # 黑名单：始终拒绝
network    = false                                # 声明式，暂不执行
```

**统一判定顺序**（read/write/run 三类一致，优先级从高到低）：

1. 命中 **deny 黑名单** → **拒绝**（最高优先级，即使同一目标也在白名单中）
2. 否则命中 **allow 白名单** → **允许**
3. 两者都不命中 → **询问用户**（交互式确认，见 §4.2.5）

> 特例：`[utils.permissions]` 整节不存在时，保持旧行为（无限制 + 一次性 warning），**不进入**上述流程——见 §4.2.3 与 §8。

#### 4.2.1 目录读取权限（read / read_deny）

当脚本调用 `ezmk.file_read(path)` 时，将 `path` 规范化为绝对路径后按统一顺序判定：
1. 命中 `read_deny` → 拒绝
2. 命中 `read` → 允许
3. 都不命中 → 询问（§4.2.5）

**特殊路径**（隐式白名单）：
- `ezmk.toml` 本身、`ezmk.pkg_dir()` 返回的工具自身包目录默认视为允许
- 但若显式写入 `read_deny`，黑名单仍优先（deny 永远最高）

#### 4.2.2 目录写入权限（write / write_deny）

当脚本调用 `ezmk.file_write(path, content)` 时：
1. 先经过已有的"禁止写项目根目录外"硬限制（不可绕过，即使白名单包含项目外路径）
2. 命中 `write_deny` → 拒绝
3. 命中 `write` → 允许
4. 都不命中 → 询问（§4.2.5）

#### 4.2.3 命令执行权限（run / run_deny）

当脚本调用 `ezmk.run(cmd)` 或 `ezmk.run_capture(cmd)` 时，提取命令第一个 token（可执行文件名，不含参数）后判定：
1. 命中 `run_deny` → 拒绝
2. 命中 `run` → 允许
3. 都不命中 → 询问（§4.2.5）

匹配规则：
- 精确匹配为主：`"g++"` 匹配 `g++`，不匹配 `g++-13`
- `*` 结尾前缀通配：`"git*"` 匹配 `git`、`git.exe`
- 全路径匹配：`"C:/msys64/ucrt64/bin/g++.exe"` 匹配完整路径

**向后兼容（旧包，`[utils.permissions]` 整节缺失）**：保持无限制，但首次调用受控 API 时打印一次 deprecation warning：`[ezmk warn] utils permissions not declared; all file/command access allowed. Define [utils.permissions] to restrict.`。一旦声明了 `[utils.permissions]`，三类权限即全部进入 deny/allow/ask 模型；某字段省略 = 空列表 = 该类访问一律询问。

#### 4.2.4 网络权限（network）

当前版本：声明式字段，暂不执行。未来可用于控制 `ezmk.run()` 是否允许执行 `curl`/`wget` 等网络命令。

#### 4.2.5 询问行为（ask）

当访问目标"既不在白名单也不在黑名单"时，向用户交互式确认：

```
[ezmk] utils 工具 'cc' 请求 READ 访问：
  /home/user/proj/external/foo.txt
该目标未被包权限声明为允许或拒绝。
允许？ [y]是 / [n]否 / [a]本会话始终允许 / [d]本会话始终拒绝：
```

- **会话内缓存**：决策按"权限类别 + 规范化目标"缓存；`a`/`d` 对同类后续访问自动生效，避免反复打断。
- **非交互环境**（无 TTY，或已传 `-y` / `--yes`）：无法询问 → 一律**拒绝**（fail-safe），并打印被拒目标，便于用户补白名单后重试。
- 询问仅在"两者都不命中"时触发；deny/allow 命中时直接判定，不打扰用户。

### 4.3 权限声明在包定义中

权限声明在**包的 `ezmk.toml`** 中（而非用户项目的 `ezmk.toml` 中）。这样：
- 包开发者定义其工具所需的最小权限
- 用户安装包时可以看到权限声明（`pkg info` 展示）
- 用户可以在项目 `ezmk.toml` 中覆盖权限（收紧，不能放宽）

**用户项目中的覆盖**（0.2.5 暂不实现，预留设计）：

```toml
# 用户项目的 ezmk.toml
[utils.overrides.ezmk-cc]
read = ["src/", "include/"]                # 收紧：不允许读其他目录
run = ["g++", "clang++", "cl.exe"]         # 收紧：只允许编译器
```

### 4.4 `pkg info` 展示权限

对定义了 `[utils.permissions]` 的包，`ezmk pkg info` 额外展示：


### 4.6 安全分析

| 威胁场景 | 缓解措施 |
|---|---|
| 恶意包读取 `~/.ssh/` | 不在 `read` 白名单 → 询问用户（非交互直接拒绝）；可显式 `read_deny` 硬拒 |
| 恶意包执行 `rm -rf /` | `rm` 在 `run_deny` → 拒绝；即使被列入白名单，deny 仍优先 |
| 恶意包写入系统目录 | 项目根目录硬限制 + `write_deny` + 白名单外询问（三层） |
| 白名单过宽被利用（如 `read = ["./"]` 却读 `.env`） | `read_deny` 精确列出敏感文件，deny > allow 保证拦截 |
| 恶意包通过 `g++` 的参数注入执行任意命令 | 编译器参数注入已有独立缓解（见 0.2.4 §2.2 shell 转义） |
| 未声明权限的旧包 | 向后兼容：保持无限制，但打印 warning 提示用户 |
| CI/自动化中静默放行未知访问 | 非交互环境 `Ask` → fail-safe 拒绝，绝不默认放行 |

### 5.3 多仓库同名包选择策略明确化

`docs/repo.md` §"与 pkg install 集成" 中描述：

> 若同一包有多个版本，取版本号最高者

需要明确的边缘情况：

**a) 同一仓库中同名包的多个版本**：
- 按 `compare_version()` 数值比较（0.2.4 已统一），取最高版本
- `index.toml` 中同包多版本应各不相同

**b) 不同仓库中的同名包**：
- 当前实现：按仓库注册顺序遍历，找到的第一个匹配即返回
- **问题**：若 repo A 有 `foo 1.0.0`，repo B 有 `foo 2.0.0`，按注册顺序可能在 A 中返回旧版本
- **修复**：扫描所有仓库，收集同名包的所有版本，取版本号最高者。返回结果中附带来源仓库名（用于日志输出）

**c) 同作用域内仓库优先级**：
- 保持按注册顺序遍历，但在同作用域内收集所有匹配后比较版本——即在同作用域内取最高版本
- 作用域优先级不变：项目 > 用户 > 全局

**实施**：
- [ ] 重构 `repo::search_package()`：扫描所有作用域的所有仓库，收集所有同名包版本，返回版本最高者
- [ ] 日志中提示所选版本的来源仓库：`[ezmk] Resolved foo 2.0.0 from repo 'community' (3 repos searched)`

## 8 注意事项

- **参数规范化的无破坏性**：§2 的目标是"只增不改"——所有现有命令行写法保持有效，仅新增 `--flag=value`、短选项紧贴取值、`--` 透传、混排能力。重构 `cli.cpp` 时必须以现有 `test/test_cli.cpp` 全绿为前提，再叠加新语法的测试。
- **spec 表与补全脚本的同步**：`completions/_ezmk` 与各命令的 `OptionSpec` 表描述同一套语法。**修改 CLI 接口（新增/删除 flag）时，需同时更新 spec 表、补全脚本、`print_usage()` 和 README。**
- **权限的向后兼容**：`[utils.permissions]` 整节缺失的旧包行为不变——read/write/run 均无限制，仅打印一次 deprecation warning。这确保所有现有 utils 包（如 `ezmk-cc`）在未更新时仍可正常工作。一旦声明该节，即全面启用 deny/allow/ask 模型。
- **deny 永远优先**：判定顺序固定为 deny > allow > ask。黑名单命中即拒绝，即使同一目标也在白名单中——这是防"白名单开得过宽"的最后一道闸。
- **ask 的非交互兜底**：无 TTY 或传了 `-y`/`--yes` 时，`Ask` 一律按拒绝处理（fail-safe），绝不静默放行。CI 场景应把所需访问显式写进白名单。
- **权限的声明性质**：权限是包开发者对"此工具需要做什么"的声明，由 ezmk 在受控 API 层（file_read/file_write/run）拦截执行，而非 OS 级沙箱。用户应审查权限声明，且 0.3.0 计划支持用户覆盖（收紧）。
- **`search_package()` 重构的性能**：扫描所有仓库的 `index.toml` 可能增加 I/O。对于典型场景（< 5 个仓库，每个 < 50 个包）性能影响可忽略。后续可在仓库数量 > 10 时引入 `index.toml` 内存缓存。
- **`--auto-update` 的默认值**：默认 `false`（不自动更新）。理由：网络延迟不可控，应由用户显式控制。在 CI 中可通过 `build.sh` 脚本先执行 `ezmk repo update` 再执行 `ezmk project build`。
- **Shell 补全仅在 zsh 测试**：bash/fish 补全留待社区贡献或后续版本。

## 10 开放问题

1. **长选项缩写**：GNU getopt 支持 `--verb` 唯一前缀匹配 `--verbose`。0.2.5 暂不支持（歧义处理与错误消息复杂度较高），是否值得在后续版本引入？
2. **bash/fish 补全**：是否需要在同一版本中支持？建议先完成 zsh，bash/fish 作为后续补丁版本（0.2.6）或社区贡献。
3. **权限的运行时覆盖**：用户是否应该能够在运行时临时提权（如 `sudo` 模式）？当前设计为"用户只能收紧不能放宽"，若需要临时放宽权限，需先修改 `ezmk.toml`。这是合理的安全设计还是过度限制？
4. **ask 决策的持久化**：当前设计只在**单次会话内**缓存询问结果。是否需要把"始终允许/拒绝"落盘（如 `.ezmk/perms.toml`）以跨会话记住？涉及缓存失效与用户可见性，0.2.5 暂只做会话内缓存。
5. **权限声明的可审计性**：是否需要 `ezmk pkg audit` 子命令来列出所有已安装 utils 包的权限摘要？0.2.5 暂不实现，`pkg info` 已展示单个包的权限。
6. **`network` 权限的实际执行**：当前仅声明式记录，何时开始实际拦截网络命令（如阻止 `ezmk.run("curl ...")`）？需要在 `run` 白名单 + 命令参数检查中实现，复杂度较高，延后到 0.3.0。
7. **仓库索引的内存缓存**：随着仓库数量增长，每次 `search_package()` 都读取所有 `index.toml` 可能成为瓶颈。0.2.5 对 < 10 个仓库不做缓存，后续可按需引入 TTL 缓存。
8. **`repo info` 的 JSON 输出**：是否需要 `--json` flag 以支持脚本化使用？当前设计为人类可读输出，脚本化需求可通过直接读取 `list.toml` + `index.toml` 满足。
