# EazyMake 0.2.6 — 翻译补全和命令行改进

---

## 1 背景

0.2.5 引入了统一的 GNU 参数解析层（`argparse.hpp`）、zsh 补全、Utils 权限管理和仓库系统完善。但随着 0.2.3~0.2.5 大量新增用户可见字符串，i18n 系统出现了**系统性缺陷**：

- **`{???}` 显示 bug**：`ezmk help`、`ezmk pkg list`、`ezmk repo list` 等命令的输出出现大量 `{???}` 占位符，而非本地化文本。
- **帮助页正文未翻译**：`print_usage()` 的分节标题走 i18n，但每条命令/flag 的说明文字是硬编码英文。
- **参数校验报错未翻译**：`src/cli.cpp` 各子命令解析函数里的"缺参数/参数过多/未知子命令"等 `util::fatal(std::string(...))` 全是硬编码英文。

同时，命令行体验有两处可低成本改进：**命令简写**（`ezmk pb` = `ezmk project build`）与 **`--color=<mode>`** 输出控制。

本版本聚焦**补全翻译系统 + 打磨命令行**，不引入新的构建/包管理能力，是一次面向可用性与一致性的收尾版本。

## 2 翻译补全

### 2.1 根因分析（关键）

`ezmk pkg list` / `ezmk repo list` / `ezmk help` 显示 `{???}` **不是**因为 `locale/*.json` 缺少翻译——这些字符串在 `en.json` 与 `zh.json` 中都存在。真正的原因在 `src/i18n.cpp`：

```cpp
// src/i18n.cpp
std::string get(I18nKey key) {
    const char *name = key_name(key);        // ← 枚举 → JSON key 名的映射
    auto it = g_strings.find(name);
    if (it != g_strings.end()) return it->second;
    return std::string("{") + name + "}";    // 未找到 → 返回 "{name}"
}
```

`key_name()` 是一个**手写的巨型 `switch`**，把 `I18nKey` 枚举值映射到 JSON 中的字符串键名。任何**未在该 `switch` 中列出**的枚举值，都会命中函数结尾的兜底 `return "???"`（i18n.cpp:263）。于是 `get()` 去查 `g_strings["???"]`（不存在），返回字面量 `"{???}"`。

**问题**：0.2.3~0.2.5 往 `I18nKey` 枚举（`include/ezmk/i18n.hpp`）和 `locale/*.json` 同步添加了约 50 个新键，但**忘了在 `key_name()` 的 `switch` 中登记**。只有极少数后补的键（`cli_unknown_option`、`cli_missing_value`、`pkg_info_permissions`）被单独追加进去。

因此本质是**三处数据源需手动同步**（枚举 / `key_name()` switch / JSON），漏了中间一处。

### 2.3 附带发现：`repo list` 复用了 pkg 的键

`src/repo.cpp` 的 `list()`（repo.cpp:475）用 `I18nKey::pkg_list_title` 作为仓库列表表头，语义上应是"Repositories (...)"而非"已安装包 (...)"。修复 2.2 后表头会正常显示，但文本是 pkg 的语义。需**新增 `repo_list_title`**（英："Repositories ({scope}):"，中："仓库 ({scope} 作用域):"）并让 `repo::list()` 改用它，同理为空列表加 `repo_list_none`（可复用 `no_repos` 逻辑）。

### 2.4 修复方案

**最小修复**：把 2.2 清单里的所有枚举值补进 `key_name()` 的 `switch`。可直接完成，但这类"枚举 / switch / JSON 三处同步"的 bug 会**再次发生**。

**推荐的根治方案（单一数据源）**：用 X-macro 把"枚举名 == JSON 键名"这一事实只写一次，`enum` 与 `key_name()` 都从它生成：

```cpp
// include/ezmk/i18n_keys.def  —— 唯一权威列表
EZMK_I18N_KEY(build_success)
EZMK_I18N_KEY(compiling)
// ... 每个键一行 ...
EZMK_I18N_KEY(repo_list_title)
```

```cpp
// i18n.hpp
enum class I18nKey {
#define EZMK_I18N_KEY(name) name,
#include "ezmk/i18n_keys.def"
#undef EZMK_I18N_KEY
};

// i18n.cpp — key_name() 自动派生，永不失配
const char* key_name(I18nKey key) {
    switch (key) {
#define EZMK_I18N_KEY(name) case I18nKey::name: return #name;
#include "ezmk/i18n_keys.def"
#undef EZMK_I18N_KEY
    }
    return "???";
}
```

**收益**：新增一个键 = 在 `.def` 加一行 + 在两个 JSON 加一条，`key_name()` 永远与枚举同步，`{???}` 类 bug 根除。这也让"枚举名与 JSON 键名必须一致"成为编译期强约束。

> 采用 X-macro 后，可选地在 `init()` 末尾加一个 debug 断言/warning：遍历全部枚举，对 `g_strings` 里缺失的键发出一次告警——便于在开发期发现 JSON 漏翻译（区别于枚举失配）。

### 2.5 帮助页面翻译（`print_usage()`）

`src/cli.cpp` 的 `print_usage()` 中，分节标题（`cli_usage_project` 等）走 i18n，但**每条命令说明与 flag 说明是硬编码英文**（如 `"Create a new project"`、`"Force recompilation of all sources"`）。

**方案**：为帮助正文的每一行说明新增 i18n 键，按现有分节组织。命令行/用法串本身（`ezmk project build [...]`）保持不译（是字面命令），只翻译**尾部说明文字**。新增键示意：

```
help_project_new         "Create a new project"
help_project_build       "Incremental build (compile + link)"
help_project_run         "Build and run"
help_project_clean       "Clean cache and temp files"
help_project_watch       "Watch sources and rebuild on change"
help_pkg_install / remove / search / info / list / update
help_repo_add / remove / update / list / info
help_utils
help_flag_disable_cache / verbose / jobs / profile / auto_update
help_flag_sha256 / yes
help_scope_project / user / global / combined
help_option_syntax_*     （"Option syntax" 段的 3 行说明）
```

翻译后 `print_usage()` 变为：命令用法串（字面）+ `get(help_xxx)` 说明。中文 `ezmk help` 完整本地化。

### 2.6 参数校验报错翻译（`src/cli.cpp`）

`argparse.cpp` 的"未知选项/缺失值"已走 i18n（`cli_unknown_option` / `cli_missing_value`），但 `cli.cpp` 各 `parse_*_args()` 里的业务级校验仍是硬编码英文 `util::fatal(std::string(...))`。需要翻译的消息（按类别，均带 `{cmd}` / `{arg}` 占位符）：

| 现状（cli.cpp 硬编码） | 建议 i18n 键 |
|---|---|
| `'ezmk project new' requires a project name` 等"缺必填位置参数" | `cli_arg_required`（`{cmd}` + `{what}`） |
| `... takes only one X`（多余位置参数） | `cli_too_many_args`（`{cmd}` + `{what}`） |
| `... takes no arguments` | `cli_takes_no_args`（`{cmd}`） |
| `unknown project type: X ...` | 复用 `config_err_invalid_type` 或新增 `cli_unknown_project_type` |
| `unknown project/pkg/repo subcommand: X` | `cli_unknown_subcommand`（`{group}` + `{sub}`） |
| `'ezmk X' requires a subcommand ...` | `cli_requires_subcommand`（`{group}`） |
| `unknown command group: X ...` | `cli_unknown_group`（`{group}`） |
| `... accepts only one scope flag ...` | `cli_one_scope`（`{cmd}`） |
| `'ezmk pkg update' requires a package name or --all` | `cli_update_needs_name_or_all` |
| `'--all' ignores the explicit package name 'X'`（warn） | `cli_all_ignores_name`（`{name}`） |
| `invalid -j value: X` / `'-j' value must be >= 0`（`fill_build_opts`） | `cli_invalid_jobs`（`{val}`）/ `cli_jobs_negative` |

> `{what}` 是位置参数的名词（"project name" / "package name" / "repository name" / "git URL or local path"）。为便于中文本地化，`{what}` 也应取自 i18n（如 `arg_project_name`、`arg_pkg_name`…），或直接为每条消息用独立键，避免把英文名词拼进中文句子。实现时择一，倾向"每条独立键"以获得自然的中文语序。

## 3 命令行改进

### 3.1 命令简写

在**不改变现有命令语义、不影响 zsh 补全**的前提下，新增一层顶层别名，仅在 `cli::parse()` 的分发前做展开。

#### 3.1.1 别名表

| 简写 | 展开为 | | 简写 | 展开为 |
|---|---|---|---|---|
| `pn` | `project new` | | `ki` | `pkg install` |
| `pb` | `project build` | | `kr` | `pkg remove` |
| `pr` | `project run` | | `ks` | `pkg search` |
| `pc` | `project clean` | | `kn` | `pkg info` |
| `pw` | `project watch` | | `kl` | `pkg list` |
| `ra` | `repo add` | | `ku` | `pkg update` |
| `rr` | `repo remove` | | `u` | `utils` |
| `rl` | `repo list` | | `h` | `help` |
| `ru` | `repo update` | | `v` | `version` |
| `ri` | `repo info` | | | |

> 命名规律：`project`→`p*`、`pkg`→`k*`（p 已被 project 占用）、`repo`→`r*`；单字母 `u`/`h`/`v` 对应 utils/help/version。

#### 3.1.2 实现方案

`cli::parse()`（cli.cpp:425）当前读 `argv[1]` 作命令组、`argv[2]` 作子命令，各 `parse_*_args()` 从固定下标 3 起解析。别名展开最干净的做法是在 `parse()` 开头**把 argv 规范化**：

1. 建一个 `static const std::map<std::string_view, std::pair<const char*, const char*>>` 别名表（值为 `{group, action}`，单字母别名 action 为 `nullptr`）。
2. 若 `argv[1]` 命中别名，构造一个新的 `std::vector<std::string>`（`argv[0]` + 展开后的 group[/action] + `argv[2..]`），再据此驱动现有 `parse_*_args()`。
3. 因 `parse_*_args()` 直接吃 `char**`，规范化后需生成 `std::vector<char*>` 传入；把新 argv 的生命周期留在 `parse()` 栈上即可。

**边界**：
- `ezmk pn`（无项目名）应复用现有"缺参数"报错（2.6），报错信息用**展开后**的命令名（`ezmk project new`），保持一致。
- 别名仅在 `argv[1]` 位置生效；`ezmk project pn` 不视为别名（`pn` 会当作 project 的未知子命令报错，符合预期）。
- `v` 与已有的 `-V`/`--version`/`version` 并存；`h` 与 `-h`/`--help`/`help` 并存。

#### 3.1.3 帮助页展示

按计划要求，**不写入 zsh 补全**，而是在 `print_usage()` 的每条命令旁标注简写，如：

```
Project management:
  ezmk project new    (pn)  <name>        Create a new project
  ezmk project build  (pb)  [flags]       Incremental build
  ...
```

### 3.2 `--color=<mode>`

新增全局 `--color=<mode>` 选项，覆盖当前"仅 `NO_COLOR` + `isatty`"的自动判定（`src/util.cpp` 的 `supports_color()`）。

**取值**（大小写不敏感）：

| mode | 别名 | 行为 |
|---|---|---|
| `always` | `enable` | 强制彩色输出 |
| `auto` | `default` | 交互式（TTY）才彩色（**默认**，即现状） |
| `never` | `disable` | 禁用彩色 |

#### 3.2.1 实现方案

1. `include/ezmk/util.hpp`：新增 `enum class ColorMode { Auto, Always, Never }` 与 `void set_color_mode(ColorMode)`；`supports_color()` 改为：
   - `Always` → `true`
   - `Never` → `false`
   - `Auto` → 现有逻辑（`NO_COLOR` + `isatty`/`GetConsoleMode`）
2. **优先级**：显式 `--color=always|never` **高于** `NO_COLOR` 环境变量；`--color=auto` 时才尊重 `NO_COLOR`。这与 `git`/`ls` 的约定一致。
3. **解析位置**：`--color` 是全局选项，可出现在任意子命令上。但 `argparse.cpp` 的 per-command spec 会把未声明的 `--color` 当未知选项报错。因此在 `cli::parse()` 分发**之前**做一次全局预扫描：
   - 识别 `--color=<mode>` 与 `--color <mode>` 两种写法（`--color always`）；
   - 调用 `util::set_color_mode(...)`；
   - 从 argv 中**移除**该 token，使后续 per-command 解析看不到它。
   - 非法 mode → 走 2.6 的 i18n 报错（新增 `cli_invalid_color`，`{val}`）。
4. `main()` 中 `set_color_mode` 需在任何输出（含预扫描自身的报错）之前生效，故预扫描应在 `i18n::init()` 之后、命令分发之前完成。

> 备选：把 `--color` 加进每个命令的 `OptionSpec` 表。因需改十余处且语义重复，不采用；全局预扫描更集中。

## 5 注意事项

- **翻译修复的根因是代码（映射），不是数据（JSON）**：不要只往 JSON 加词——`en.json`/`zh.json` 已有这些键。真正要动的是 `key_name()`（或改为 X-macro 单一数据源）。
- **单一数据源优先**：X-macro 方案一次性根除"枚举 / switch / JSON 三处失配"，强烈建议在本版本落地，否则下个版本加键很可能再触发 `{???}`。
- **简写不进补全**：按计划明确要求，`completions/_ezmk` 不新增别名（补全仍用全名，语义清晰）；简写仅在帮助页展示。
- **`--color` 与 `NO_COLOR` 的优先级**：显式 flag 覆盖环境变量，仅 `auto` 尊重 `NO_COLOR`；需在文档中写清。
- **无破坏性**：命令简写、`--color`、报错/帮助翻译均为增量，现有命令行写法与脚本不受影响。
- **构建依赖 Python**：新增 i18n 键后必须重跑 `build.sh`（内部调用 `scripts/embed_locale.py` 重新嵌入 locale）；若构建环境无 Python，将退化为英文兜底（`build.sh` 已有该分支）。
