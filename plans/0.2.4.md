# EazyMake 0.2.4 — 健壮性与完善

> 所有条目均已在对应版本中实现。

---

## 1 背景

0.2.3 完成了开发者体验提升（并行编译、构建 Profile、Hooks、Watch 模式、pkg list/update）。经过对 `docs/`、`plans/` 及全部源代码的系统性审查，发现以下问题需要在 0.2.4 中解决：

- **正确性缺陷**：版本比较逻辑不一致、Shell 注入风险、`/tmp` 路径硬编码
- **工程规范**：LICENSE 命名不规范、计划文件 checkbox 未勾选
- **代码质量**：`build_project()` 臃肿、CLI 解析重复、帮助文本未 i18n
- **测试覆盖**：核心函数（`build_project`、`pkg::install` 等）零集成测试
- **已知限制**：C23 未支持、MSVC 标志映射不全、`pkg update` 仅支持单包

本版本聚焦于**修复已知缺陷 + 补全明显缺失**，为后续大功能（多目标构建、project test 等）打好基础。

## 2 Bug 修复

### 2.1 版本比较逻辑统一

**现状**：`pkg.cpp:849` 使用简单字符串比较，`repo.cpp:220-245` 使用数值分段比较。同一项目内两种逻辑不一致会导致 `pkg update` 误判"已是最新"。

**目标**：抽取统一的 `compare_version()` 函数，支持 SemVer 三段式数值比较（major.minor.patch），替换两处实现。

**实施步骤**：

- [x] 在 `include/ezmk/pkg.hpp`（或新建 `include/ezmk/version.hpp`）中声明 `compare_version(a, b)` → `int`（-1/0/1）
- [x] 实现三段式数值比较（支持 `1.2.3` 格式，忽略预发布标签）
- [x] `src/repo.cpp` 中 `search_package()` 的版本比较改用新函数
- [x] `src/pkg.cpp` 中 `update()` 的版本比较改用新函数
- [x] 单元测试：`test_pkg.cpp` 新增 version compare 测试（`1.10.0` vs `9.0.0` 等边界）

### 2.2 Shell 注入风险修复

**现状**：`build.cpp` 中编译/链接命令构造对用户路径的 shell 转义不一致。`macros_to_flags()`（line 68-80）使用了 `escape_shell_arg()`，但链接命令构造处未统一使用。

**目标**：审计 `build.cpp` 中所有拼入命令行的外部路径/标志，统一使用 `escape_shell_arg()`。

**实施步骤**：

- [x] 审计 `build.cpp` 中 `make_gcc_compile_cmd()`、`make_gcc_link_cmd()`、MSVC 对应函数
- [x] 对所有文件路径、包名路径使用 `escape_shell_arg()` 包装
- [x] 单元测试：`test_build.cpp` 中验证含空格/特殊字符的路径正确转义

### 2.3 `/tmp` 硬编码修复

**现状**：`src/util.cpp:622-623` 硬编码 `/tmp/ezmk_stdout_XXXXXX` 和 `/tmp/ezmk_stderr_XXXXXX`。在非 MSYS2 Windows 环境（原生 cmd/PowerShell + MSVC）中 `/tmp` 不存在。

**目标**：使用平台适配的临时目录。

**实施步骤**：

- [x] `#ifdef _WIN32`：使用 `GetTempPathA()` + `GetTempFileNameA()`
- [x] 非 Windows：保持 `mkstemp()` 但使用 `$TMPDIR` 或 `/tmp` fallback
- [x] 确保临时文件在进程退出前清理

## 3 代码质量改进

### 3.1 `build_project()` 重构

**现状**：`src/build.cpp` 中 `build_project()` 超过 530 行，混合了编译、链接、钩子执行、包处理多种职责。难以测试和维护。

**目标**：拆分为 3 个子函数，保持外部接口不变。

**实施步骤**：

- [x] 提取 `compile_phase(config, cache, options)` → 处理源文件收集 + 缓存检查 + 编译
- [x] 提取 `link_phase(config, objects, options)` → 处理链接命令构造和执行
- [x] 提取 `run_hooks_phase(config, hook_type, ctx)` → 统一钩子调用逻辑
- [x] `build_project()` 变为编排函数，依次调用上述三个 phase
- [x] 回归测试：现有 400+ 用例全部通过

### 3.2 CLI 标志解析去重

**现状**：`cli.cpp:159-288` 中 `--disable-cache`/`--verbose`/`-j`/`--profile` 的解析代码在 `project build`、`project run`、`project watch` 之间复制了 3 次。

**目标**：提取共享解析函数。

**实施步骤**：

- [x] 新增 `parse_build_options(args, opts)` 静态函数
- [x] `project build`、`project run`、`project watch` 的命令解析改为调用共享函数
- [x] 验证三个命令的 flag 行为完全一致

### 3.3 CLI 帮助文本 i18n

**现状**：`cli.cpp:588-628` 帮助文本为硬编码英文，未使用 i18n 系统。

**目标**：将帮助文本注册为 I18nKey，支持中英文切换。

**实施步骤**：

- [x] 为 `print_usage()` 中的各节标题新增 I18nKey（`cli_usage_*`）

## 6 设计决策

- **版本比较统一**：采用简单的三段式数值比较（major.minor.patch），不引入完整 SemVer 解析器（预发布/构建元数据在包分发场景中极少使用）。完整 SemVer 留待 lockfile 功能时统一考虑。
- **`build_project()` 拆分粒度**：三个 phase 提供足够的关注分离，同时保持调用关系清晰。不做更细粒度的拆分（如独立的 `build_compile.cpp` 文件）以避免过度工程。
- **MSVC 标志映射策略**：对于 GCC 标志在 MSVC 下无明确等效项的场景（如 `-march=native`），以 warning 形式提示用户手动使用 `msvc_flags`，而非静默忽略或强制映射。
- **`pkg update --all` 版本检查**：依赖 2.1 的统一版本比较。每次更新前自动执行 `repo update` 确保索引最新（来自 `docs/repo.md` 的建议）。
- **向后兼容**：所有改动保持 CLI 接口和 `ezmk.toml` 格式不变。`pkg update` 不加 `--all` 时行为与 0.2.3 完全一致。

## 8 开放问题

1. **完整 SemVer 解析**：当前采用简化的三段式比较，预发布标签（`-alpha`、`-beta`）和构建元数据（`+build`）被忽略。这对于当前包分发场景是足够的，但 lockfile 功能可能需要完整 SemVer。延后到 0.3.0。
2. **更多重构粒度**：`build.cpp` 拆分为三个 phase 后是否进一步提取到独立文件？短期观察三个 phase 的增长情况再决策。
3. **临时文件清理策略**：`/tmp` 修复后，MSVC 环境下临时文件的清理由 `util::run_command()` 负责，但进程崩溃时可能残留。未来是否需要全局 temp 目录管理？
