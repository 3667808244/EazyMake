# EazyMake 0.2.1 — MSVC 支持

> 所有条目均已在对应版本中实现。

本版本为 `0.1.8 跨平台支持与编译器探测` 计划第 4 节的收尾，实现 MSVC（Microsoft Visual C++）编译器的完整编译流程支持。

---

## 1 现状与差距

### 1.1 已支持的部分

- `0.1.8` 中的编译器探测可检测到 `cl.exe` 的存在并给出 info 提示
- 现有的项目结构（`src/`、`include/`、`ezmk.toml`）是平台无关的
- 配置系统（`config.hpp` / `config.cpp`）的语言解析与编译器无关
- 缓存系统（`cache.hpp` / `cache.cpp`）的哈希逻辑与编译器无关

### 1.2 尚未支持的部分（需本版本实现）

| 差异维度 | GCC/MinGW/Clang | MSVC (cl.exe) |
|---------|----------------|---------------|
| 编译器可执行文件 | `g++` / `gcc` / `clang++` / `clang` | `cl.exe` |
| 链接器 | `ld`（通过 `g++` 调用） | `link.exe` |
| 静态库工具 | `ar rcs` | `lib.exe` / `link /LIB` |
| 目标文件后缀 | `.o` | `.obj` |
| 静态库后缀 | `.a` | `.lib` |
| 动态库后缀 | `.dll`（MinGW）/ `.so`（Linux） | `.dll` + `.lib`（导入库） |
| 可执行文件后缀 | `.exe`（MinGW）/ 无后缀（Linux） | `.exe` |
| 编译标志风格 | `-Wall`、`-O2`、`-std=c++17` | `/W4`、`/O2`、`/std:c++17` |
| 标准库标志 | `-std=c++17` | `/std:c++17` |
| Include 路径 | `-I<path>` | `/I<path>` |
| 链接路径 | `-L<path>` | `/LIBPATH:<path>` |
| 链接库 | `-l<name>` | `<name>.lib` |
| 依赖文件生成 | `-MMD -MF <file>.d` | `/showIncludes`（解析输出） |
| 动态库标志 | `-shared` | `/DLL` |
| PIC 标志 | `-fPIC` | 不需要（默认 PIC） |
| 输出指定 | `-o <file>` | `/Fe<file>`（exe/dll）或 `/Fo<file>`（obj） |
| 预处理器输出 | `-E` | `/E` |
| 定义宏 | `-D<name>=<value>` | `/D<name>=<value>` |
| 调试符号 | `-g` | `/Zi` |
| 优化标志 | `-O0`/`-O1`/`-O2`/`-O3` | `/Od`/`/O1`/`/O2`/`/Ox` |
| 警告标志 | `-Wall -Wextra` | `/W4` |
| 平台 SDK | 系统头文件 (MinGW) | Windows SDK + MSVC CRT |

## 2 设计原则

1. **配置透明**：`ezmk.toml` 中的 `[compile].flags` 和 `[link].flags` 继续使用 GCC 风格语法（`-Wall`、`-l<lib>` 等）。MSVC 编译时，内部做标志翻译
2. **用户可选**：用户可以在 `ezmk.toml` 中指定 MSVC 风格 flag，也可以在 `[compile].flags` 中混用（不推荐），MSVC 模式下忽略无法翻译的 GCC flag
3. **增量实现**：优先实现最常用的 `executable` 和 `static` 类型，`shared` 类型可选支持
4. **不改变包管理**：包编译（`src/pkg.cpp` 中的 `compile_package()`）仍使用 GCC/MinGW，因为包分发以 GCC 编译为目标

## 4 与现有关键模块的交互

### 4.1 缓存系统（`cache.cpp`）

- MSVC 模式下，依赖文件格式从 `.d`（GCC `-MMD` 输出）变为 `/showIncludes` 输出解析
- `record.json` 中的 `compiler` 字段记录 `cl.exe` 或 `g++`
- 不同编译器的缓存互相隔离（不会 GCC 编译的 `.o` 被 MSVC 复用）
- 缓存记录的 `compile_opts` 存储翻译后的 MSVC 标志（而非 GCC 原始标志），确保切换编译模式时缓存失效

### 4.2 包管理（`pkg.cpp`）

- 包编译（`compile_package()`）**不**支持 MSVC 模式
  - 原因：现有包以 GCC/MinGW 为目标分发，交叉编译器编译包引入的复杂性过高
  - 如果项目在 MSVC 模式下构建，包仍需 MinGW g++ 来编译
  - 如果 MinGW 不可用 → 尝试 MSVC 编译包 → warn（可能不兼容）
  - 未来版本可添加包级别的编译器声明（`ezmk.toml` 中 `[project].compiler = "msvc"`）

### 4.3 构建脚本（`build.sh`）

- `build.sh` 本身使用 GCC/MinGW 编译 ezmk（不变）
- ezmk 自身始终用 GCC/MinGW 编译（嵌入 Lua 等依赖与 MinGW 兼容）
- MSVC 支持仅针对用户项目（`ezmk project build` 的目标项目）

## 5 注意事项

- **MSVC 环境变量**：`cl.exe` 的正常运行依赖 VS 开发者命令提示符设置的环境变量（`INCLUDE`、`LIB`、`LIBPATH` 等）。ezmk 需要在启动时检测 MSVC 环境是否已配置：
  - 若用户从 "Developer Command Prompt for VS" 启动终端 → 环境已就绪
  - 若用户从普通终端启动 → ezmk 应自动执行 `vcvars64.bat` 来加载环境
  - `vcvars64.bat` 执行后的环境变量通过 `util::run_command()` + 解析 `set` 输出来捕获（或直接将编译命令包裹在 `vcvars64.bat && cl ...` 中）
- **链接顺序**：MSVC 的 `link.exe` 对库的链接顺序不敏感（与 GCC 不同），但导入库必须在对应的 DLL 之前
- **MSVC 运行时库选择**：
  - `/MD`：动态链接 CRT（默认，推荐）
  - `/MT`：静态链接 CRT
  - 用户可通过 `[link].msvc_flags` 指定
- **标准库差异**：MSVC 的 STL 实现与 libstdc++ 有差异，某些 C++ 标准特性可能表现不同——这是编译器差异，非 ezmk 问题
- **Windows SDK 版本**：编译时需要 Windows SDK 头文件和库；通过 VS 安装器自动获得
- **MinGW/MSYS2 用户不受影响**：`detect_toolchain()` 在 MinGW 环境下返回 GCC 家族，编译流程不变
- **标志翻译的 warn**：无法翻译的 GCC 标志不应阻止编译（仅 warn），因为用户可能在 `flags` 中混用了 GCC 和 MSVC 通用的标志（如 `/D` 在 MSVC 中也是 `/D`，会被尝试翻译失败后 warn — 需要特殊处理）
- **跨平台兼容性**：`ezmk.toml` 中的 `msvc_flags` 在 Linux/macOS 上被忽略（无需修改配置文件即可跨平台编译）
- **build.sh 不变**：`build.sh` 仍然使用 GCC/MinGW 编译 ezmk 自身（不引入 MSVC 构建 ezmk 自身的流程）
- **项目创建默认值**：`project new` 生成的 `ezmk.toml` 不包含 `msvc_flags`，保持默认配置简洁


> 所有条目均已在对应版本中实现。
