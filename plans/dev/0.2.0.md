# EazyMake 0.2.0 — Lua 工具链

> 所有条目均已在对应版本中实现。

Lua 嵌入式脚本引擎，将 `ezmk utils <name>` 从占位实现改造为完整的插件系统。

---

## 1 背景

0.1.5 的 `ezmk utils <name>` 是占位实现（`main.cpp` 第 106-109 行输出 "utils subcommand is reserved for future use"）。本版本将其改造为基于 **Lua** 的插件系统：ezmk 内嵌 Lua 解释器，utils 工具以 Lua 脚本形式分发。相比 shell 方案：

| 对比维度 | Shell (.sh/.ps1/.bat)        | Lua 内嵌                              |
| -------- | ---------------------------- | ------------------------------------- |
| 跨平台   | 需为每个平台编写不同脚本     | 一份 `.lua` 到处运行                  |
| 依赖     | 依赖 bash / powershell / cmd | 无外部依赖（lua 编译进 ezmk）         |
| 功能边界 | 只能调用外部命令             | 可调用 ezmk 暴露的 C++ API + 外部命令 |
| 错误处理 | 各平台语法不同，易出错       | 统一的 `pcall` / `error()`            |
| 分发     | 文本脚本，但需注意换行符     | 纯文本，天然跨平台                    |

## 2 Lua 版本与集成方式

- 使用 **Lua 5.4.7**（当前最新的 Lua 5.4.x 稳定版）
- `pkg/lua.tar.gz` 需确认：
  - [x] 验证 `pkg/lua.tar.gz` 文件存在且版本正确
  - [x] 若不存在，从 https://www.lua.org/ftp/lua-5.4.7.tar.gz 下载
- 集成方式：**静态链接**进 `ezmk` 二进制
  - 头文件 → `include/vendor/lua/`（`lua.h`、`lualib.h`、`lauxlib.h`、`luaconf.h`）
  - 源文件 → `src/vendor/lua/`（`lapi.c`、`lcode.c`、`lctype.c`、`ldebug.c`、`ldo.c`、`ldump.c`、`lfunc.c`、`lgc.c`、`llex.c`、`lmem.c`、`lobject.c`、`lopcodes.c`、`lparser.c`、`lstate.c`、`lstring.c`、`ltable.c`、`ltm.c`、`lundump.c`、`lvm.c`、`lzio.c`、`lauxlib.c`、`lbaselib.c`、`lcorolib.c`、`ldblib.c`、`liolib.c`、`lmathlib.c`、`loadlib.c`、`loslib.c`、`lstrlib.c`、`ltablib.c`、`lutf8lib.c`、`linit.c`）
  - 注意：`linit.c` 需修改以限制可用的标准库（如移除 `os.execute` 和 `io.popen`）
- `build.sh` 更新：Lua 源文件加入 `SRC` 通配符（`src/vendor/lua/*.c`）
  - Lua 使用 `-DLUA_COMPAT_5_3` 编译以保持部分向后兼容

## 3 ezmk Lua API 设计

ezmk 向 Lua 脚本暴露一个 `ezmk` 全局模块（通过 Lua C API 注册为全局 table），提供以下函数。所有路径均返回正斜杠格式的绝对路径。

### 3.1 项目信息（只读）

```lua
ezmk.project_root()       -- → string   项目根目录绝对路径
ezmk.project_name()       -- → string   项目名（ezmk.toml [project].name）
ezmk.project_type()       -- → string   "executable" | "static" | "shared"
ezmk.project_config()     -- → string   ezmk.toml 绝对路径
ezmk.build_dir()          -- → string   build/ 目录绝对路径
```

### 3.2 编译选项（只读）

```lua
ezmk.compile_flags()      -- → {string...}  [compile].flags 数组
ezmk.include_dirs()       -- → {string...}  [compile].include_dirs 绝对路径数组
ezmk.link_flags()         -- → {string...}  [link].flags 数组
ezmk.link_dirs()          -- → {string...}  [link].link_dirs 绝对路径数组
```

### 3.3 文件系统

```lua
ezmk.list_sources()       -- → {string...}  src/ 下所有源文件绝对路径
ezmk.file_exists(path)    -- → bool
ezmk.file_read(path)      -- → string | nil, err
ezmk.file_write(path, content)  -- → bool, err
```

- 相对路径以项目根目录为基准
- `file_write` 拒绝写入项目根目录之外的绝对路径（安全限制）
- `file_write` 的父目录自动创建（与 `util::file_write` 行为一致）

### 3.4 进程执行

```lua
ezmk.run(cmd)             -- → {exit_code=int, stdout=string, stderr=string}
ezmk.run_capture(cmd)     -- → string   stdout 内容，失败则 error()
```

- `run_capture` 在 exit code ≠ 0 时抛出 Lua error，包含 stderr 信息
- 内部使用 `util::run_command()`，Windows 上通过 CreateProcess 管道捕获，Linux 上通过临时文件
- **超时**：`run()` 和 `run_capture()` 在当前版本不设超时（由调用方脚本自行处理长时间运行的命令）。后续版本可添加 `ezmk.run(cmd, timeout_ms)` 重载

### 3.5 日志输出

```lua
ezmk.info(msg)            -- 输出到 stderr，带 [ezmk] 前缀（绿色）
ezmk.warn(msg)            -- 输出到 stderr，带 [ezmk warn] 前缀（黄色）
ezmk.error(msg)           -- 输出到 stderr，带 [ezmk error] 前缀（红色）
```

- 颜色行为与 C++ 端 `util::info/warn/error` 一致（受 `NO_COLOR` 和终端检测控制）

### 3.6 路径工具

```lua
ezmk.pkg_dir()            -- → string   当前工具所在包的根目录
ezmk.temp_dir()           -- → string   .ezmk/temp/ 目录绝对路径
ezmk.cache_dir()          -- → string   .ezmk/cache/ 目录绝对路径
```

### 3.7 JSON

```lua
ezmk.json_encode(table)   -- → string   将 Lua table 编码为 JSON
ezmk.json_decode(string)  -- → table    将 JSON 字符串解码为 Lua table
```

- 复用 `nlohmann/json.hpp`（已在 `include/vendor/` 中）
- **空 table 二义性**：Lua 中 `{}` 既可以是空数组也可以是空对象。约定如下：
  - `json_encode`：若 table 的所有 key 是从 1 开始的连续整数 → 编码为 JSON 数组 `[]`；否则 → 编码为 JSON 对象 `{}`
  - 如需强制编码为空对象，使用 `{["_empty"]=true}` 再手动删除
  - `json_decode`：JSON 数组 → Lua 整数 key table（`{[1]=..., [2]=...}`）；JSON 对象 → Lua 字符串 key table

## 4 utils 包结构

utils 工具以包形式分发（遵循 `docs/pkg.md`），`utils/` 目录下放置 Lua 脚本：

```
<utils_pkg>/
    ezmk.toml             # [project] name, version, type
    utils/
        <name>.lua        # 入口脚本（文件名即工具名）
        lib/              # 可选的 Lua 辅助模块（纯 Lua 文件）
    include/              # 可选：C/C++ 头文件
    src/                  # 可选：C/C++ 源码（编译为 .a 供 Lua 通过 FFI 调用）
```

**`ezmk.toml` 格式**：

```toml
[project]
name = "ezmk-cc"
version = "0.1.0"
type = "utils"              # 标记为工具包

[utils]
tools = ["cc", "compile-commands"]   # 本包提供的工具名列表
```

| 字段 | 类型 | 必须 | 说明 |
|---|---|---|---|
| `[project].type` | string | 是 | 取值 `"utils"`，标记为工具包 |
| `[utils].tools` | string[] | 是 | 提供的工具名列表，每个对应 `utils/<name>.lua` |

- 若包同时包含 `src/` 和 `utils/`：先编译 `src/` → `build/*.a`，再注册 tools
- 若包仅有 `utils/`（无 `src/`）：安装时跳过编译，仅解压到目标路径
- **注意**：0.2.0 计划中 `entry` 字段（可选默认入口）已移除——简化设计，取 `tools[0]` 为默认入口

## 5 Lua 入口脚本约定

每个工具入口脚本（如 `cc.lua`）需实现以下函数：

```lua
-- 可选：返回帮助文本
function help()
    return [[
usage: ezmk utils cc [options]

Generate compile_commands.json for the current project.

Options:
  -o, --output  <path>   Output path (default: <project_root>/compile_commands.json)
  -h, --help             Show this help
]]
end

-- 必选：入口函数，args 为字符串数组（ezmk utils <name> 之后的部分）
-- 返回整数 exit code（0 = 成功），或调用 error("msg") 让 ezmk 处理
function run(args)
    -- args is a Lua table of strings: {[1]="-o", [2]="custom/path.json", ...}
    -- args may be empty: {}
    return 0
end
```

**调用流程**：
1. ezmk 找到 `utils/<name>.lua`
2. 加载并执行脚本（`luaL_dofile`），注册 `help()` 和 `run()`
3. 若 args 包含 `-h` / `--help`，调用 `help()` 输出帮助到 stdout
4. 否则调用 `run(args)`，透传剩余参数（Lua table 格式，整数 key 从 1 开始）
5. `run()` 返回整数 → 作为进程 exit code
6. `run()` 中调用 `error("msg")` → C++ 层捕获 Lua 异常，通过 `util::error()` 输出并返回 exit code 1

## 6 查找顺序

`ezmk utils <name>` 按以下优先级查找 `<name>.lua`：

| 优先级      | 路径                                        | 说明                       |
| ----------- | ------------------------------------------- | -------------------------- |
| 1（项目级） | `<project>/.ezmk/pkg/*/utils/<name>.lua`    | 扫描项目作用域所有已安装包 |
| 2（用户级） | `~/.local/ezmk/pkg/*/utils/<name>.lua`      | 扫描用户作用域所有已安装包 |
| 3（全局级） | `<ezmk_install_dir>/pkg/*/utils/<name>.lua` | 扫描全局作用域所有已安装包 |

- 每个作用域内扫描该作用域下所有已安装包的 `utils/` 子目录，按包名字典序
- 同一作用域找到第一个匹配的 `<name>.lua` 即停止
- 查找基于文件系统：直接检测 `utils/<name>.lua` 是否存在（不依赖 `ezmk.toml` 中的 `[utils].tools`，toml 声明仅用于 `pkg info` 展示）
- 实现函数：`fs::path find_utils_script(const std::string& name)`，放在 `util.cpp` 或新建 `src/lua_api.cpp` 中

## 7 安装与卸载

与普通包完全相同的流程（参见 `docs/pkg.md`）：

```bash
# 从本地文件安装
ezmk pkg install -u ./ezmk-cc-0.1.0.zip

# 从 URL 安装
ezmk pkg install -g https://example.com/tools/ezmk-cc-0.1.0.tar.gz

# 从已注册仓库安装
ezmk pkg install -p ezmk-cc

# 卸载
ezmk pkg remove -u ezmk-cc
```

安装后工具立即可用；卸载后自动不可用。

## 9 安全性

Lua 脚本运行在 **sandbox 环境** 中：

- **Sandbox 环境表**：每个脚本运行在独立的 sandbox 环境表中（而非 `_G` 全局表）：
  ```cpp
  // 创建 sandbox 环境
  lua_newtable(L);                    // sandbox env table
  lua_newtable(L);                    // metatable
  lua_getglobal(L, "_G");
  lua_setfield(L, -2, "__index");     // fallback to _G for read
  lua_setmetatable(L, -2);
  // 将 ezmk table 注入 sandbox
  register_ezmk_table(L);             // sandbox.ezmk = {...}
  // 加载脚本时绑定 sandbox 环境
  luaL_loadfile(L, script_path);
  // 设置脚本 chunk 的 _ENV 为 sandbox
  lua_pushvalue(L, env_idx);
  lua_setupvalue(L, -2, 1);
  ```
- **移除危险函数**：
  - `os.execute` 和 `io.popen` 在 `linit.c` 编译期移除（见阶段 A）
  - 不暴露 `require` 加载 C 扩展的能力（`package.loadlib` 已移除）
  - `dofile` / `loadfile` 在 sandbox 环境中不可用（sandbox 环境的 `_G` 只读）
- **文件系统写入限制**：`ezmk.file_write()` 内部检查：
  - 若路径为绝对路径且不在项目根目录下 → 返回 `false, "access denied: write outside project root"`
  - 相对路径以项目根目录为基准解析
- **进程执行限制**：
  - 所有外部命令通过 `ezmk.run()` 执行（可记录和审计）
  - 当前版本不限制可执行命令的白名单（后续版本可添加）
  - Shell 注入防护：`ezmk.run()` 内部使用 `util::run_command()`，命令字符串通过 `CreateProcess`（Windows）或 `system()`（Linux）执行。调用方 Lua 脚本负责正确转义参数
- **沙箱隔离**：每次 `run_script()` 调用创建新的 sandbox 环境，前一次调用的全局变量不会污染后续调用

## 10 注意事项

- Lua 状态机在 ezmk 进程生命周期内全局复用（一个 `lua_State*`），存储在全局或命名空间级别变量中
- 每次 `run_script()` 前创建新的 sandbox 环境表，避免脚本间的全局变量污染（见 9. 安全性）
- `ezmk.file_write()` 和 `ezmk.file_read()` 的路径参数若为相对路径，以项目根目录为基准
- JSON 编解码处理 Lua table 与 JSON 的映射：空数组 `[]` vs 空对象 `{}` 的二义性按 3.7 节约定处理
- `lua.tar.gz` 中的 Lua 源码需要确认版本号（5.4.x），并在 `ezmk.toml` 的 vendor 依赖或 `build.sh` 的注释中记录
- `build.sh` 中 Lua 源文件编译需要添加 `-DLUA_COMPAT_5_3` 宏以保持部分 Lua 5.3 API 兼容
- Lua 的 `luaL_checkstring` 等函数内部会调用 `luaL_error`（longjmp），C++ 端需确保没有 RAII 对象跨越 lua_pcall 边界（使用 `lua_pcall` 在保护模式下调用，longjmp 被捕获）
- `cc.lua` 作为首个内置工具，应编写足够好的错误处理，作为后续工具开发的参考模板
- **0.1.7 (i18n) 集成**：如果 0.1.7 在 0.2.0 之前完成，Lua API 绑定中的日志函数（`ezmk.info/warn/error`）内部应使用 i18n key 而非硬编码字符串
- **0.1.8 (compiler detection) 集成**：`compile_commands.json` 的 `command` 字段需使用实际的编译器路径（而非默认的 `g++`），Lua API 应能获取检测到的编译器信息
- `find_utils_script()` 的性能：在工具数量较少（< 50 个包）时直接文件系统扫描即可；若后续工具包增多，可添加索引缓存（类似 `record.json`）
- Lua 错误处理：`run()` 中抛出的 Lua error（`error("msg")`）被 C++ 层 `lua_pcall` 捕获，通过 `util::error()` 输出 Lua 错误消息（包含 Lua 栈追踪），返回 exit code 1


> 所有条目均已在对应版本中实现。
