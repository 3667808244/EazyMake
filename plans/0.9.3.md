# EazyMake 0.9.3 — 捆绑包迁移到官方仓库

---

## 1 背景

EazyMake 当前在 `pkg/` 目录下捆绑了 7 个预编译库包（`.tar.gz`），由 `install.sh` 随二进制一同拷贝到 `<exe_dir>/pkg/`，供全局作用域使用。此外还有 1 个内置工具 `ezmk-cc`（编译进 ezmk 本体，不走仓库）。

这些捆绑包是**仓库系统成熟之前的过渡方案**：用户装完 ezmk 就有包可用。但 0.9.1 已建立官方仓库（`ezmk-repo`），并已在 `install.sh` 中预注册，因此这些捆绑包应当迁移到仓库中，让它们通过标准的 `repo → search → install` 流程使用。

| 问题                  | 说明                                                                                |
| --------------------- | ----------------------------------------------------------------------------------- |
| 捆绑包无版本号        | 所有 `ezmk.toml` 缺少 `version` 字段，`pkg info` 无法显示版本                       |
| 旧 TOML 格式          | 使用 `include_dir`（单数）而非 `include_dirs`（复数），缺少 `src_dirs`              |
| 仓库与二进制重复      | 同样的包存在两份：一份在 `pkg/`（捆绑），一份在仓库 `packages/`（迁移后）           |
| 无源码可审计          | 捆绑包只有归档，没有对应的 `sources/` 源工程，不符合 0.9.1 的"源→归档→哈希"一致原则 |
| `install.sh` 多余拷贝 | 已经预注册官方仓库，再拷贝 `pkg/` 属于画蛇添足                                      |

---

## 2 目标

1. 将 7 个捆绑包的源工程提取/重建到 `ezmk-repo/sources/`，补全 `version` 和 TOML 格式。
2. 用 `pack.sh` 生成新归档 + sha256 + 更新 `index.toml`，与既有 `hello-lib` / `example-utils` 共处。
3. `ezmk-cc` **不迁移**（内置工具，编译在 ezmk 二进制内，不属于仓库），但保留 `pkg/ezmk-cc/` 作为内置工具源码参考。
4. 从主仓库移除 `pkg/*.tar.gz`（捆绑归档），`install.sh` 不再拷贝 `pkg/`。
5. 保持向后兼容：迁移后 `pkg install` 与原有捆绑包行为一致。

## 4 迁移逐包要点

### 4.1 通用修正

所有迁移包需统一做以下修正：

- **补 `version`**：查阅各库上游版本（fmt 10.x、spdlog 1.x、sqlite3 3.x 等），写入 `ezmk.toml`
- **TOML 格式更新**：`include_dir` → `include_dirs`，对于有 `src/` 的补 `src_dirs`
- **补 `language`**：C 库（lua, sqlite3）标记 `language = "C"`，其他标记 `"C++17"`
- **审查 flags**：去掉不必要的 `-Wall -O2`（由用户项目控制，包不应硬编码优化级别）

### 4.2 逐包说明

| 包              | 版本建议 | 特殊处理                                                                                              |
| --------------- | -------- | ----------------------------------------------------------------------------------------------------- |
| `catch2`        | v3.4.0   | 大部分 header-only，`src/` 含少量编译单元。需确认上游版本号                                           |
| `fmt`           | 10.2.1   | 成熟库，`include/fmt/` + `src/`。需确认上游版本号                                                     |
| `lua`           | 5.4.7    | 与 ezmk 内嵌同版本。纯 C 库。`language = "C"`                                                         |
| `nlohmann_json` | 3.11.3   | header-only + 一个 stub `src/json_stub.cpp` 用于编译验证                                              |
| `spdlog`        | 1.14.1   | 依赖 `fmt`，`compile.flags` 含 `-DSPDLOG_COMPILED_LIB -DSPDLOG_FMT_EXTERNAL`；`depends.lib = ["fmt"]` |
| `sqlite3`       | 3.45.0   | 单文件 amalgamation（`sqlite3.c` ~8MB），`language = "C"`                                             |
| `tinyxml2`      | 10.0.0   | 轻量，仅 2 个文件（`tinyxml2.h` + `tinyxml2.cpp`）                                                    |

> 版本号需通过查阅各包的实际源码（头文件中的版本宏、CHANGELOG 等）确认，上表为估计值。

### 4.3 依赖关系

```
spdlog ──→ fmt
```

其它包无互相依赖。`pack.sh` 按顺序排列 `index.toml` 不影响（ezmk 客户端自己解决依赖）。

## 6 兼容性

| 场景                 | 影响                                        | 处理                                                           |
| -------------------- | ------------------------------------------- | -------------------------------------------------------------- |
| 新用户 `install.sh`  | `pkg/` 不再有捆绑归档，但已预注册官方仓库   | 用户 `pkg install <name>` 自动从仓库拉取，无感知               |
| 旧用户已安装的捆绑包 | 已安装到 `<exe_dir>/pkg/` 或用户/项目作用域 | 不受影响，`pkg list` 仍可见。`pkg update` 会从仓库拉取新版本   |
| 离线环境             | 捆绑包删除后，无网络时无法从仓库安装        | 需提前 `repo add` 本地镜像或保留离线包。可在文档中给出离线方案 |
| `ezmk utils cc`      | 内置工具，不受仓库影响                      | 仍然可用                                                       |

## 9 开放问题

1. **版本号确认**：捆绑包制作时的上游版本号已无从追溯（原 `ezmk.toml` 无版本字段）。需逐包打开头文件版本宏确认，或重新打包已知版本。
2. **`install.sh` 中 `pkg/` 拷贝的过渡策略**：是直接移除还是保留一个版本过渡期？建议直接移除——`install.sh` 已预注册仓库，用户无感知。
3. **`ezmk-cc` 的归属**：当前 `pkg/ezmk-cc/` 目录保留在源码中作为内置工具参考，但实际代码编译进 `src/lua_api.cpp`。是否需要同步更新其 `ezmk.toml` 格式以保持一致性？
4. **包的更新机制**：迁移后，用户如何更新这些包？当前 `pkg update` 从仓库取最高版本——迁移后如果仓库有新版本，用户可正常更新。需确保初始版本号设置合理（太低则仓库更新不生效）。
5. **Catch2 的 header-only 特性**：Catch2 v3 主要是 header-only，但当前包里有少量 `.cpp` 编译单元。迁移时是否保持原样（含编译单元），还是改为纯 header-only？建议保持原样以兼容既有使用方式。
