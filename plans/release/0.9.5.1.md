# EazyMake 0.9.5.1 — 代码重构与质量清理

---

## 1 背景

0.9.5 完成了跨平台体验与质量保障（install.ps1 + 集成测试），在 0.9.6 开始新功能之前，进行一次集中的代码质量清理。

经过四维度代码审查（项目结构、源码质量、测试覆盖、文档与构建），识别出以下技术债务优先级：

| 优先级 | 类别 | 影响 |
|--------|------|------|
| **P0** | 代码重复（DRY） | 约 600+ 行重复逻辑，bug 修复需改两处 |
| **P0** | 资源管理 | 原始指针泄漏风险（lua_api, file_watcher） |
| **P1** | 全局可变状态 | lua_api 中 5 个全局变量无同步保护 |
| **P1** | 测试盲区 | compare_version、extract_archive 等核心函数零测试 |
| **P2** | 魔术字符串 | .ezmk 路径散落 7+ 个文件 |
| **P2** | 死代码 | ParsedOptions::count()、native_path() |
| **P3** | 测试辅助重复 | TempDir/CwdGuard/EnvGuard 在 3+ 文件中各自定义 |

本版本**不新增用户可见功能**，仅做代码级清理。所有行为应与 0.9.5 完全一致。

---

## 2 目标

1. **消除主要代码重复** — link_phase、compile_sources/compile_one_source、debounce 刷新
2. **修复资源管理问题** — 原始指针转 RAII、全局变量加同步保护
3. **补全测试盲区** — compare_version、extract_archive、依赖解析等关键路径
4. **清理死代码与魔术字符串** — 统一路径常量、移除未使用函数
5. **提取共享测试基础设施** — TempDir、CwdGuard 等公共 fixtures

## 3 详细设计

### 3.1 P0：消除代码重复

#### 3.1.1 `build.cpp` — 链接阶段统一

**现状**（`src/build.cpp:732-907`）：

`link_phase()` 中 6 个几乎相同的代码块（static-MSVC / static-GCC / shared-MSVC / shared-GCC / exe-MSVC / exe-GCC），每个块 ~30-40 行，模式相同：创建 temp 路径 → 构建命令 → 运行 → 检查 exit code → 清理/rename。

**重构方案**：提取 `link_with_toolchain()` 模板函数

```cpp
// 通用链接执行器：构建命令 → 执行 → 检查 → 原子重命名
static fs::path execute_link(
    const std::string& cmd,
    const fs::path& output,
    const fs::path& output_tmp,
    const BuildState& st,
    const cli::BuildOptions& opts,
    const config::EzConfig& cfg,
    ezmk::i18n::I18nKey fail_key,
    ezmk::i18n::I18nKey success_key);

// 各项目类型仅需提供命令构建 lambda
if (cfg.project.type == "static") {
    return execute_link(build_static_cmd(...), lib, lib_tmp, ...);
} else if (cfg.project.type == "shared") {
    return execute_link(build_shared_cmd(...), dll, dll_tmp, ...);
} else {
    return execute_link(build_exe_cmd(...), exe, exe_tmp, ...);
}
```

**收益**：减少约 150 行重复代码。

#### 3.1.2 `cache.cpp` — 编译逻辑统一

**现状**（`src/cache.cpp`）：

`compile_one_source()`（第 295-545 行）和 `compile_sources()`（第 551-864 行）包含完全相同的 MSVC/GCC 编译管道代码：命令构建、编译运行、原子重命名、缓存复制、依赖解析。两个函数之间约 200+ 行逻辑重复。

**原因**：`compile_one_source` 是 0.2.3 新增的线程安全版本，从 `compile_sources` 复制而来，未做抽取合并。

**重构方案**：`compile_sources` 内部调用 `compile_one_source`

```cpp
CompileResult compile_sources(const CompileInput& in, CacheRecord& record) {
    CompileResult result;
    for (auto& src : in.sources) {
        auto sr = compile_one_source(src, in, record);
        // 聚合结果...
    }
    return result;
}
```

需要微调：`compile_one_source` 当前接收 `const CacheRecord&`（只读），需要将其中的 record 更新逻辑移出，或在调用方处理。

**收益**：`compile_sources` 从 314 行缩减到约 40 行。

#### 3.1.3 `file_watcher.cpp` — debounce 刷新统一

**现状**（`src/file_watcher.cpp`）：

同样的 "sleep 100ms → 检查 elapsed → flush_pending" debounce 逻辑在 Windows（第 409-428 行）、Linux（第 470-489 行）和 macOS（第 548-567 行）各重复约 20 行。

**重构方案**：抽取为成员函数

```cpp
void FileWatcher::check_and_flush() {
    bool should_flush = false;
    {
        std::lock_guard<std::mutex> lock(pending_mutex_);
        if (!pending_paths_.empty() &&
            last_event_ != std::chrono::steady_clock::time_point{}) {
            auto now = std::chrono::steady_clock::now();
            auto elapsed = std::chrono::duration_cast<std::chrono::milliseconds>(
                now - last_event_).count();
            if (elapsed >= debounce_ms_) should_flush = true;
        }
    }
    if (should_flush) flush_pending();
}
```

三个平台的 `run()` 循环中调用 `check_and_flush()`。

**收益**：减少约 40 行重复代码。

#### 3.1.4 `main.cpp` — 自动更新仓库代码统一

**现状**（`src/main.cpp`）：

`ProjectBuild`、`ProjectRun`、`ProjectWatch` 三个分支中逐字重复了相同的仓库自动更新代码块（`--auto-update` 检测 + `repo::update` 调用），约 10 行 × 3。

**重构方案**：抽取为 `auto_update_repos(const cli::BuildOptions&)` 函数。

**收益**：减少约 20 行重复代码。

#### 3.1.5 `config.cpp` — 配置文件名称校验统一

**现状**（`src/config.cpp`）：

配置文件名称（字母数字+下划线+连字符）的校验逻辑在 `[compile.profile.*]`（第 229-239 行）和 `[link.profile.*]`（第 282-292 行）各实现了一次。

**重构方案**：抽取 `static bool is_valid_profile_name(std::string_view)` 函数。

### 3.2 P0：修复资源管理问题

#### 3.2.1 `lua_api.cpp` — `g_cached_config` 改为 `unique_ptr`

**现状**（`src/lua_api.cpp:327`）：

```cpp
g_cached_config = new config::EzConfig(config::parse_config(cfg_path));
```

在第 935 行通过 `delete` 清理。若 `lua::shutdown()` 未正确调用或异常退出，泄漏整个 `EzConfig` 对象。

**修复方案**：

```cpp
// 旧：config::EzConfig* g_cached_config = nullptr;
// 新：
std::unique_ptr<config::EzConfig> g_cached_config;

// 赋值：
g_cached_config = std::make_unique<config::EzConfig>(config::parse_config(cfg_path));

// shutdown() 中：
g_cached_config.reset();
```

**收益**：消除手动内存管理，异常安全。

#### 3.2.2 `file_watcher.cpp` — `OVERLAPPED*` 改为 RAII

**现状**（`src/file_watcher.cpp:127,232`）：

```cpp
OVERLAPPED* ov = new OVERLAPPED{};
// ... 在 win32_cleanup() 中：
delete static_cast<OVERLAPPED*>(w.overlapped);
```

若 watch 已添加但 worker 未启动时发生异常，`OVERLAPPED*` 泄漏。

**修复方案**：将 `WatchEntry::overlapped` 改为 `std::unique_ptr<OVERLAPPED>`（或内嵌 `OVERLAPPED` 值对象，因为结构体大小固定且可平凡初始化）。

```cpp
struct WatchEntry {
    std::string dir_path;
    HANDLE dir_handle = nullptr;
    std::vector<uint8_t> buffer;
    OVERLAPPED overlapped{};  // 值语义，无需 new/delete
};
```

**收益**：消除原始 new/delete，无泄漏风险。

### 3.3 P1：全局可变状态保护

#### 3.3.1 `lua_api.cpp` — 全局变量同步保护

**现状**：以下全局变量在无锁保护下读写：

| 变量 | 写入者 | 读取者 |
|------|--------|--------|
| `g_current_script_pkg_root` | `run_script()` | `ezmk_pkg_dir()` |
| `g_current_perms` | `run_script()` | `check_*_permission()` |
| `g_in_script_context` | `run_script()` / `run_hook_script()` | 多个 API 入口 |
| `g_ask_allow` / `g_ask_deny` | `resolve_ask()` | 多个 API 入口 |

当前脚本是顺序执行的，暂无竞态条件。但为防御未来引入的并行执行场景，应加文档化说明并考虑轻量同步。

**修复方案**：添加 `static std::mutex g_script_mutex` 保护上述全局变量的读写。或将它们聚合到 `ScriptContext` 结构体中，通过 `ScriptContextGuard`（已有 RAII 守卫，第 973 行）统一管理。

**注意**：此变更需谨慎，引入 mutex 可能影响 Lua 回调路径的死锁风险。当前阶段先加注释说明线程安全性假设即可。

#### 3.3.2 `cache.cpp` — 并行编译的 record 只读假设文档化

`compile_one_source()` 宣称"线程安全 — 在 record 上只读"，但调用方（`build.cpp:673-688`）在所有线程完成前不会修改 record。此不变量未显式文档化。

**修复方案**：在 `compile_one_source()` 和 `build.cpp` 的并行调度处添加注释说明这一不变量。

**现状**：以下辅助类型在多个测试文件中各自定义：

#pragma once
#include "catch2.hpp"
#include <filesystem>
#include <cstdlib>
#include <string>

// RAII 临时目录
struct TempDir {
    std::filesystem::path path;
    TempDir();
    ~TempDir();
};

// RAII 工作目录保护
struct CwdGuard {
    std::filesystem::path original;
    CwdGuard();
    ~CwdGuard();
};

// RAII 环境变量保护
struct EnvGuard {
    std::string name;
    std::string original;
    explicit EnvGuard(const char* name);
    ~EnvGuard();
};

// 写入最小 ezmk.toml 到指定目录
void write_minimal_config(const std::filesystem::path& dir,
                          std::string_view name = "test",
                          std::string_view version = "0.1.0");

## 5 兼容性矩阵

| 变更 | 影响 | 处理 |
|------|------|------|
| `link_phase()` 重构 | 构建行为 | 提取纯函数，行为完全不变 |
| `compile_sources()` 改为调用 `compile_one_source()` | 编译行为 | 逻辑等价，仅消除重复 |
| `g_cached_config` → `unique_ptr` | 内存管理 | 无行为变化，仅更安全 |
| `OVERLAPPED*` → 值对象 | 文件监视 | 无行为变化，仅消除 new/delete |
| 移除 `count()` / `native_path()` | API | 无调用方，零影响 |
| 引入 `ezmk::path::*` 常量 | 路径 | 字符串值相同，仅集中化 |
| 测试辅助共享 | 测试 | 纯增量，不影响产品代码 |
| i18n 补全 | 用户界面 | toolchain/cache 日志现在可以翻译 |

**核心原则**：所有变更必须保持与 0.9.5 **二进制等价**的行为。编译产物在相同输入下产生相同输出。
