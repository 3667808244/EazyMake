# EazyMake 0.9.6 — 功能补全与生态完善

---

## 1 背景

0.9.4 和 0.9.5 分别补全了文档质量与跨平台体验。本版本聚焦 temp.md 分析中识别出的**最后一个核心功能缺口——依赖版本锁定**，以及若干提升开发体验的锦上添花功能：构建进度显示、代码格式化基础设施、启动 Logo。

完成本版本后，EazyMake 在功能、文档、质量、跨平台四个维度上达到 1.0.0 标准。

---

## 2 目标

1. **依赖版本锁定** — `ezmk.toml` 中 `[depends]` 支持 `foo@1.2.3` 和 `foo>=1.0` 版本约束语法
2. **构建进度显示** — 并行编译时显示当前编译的文件名和总体进度
3. **.clang-format / .clang-tidy 基础设施** — 项目根目录添加 `.clang-format` 配置 + `CONTRIBUTING.md` 代码风格指南
4. **ASCII Logo** — `ezmk` 无参数运行时显示项目 Logo
5. **魔术字符串常量化**（0.9.5.1 延后）— 引入 `ezmk::path::*` 内联常量，消除散落在 7+ 文件中的 `.ezmk/temp`、`.ezmk/cache` 等硬编码路径字符串

---

## 3 详细设计

### 3.1 依赖版本锁定

这是 temp.md 用户反馈中最重要的功能需求（"锁定依赖的特定版本是刚需"）。

#### 3.1.1 语法设计

在 `[depends]` 的 `lib` 和 `want` 数组中，包名后可选追加版本约束：

```toml
[depends]
lib = [
    "fmt",              # 无版本约束（取最新，向后兼容）
    "spdlog@1.14.1",    # 精确版本
    "catch2^3.6.0",     # 兼容版本（>=3.6.0, <4.0.0）
    "nlohmann_json~3.11" # 近似版本（>=3.11.0, <3.12.0）
]
```

**约束运算符**：

| 语法 | 含义 | 示例 |
|---|---|---|
| `@1.2.3` | 精确版本 | `fmt@10.2.1` |
| `^1.2.3` | 兼容版本（主版本不变） | `^3.6.0` → `>=3.6.0, <4.0.0` |
| `~1.2.3` | 近似版本（次版本不变） | `~3.11.0` → `>=3.11.0, <3.12.0` |
| `>=1.2.3` | 大于等于 | `>=1.0.0` |
| `>1.2.3` | 大于 | `>1.0.0` |

**设计原则**：
- 向后兼容：不加约束的包名行为不变（取最高版本）
- 不引入锁定文件（`ezmk.lock`）——留待后续版本；1.0 之前保持简单
- `want`（可选依赖）同样支持版本约束

#### 3.1.2 实现方案

**数据结构**（`include/ezmk/config.hpp`）：

```cpp
struct VersionConstraint {
    enum Op { None, Exact, Compatible, Approx, Gte, Gt };
    Op op = None;
    std::string version;  // "1.2.3"
};

struct DependsEntry {
    std::string name;
    VersionConstraint constraint;
};

struct DependsSection {
    std::vector<DependsEntry> libs;   // 替代 std::vector<std::string>
    std::vector<DependsEntry> want;
};
```

**解析**（`src/config.cpp`）：
- 复用现有 `extract_string_array()` 签名
- 新增 `parse_depends_entry(raw: string) -> DependsEntry`
- 解析逻辑：检测首个 `@` / `^` / `~` / `>=` / `>` 运算符，拆分包名和版本号

**依赖解析**（`src/pkg.cpp`）：
- 复用现有 `compare_version()` 工具函数（`src/version.cpp`）
- 在拓扑排序后的版本选择阶段，对每个 `DependsEntry` 应用约束过滤
- 若无匹配版本 → `missing dependency` 错误（附带约束信息）
- 若精确版本不存在 → 明确报错 "foo@1.2.3 not found, available: 1.2.0, 1.3.0"

**兼容性处理**：
- `ezmk.toml` 中 `lib = ["foo"]`（旧格式，字符串数组）继续正常工作
- 解析时 `DependsEntry` 的 `op == None` 即无约束

#### 3.1.3 测试

- 解析测试：每种运算符 + 边界情况（缺少版本号、无效版本号、多余空格）
- 解析测试：新旧格式混用（`"foo"` + `"bar@1.0"` 在同一数组）
- 解析测试：纯旧格式 `lib = ["foo", "bar"]`
- 依赖解析测试：精确版本匹配 / 不匹配 / 兼容范围匹配 / 无可匹配版本报错
- `test_pkg.cpp` 和 `test_config.cpp` 扩展

### 3.2 构建进度显示

**现状**：并行编译时各线程通过 `g_log_mutex` 保护日志输出，但没有进度汇总。

**设计**：

```
[ezmk] Building myapp 0.1.0 (profile: debug, -j 8)
[ezmk] [1/12] src/main.cpp           ← 文件级进度
[ezmk] [2/12] src/util.cpp  (cached) ← 缓存命中标记
[ezmk] [3/12] src/config.cpp
...
[ezmk] [12/12] src/app.cpp
[ezmk] Linking myapp.exe...
[ezmk] Build succeeded (3 cached, 9 compiled, 2.4s)
```

**实现**：
- `ThreadPool` 或编译调度层维护 `std::atomic<int> completed{0}` + `int total`
- 每个编译任务完成时 `++completed`，格式化输出 `[{}/{}] {}`
- 缓存命中时追加 `(cached)` 标记
- 仅在 `-j > 1` 或 `--verbose` 时显示逐文件进度；单文件编译保持简洁

### 3.3 .clang-format / .clang-tidy 基础设施

**`.clang-format`**（项目根目录）：
- 基于 LLVM 风格，缩进 4 空格、120 列宽
- 匹配项目现有代码风格

**`CONTRIBUTING.md` 更新**：
- 新增"代码风格"章节，说明 `.clang-format` 的使用方式
- 推荐 VS Code / CLion / vim 的 clang-format 集成

**不做**：
- 不添加 `ezmk utils format` / `ezmk utils lint` 命令（留待后续版本）
- 不强制要求 clang-format 格式化现有代码

### 3.4 ASCII Logo

**设计**（`main.cpp` 无参数时显示）：

```
  ______                __  ___
 /_  __/__ ___ _  __ _/  |/  /___ _____
  / / /_  __ `/ |/_/ / /|_/ / _ `/ __ \
 /_/ /_/ /_/ /_>  </ /_/ /_/  \_,_/_/ /_/
```

- 仅在无子命令时显示（`ezmk` 裸运行等价于 `ezmk --help` 的行为）
- `--version` 不显示 Logo（保持简洁）
- 彩色输出：与 `--color` 模式联动
- 版本号紧随 Logo 下方显示

### 3.5 魔术字符串常量化（0.9.5.1 延后）

0.9.5.1 中此 P2 项因涉及 7+ 文件、跨文件风险较高而延后至本版本。

**现状**：`".ezmk/temp"`、`".ezmk/cache"`、`".ezmk/pkg"`、`".ezmk/repo"` 等路径字符串散落在 `main.cpp`、`build.cpp`、`cache.cpp`、`pkg.cpp`、`repo.cpp`、`util.cpp`、`config.cpp` 中。

**方案**：在 `include/ezmk/util.hpp` 中定义内联常量：

```cpp
namespace ezmk::path {
    inline constexpr std::string_view EZMK_DIR   = ".ezmk";
    inline constexpr std::string_view TEMP_DIR    = ".ezmk/temp";
    inline constexpr std::string_view CACHE_DIR   = ".ezmk/cache";
    inline constexpr std::string_view PKG_DIR     = ".ezmk/pkg";
    inline constexpr std::string_view REPO_DIR    = ".ezmk/repo";
    inline constexpr std::string_view CACHE_OBJ   = ".ezmk/cache/obj";
    inline constexpr std::string_view RECORD_FILE = ".ezmk/cache/record.json";
}
```

**迁移策略**：逐文件替换，每次提交只改一个文件。

## 5 兼容性矩阵

| 变更 | 影响 | 处理 |
|---|---|---|
| `DependsEntry` 替代 `string` | `ezmk.toml` 中纯字符串格式继续工作 | 解析器兼容新旧格式 |
| 版本约束不满足 | 之前 `lib = ["foo"]` 任何版本都接受 | 仅当用户显式添加约束后才启用过滤 |
| 构建输出格式变化 | 依赖输出格式解析的脚本 | 概率低；`--verbose` 保持不变 |
| 新增 Logo | `ezmk` 裸运行输出更长 | 纯增量，美化 |
| `.clang-format` | 无 | 新增，不影响功能 |

## 6 跨版本关注点

- **依赖**: 依赖 0.9.5（集成测试就绪后，可对本版本进行端到端验证）
- **向后兼容**: 依赖版本锁定是 1.0.0 之前最后的破坏性风险点；需确保纯字符串格式（无约束）与 0.9.5 行为完全一致
- **SemVer 合规**: 本版本新增功能，符合 0.9.x → 1.0.0 的语义化版本演进路径
- **1.0.0 准备**: 本版本完成后，所有 temp.md 识别的关键缺口均已填补，项目达到 1.0.0 发布标准
