# EazyMake 0.2.2 — 精细化编译控制

---

## 1 背景

当前 `ezmk.toml` 的编译控制粒度较粗：`depends.lib` 中的依赖若缺失会导致构建直接失败；所有源文件必须位于单一的 `src/` 目录下；预处理器宏只能通过 `compile.flags` 的 `-D` 形式传入，缺乏语义化的配置入口；构建系统不自动注入任何项目元信息宏，用户无法在代码中获知项目名称、版本等信息。

本版本针对这四个痛点提供精细化控制。

---

## 2 可选依赖（`want.lib`）

### 2.1 现状

`[depends]` 中的 `lib` 列表为硬性依赖：任一包未安装 → 构建失败。实际场景中常有"如果存在就使用，否则退化到纯标准库实现"的需求（如可选的压缩库、加密库）。

### 2.2 目标行为

新增 `[depends]` 节中的 `want` 字段：

```toml
[depends]
lib = ["fmt"]       # 硬性依赖：缺失 → fatal error
want = ["sqlite3"]  # 可选依赖：缺失 → 定义标志宏 + warn，继续构建
```

**处理逻辑**（在 `pkg.cpp` 的依赖解析阶段）：

1. 遍历 `want` 列表中的每个包名
2. 在已安装包中查找（项目 → 用户 → 全局作用域）
3. 若找到 → 行为与 `lib` 完全一致（加入依赖图，传递 include/lib）
4. 若未找到 → **不报错**，而是：
   - 输出 `warn` 提示可选依赖未安装
   - 自动定义预处理器宏 `EZMK_LIB_MISS_<NAME>`（大写、去特殊字符）
   - 继续后续编译流程

**自动宏命名规则**：

| want 包名 | 生成的宏 | 说明 |
|-----------|---------|------|
| `sqlite3` | `EZMK_LIB_MISS_SQLITE3` | 基础命名 |
| `fmt` | `EZMK_LIB_MISS_FMT` | 字母+数字保留 |
| `boost-filesystem` | `EZMK_LIB_MISS_BOOST_FILESYSTEM` | `-` → `_` |
| `nlohmann_json` | `EZMK_LIB_MISS_NLOHMANN_JSON` | 保持大写 |

**用户代码中的使用方式**：

```cpp
#ifndef EZMK_LIB_MISS_SQLITE3
  #include <sqlite3.h>
  // ... use SQLite ...
#else
  // fallback: use flat-file storage
#endif
```

### 2.3 实施步骤

- [ ] 在 `include/ezmk/config.hpp` 的 `DependsSection` 中新增 `std::vector<std::string> want` 字段
- [ ] 在 `src/config.cpp` 的 `parse_config()` 中解析 `[depends]` 的 `want` 数组
- [ ] 在 `src/pkg.cpp` 的依赖解析函数中新增可选依赖处理逻辑：
  - 遍历 `want` 列表，对每个未安装的包生成宏定义
  - 将生成的宏名追加到 `compile.flags` 中（`-DEZMK_LIB_MISS_XXX`）
- [ ] 实现 `want_to_macro_name()` 辅助函数：包名 → 合法宏名
  - 大写转换、`-`/`.`/空格 → `_`、去除非字母数字下划线字符
- [ ] `ezmk.toml` 可选依赖示例文档

## 3 语义化宏定义（`compile.macros`）

### 3.1 现状

预处理器宏只能通过 `compile.flags` 中的 `-D` 标志定义：

```toml
[compile]
flags = ["-DDEBUG", "-DAPPLICATION_NAME=\"my_app\"", "-DUNICODE"]
```

问题：
- 与编译警告标志（`-Wall`）混在一起，语义不清
- 宏值含空格/引号时转义复杂，易出错
- 后续 MSVC 翻译时（0.2.1），`-D` 需要翻译为 `/D`，混在 flags 中处理困难

### 3.2 目标行为

新增 `[compile.macros]` 独立节（TOML inline table 或独立 `[compile.macros]` 节）：

```toml
[compile]
flags = ["-Wall", "-Wextra", "-O2"]
include_dirs = ["include"]

[compile.macros]
DEBUG = ""
VERSION = "0.2.0"
APPLICATION_NAME = "my_app"
USE_UNICODE = ""       # 空值 → 只定义宏名，无等号值
```

等价于 GCC 命令行的：
```
-DDEBUG -DVERSION="0.2.0" -DAPPLICATION_NAME="my_app" -DUSE_UNICODE
```

**语义规则**：

| TOML 写法 | 生成的 GCC 标志 | 说明 |
|-----------|----------------|------|
| `DEBUG = ""` | `-DDEBUG` | 空值 → 仅定义符号 |
| `VERSION = "0.2.0"` | `-DVERSION="0.2.0"` | 字符串值 → key=value |
| `MAX_SIZE = 4096` | `-DMAX_SIZE=4096` | 整数值 → 不加引号 |
| `ENABLED = true` | `-DENABLED=1` | 布尔 true → 1 |
| `ENABLED = false` | （不生成） | 布尔 false → 跳过 |

**与 `flags` 的关系**：
- `macros` 中定义的宏**追加**到 `flags` 的 `-D` 宏之后
- 编译命令中宏的顺序：`flags` 中的 `-D` → `macros` 中的条目
- 如果 `flags` 和 `macros` 中定义了同名宏，`macros` 中的值覆盖前者（后者优先，符合编译器行为）

**跨平台兼容性**：
- GCC/Clang：生成 `-D` 格式
- MSVC（0.2.1+）：自动翻译为 `/D` 格式
- 宏的值自动做 shell 转义（`escape_shell_arg`）

### 3.3 实施步骤

- [ ] 在 `include/ezmk/config.hpp` 的 `CompileSection` 中新增：
  ```cpp
  std::map<std::string, std::string> macros; // key → value (empty string = define-only)
  ```
- [ ] 在 `src/config.cpp` 的 `parse_config()` 中解析 `[compile.macros]` 节：
  - 遍历 `macros` 子表的所有键值对
  - 支持三种值类型：字符串、整数、布尔
  - 布尔 false 跳过，其他类型转为字符串存储
- [ ] 实现 `macros_to_flags()` 函数：
  - 遍历 `macros` map → 生成 `-D<key>` 或 `-D<key>=<value>` 数组
  - 空值 → `-D<key>`；非空值 → `-D<key>=<escaped_value>`
- [ ] 在构建流程中（`build.cpp` / `cache.cpp`）将 `macros` 生成的标志合并到编译命令
- [ ] 更新 `write_default_config()` — 不写入 `macros`（保持默认模板简洁）
- [ ] 校验：`macros` 的 key 必须是合法的 C 标识符（字母/数字/下划线，不以数字开头）

## 4 标准预定义宏（`compile.ezmk_macros`）

### 4.1 现状

EazyMake 目前不自动定义任何构建系统级预处理器宏。用户无法在源代码中判断当前是否处于 EazyMake 构建环境，也无法在编译时获取项目元信息（名称、版本、类型等）。类似地，CMake 会定义 `CMAKE_*` 宏族，Rust 的 `build.rs` 会注入 `CARGO_*` 环境变量，EazyMake 缺乏对应的机制。

### 4.2 目标行为

新增 `[compile]` 中的 `ezmk_macros` 字段（布尔值，默认 `true`），控制是否自动注入 `EZMK_*` 标准预处理器宏：

```toml
[compile]
flags = ["-Wall", "-Wextra", "-O2"]
ezmk_macros = true   # 默认值：自动定义标准 EZMK_ 宏
```

设置为 `false` 可完全禁用：

```toml
[compile]
ezmk_macros = false  # 不注入任何 EZMK_ 宏
```

**自动定义的宏**（`ezmk_macros = true` 时）：

| 宏名 | 类型 | 示例值 | 说明 |
|------|------|--------|------|
| `EZMK` | 整数 | `1` | 始终定义为 `1`，标识构建系统为 EazyMake |
| `EZMK_VERSION` | 字符串 | `"0.2.2"` | EazyMake 自身的版本号（来自 `version.hpp`） |
| `EZMK_PROJECT_NAME` | 字符串 | `"myapp"` | 项目名称（`[project].name`） |
| `EZMK_PROJECT_VERSION` | 字符串 | `"1.0.0"` | 项目版本（`[project].version`） |
| `EZMK_PROJECT_TYPE` | 字符串 | `"executable"` | 项目类型（`[project].type`） |
| `EZMK_LANG` | 字符串 | `"C++17"` | 项目语言标准（`[project].language`） |

**等价 GCC 命令行**：

```
-DEZMK=1 -DEZMK_VERSION="0.2.2" -DEZMK_PROJECT_NAME="myapp" -DEZMK_PROJECT_VERSION="1.0.0" -DEZMK_PROJECT_TYPE="executable" -DEZMK_LANG="C++17"
```

**用户代码中的使用方式**：

```cpp
#include <iostream>

int main() {
#ifdef EZMK
    std::cout << "Built with EazyMake " << EZMK_VERSION << std::endl;
    std::cout << "Project: " << EZMK_PROJECT_NAME
              << " v" << EZMK_PROJECT_VERSION << std::endl;
#endif

#ifdef EZMK_PROJECT_TYPE
    // 根据项目类型选择不同代码路径
    #if EZMK_PROJECT_TYPE == "shared"
        // shared library 特定逻辑
    #endif
#endif
    return 0;
}
```

**与 `compile.macros` 的关系**：

- 标准宏在用户自定义的 `compile.macros` **之前**注入到编译命令
- 如果用户在 `compile.macros` 中定义了同名宏（如 `EZMK = ""`），**用户定义的值覆盖标准宏**（符合 GCC/Clang 的"后定义的 `-D` 覆盖前一个"行为）
- 标准宏的注入不修改 `compile.macros` 的内容（两套机制独立）

### 4.3 实施步骤

- [ ] 在 `include/ezmk/config.hpp` 的 `CompileSection` 中新增：
  ```cpp
  bool ezmk_macros = true;  // 是否注入标准 EZMK_ 预定义宏（默认 true）
  ```
- [ ] 在 `src/config.cpp` 的 `parse_config()` 中解析 `compile.ezmk_macros`：
  - 存在且为布尔 false → 设为 false
  - 不存在或为 true → 默认 true
  - 其他类型 → 报错（仅接受布尔值）
- [ ] 实现 `generate_ezmk_macros()` 函数（`src/build.cpp`）：
  - 从 `EzConfig` 生成 `std::vector<std::string>`（每个元素为 `-D<KEY>=<VALUE>` 格式）
  - `EZMK=1`：始终为 `1`（标识宏）

  - 其余宏（`EZMK_PROJECT_NAME` 等）从 `[project]` 节对应字段读取
  - 值类型规则与 `compile.macros` 一致：空值 → `-D<KEY>`，非空 → `-D<KEY>=<VALUE>`
- [ ] 在构建流程中（`build.cpp` 的编译命令构建逻辑）集成：

  - 若为 false → 跳过
- [ ] 更新 `write_default_config()` — 不写入 `ezmk_macros`（默认值 `true`，保持模板简洁）

## 5 非标项目结构支持（`compile.src_dirs`）

### 5.1 现状

所有源文件必须位于 `src/` 目录下。部分项目有更复杂的目录结构：

```
myapp/
  src/
    core/
    ui/
    utils/
  lib/
    internal/
  thirdparty/
```

当前只能将所有源文件放在 `src/`，或通过包管理拆分为多个子包。

### 5.2 目标行为

新增 `compile.src_dirs` 字段，允许指定多个源目录：

```toml
[compile]
flags = ["-Wall", "-Wextra", "-O2"]
include_dirs = ["include", "lib/internal/include"]
src_dirs = ["src", "lib/internal", "thirdparty/json"]
```

**语义规则**：

| 行为 | 规则 |
|------|------|
| 默认值 | `src_dirs` 未指定 → 默认 `["src"]`（向后兼容） |
| 多目录扫描 | 每个目录独立扫描 `.c`/`.cpp`/`.cxx`/`.cc` 文件 |
| 目录不存在 | warn 并跳过（非致命错误） |
| 相对路径 | 相对于项目根目录解析 |
| 空目录 | 不报错（只要至少有一个目录提供了源文件） |
| 无源文件 | 所有目录扫描完毕后仍无源文件 → fatal |
| 文件去重 | 同一文件名出现在多个目录中 → warn，后面的覆盖前面的（以首次出现的为准） |
| `main.cpp` 检查 | `type = "executable"` 时，在**所有** `src_dirs` 中查找 `main.cpp`/`main.c` |

**构建产物路径**：

源文件来自不同目录时，目标文件（`.o`）保持相对目录结构：

```
# 源文件:
#   src/core/main.cpp       → build/src/core/main.o
#   lib/internal/helper.cpp → build/lib/internal/helper.o
```

缓存 key 基于相对于项目根目录的路径，天然支持多目录。

**与现有行为的兼容性**：
- 不配置 `src_dirs` → 行为与当前版本完全一致
- 配置了 `src_dirs` 但只包含 `["src"]` → 行为一致
- 配置了其他目录 → 替代 `src/`（不自动追加 `src/`）

### 5.3 实施步骤

- [ ] 在 `include/ezmk/config.hpp` 的 `CompileSection` 中新增：
  ```cpp
  std::vector<std::string> src_dirs; // source directories (default ["src"])
  ```
- [ ] 在 `src/config.cpp` 的 `parse_config()` 中解析 `src_dirs`：
  - 存在 → 使用用户指定的值
  - 不存在 → 默认 `{"src"}`
  - **重要**：若用户显式指定了 `src_dirs`，不自动追加 `"src"`（尊重用户选择）
- [ ] 修改 `src/build.cpp` 的源文件扫描逻辑（当前位置：`util::list_files(src_dir, ...)`）：
  - 遍历 `cfg.compile.src_dirs`
  - 对每个目录调用 `util::list_files(dir, {".c", ".cc", ".cpp", ".cxx"})`
  - 合并所有结果（去重）
  - `main.cpp` 检查在所有目录中查找
- [ ] 修改 `src/cache.cpp` 的目标文件路径生成（`CompileInput::sources`）：
  - 目标文件保留源目录的相对结构：`src/core/main.cpp` → `obj_dir/src/core/main.o`
  - 这已经是当前行为（`fs::relative(src, proj_root)` 产生相对路径）
- [ ] 更新 `write_default_config()` — 不写入 `src_dirs`（默认行为是 `["src"]`）
- [ ] 新增 util 工具函数 `collect_sources()`：给定 src_dirs 列表，返回去重后的源文件列表

## 6 配置结构变更汇总

### 6.1 `include/ezmk/config.hpp`

```cpp
struct CompileSection {
    std::vector<std::string> flags;
    std::vector<std::string> msvc_flags;     // 0.2.1+
    std::vector<std::string> include_dirs;
    std::vector<std::string> src_dirs;       // 0.2.2+ (default ["src"])
    std::map<std::string, std::string> macros; // 0.2.2+
    bool ezmk_macros = true;                 // 0.2.2+ 注入 EZMK_ 标准宏
};

struct DependsSection {
    std::vector<std::string> libs;  // hard dependencies
    std::vector<std::string> want;  // 0.2.2+ optional dependencies
};
```

### 6.2 `ezmk.toml` 完整示例（0.2.2）

```toml
[project]
name = "myapp"
type = "executable"
version = "1.0.0"
language = "C++17"

[compile]
flags = ["-Wall", "-Wextra", "-O2"]
include_dirs = ["include", "lib/util/include"]
src_dirs = ["src/core", "src/ui", "lib/util"]
ezmk_macros = true

[compile.macros]
DEBUG = ""
APPLICATION_NAME = "MyApp"
VERSION = "1.0.0"
MAX_CONNECTIONS = 64

[link]
flags = []
link_dirs = []
system_target = ["pthread"]

[depends]
lib = ["fmt"]
want = ["sqlite3", "zlib"]
```

## 7 各特性的交互关系

### 7.1 `want.lib` × `compile.macros`

可选依赖生成的 `EZMK_LIB_MISS_*` 宏**追加**到 `compile.macros` 中，而非 `compile.flags`。这意味着：
- 宏与编译标志分离，语义清晰
- 用户可用 `ezmk.toml` 的 `macros` 覆盖（但通常不会这么做）
- `--verbose` 输出中宏的来源清晰可辨

### 7.2 `src_dirs` × `include_dirs`

`src_dirs` 仅影响源文件扫描，不影响头文件搜索路径。如果非标目录中有头文件需要被引用，仍需在 `include_dirs` 中显式声明。

### 7.3 `src_dirs` × 包编译

`pkg.cpp` 中的 `compile_package()` **不使用** `src_dirs` — 包的源文件仍然统一在包的 `src/` 目录下。`src_dirs` 仅影响用户项目（`project build`）。

### 7.4 `ezmk_macros` × `compile.macros`

标准宏在用户自定义宏**之前**注入编译命令。由于 GCC/Clang 的"后定义的 `-D` 覆盖前一个"行为，用户在 `compile.macros` 中定义同名宏可以覆盖标准宏。例如：

```toml
[compile.macros]
EZMK_PROJECT_VERSION = "2.0.0-dev"  # 覆盖自动生成的 "1.0.0"
```

这提供了灵活性：用户可以在不关闭 `ezmk_macros` 的情况下，按需覆盖特定宏值。

### 7.5 `ezmk_macros` × `want.lib`

`EZMK_LIB_MISS_*` 宏（来自可选依赖）与 `EZMK_*` 标准宏命名前缀一致（均为 `EZMK_`），但功能正交：
- 标准宏描述**项目元信息**（名称、版本、类型）
- `EZMK_LIB_MISS_*` 描述**依赖可用性**（可选包是否已安装）

两者互不干扰，用户可在条件编译中组合使用：
```cpp
#ifdef EZMK
  #ifndef EZMK_LIB_MISS_SQLITE3
    // EazyMake 构建 + SQLite3 可用
  #endif
#endif
```

## 8 注意事项

- **向后兼容**：所有新字段均为可选，不配置时行为与 0.2.0/0.2.1 完全一致
- **`src_dirs` 不自动追加 `"src"`**：如果用户显式指定了 `src_dirs = ["app"]`，则只扫描 `app/` 目录。这是一种合理的"显式优于隐式"选择；用户若需保留 `src/`，必须在列表中显式写出
- **宏名称校验**：`macros` 的 key 必须是合法 C 标识符（`[A-Za-z_][A-Za-z0-9_]*`），非法 key 报错
- **`want` 与 `lib` 的去重**：同一个包名同时出现在 `lib` 和 `want` 中时，`lib` 优先（作为硬性依赖处理），并 warn 冗余配置
- **宏值转义**：`macros` 中的值可能包含空格、引号等特殊字符，生成 `-D` 标志时需正确转义。字符串值两边加双引号，内部双引号和反斜杠做转义
- **`src_dirs` 为空**：如果用户写了 `src_dirs = []`（空数组），应报错提示至少需要一个源目录
- **`ezmk_macros` 默认启用**：新建项目默认注入 `EZMK_*` 宏（`ezmk_macros = true`）。这符合"开箱即用"原则，且标准宏不会与用户代码产生命名冲突（`EZMK_` 前缀足够独特）
- **`EZMK` 始终为 `1`**：`EZMK` 宏不带项目信息，仅作为"是否在 EazyMake 中构建"的布尔标志。用户不应依赖其具体数值
- **标准宏的值来源**：`EZMK_VERSION` 来自编译期常量（`version.hpp`），其余宏值在构建时从 `ezmk.toml` 的 `[project]` 节动态读取。若 `ezmk.toml` 中缺少对应字段（如 `language` 未设置），使用默认值
- **性能**：多目录扫描使用 `util::list_files()`（非递归），每个目录独立调用，开销与目录数量线性增长。对于 99% 的项目（1-5 个源目录），性能影响可忽略
