# EazyMake 0.9.2 — 文档多语言

---

## 1 背景

EazyMake 从 0.1.7 起就建立了中英双语的 i18n 体系：所有用户可见字符串（CLI 帮助、错误信息、日志）均通过 `I18nKey` 枚举 + `en.json`/`zh.json` 双语覆盖。0.9.0 的 README 也做了双语互链（`README.md` ↔ `README_ZH.md`）。

但**项目文档和教程**目前仍以中文为主：

| 路径              | 当前语言 | 问题                               |
| ----------------- | -------- | ---------------------------------- |
| `docs/*.md`       | 中文     | 英文用户无法阅读设计文档           |
| `tutorial/*.md`   | 中文     | 英文用户没有入门路径               |
| `CLAUDE.md`       | 英文     | 唯一例外，面向 AI 工具             |
| `README.md`       | 英文     | 0.9.0 已做双语 README              |
| `README_ZH.md`    | 中文     | 同上                               |
| `plans/*.md`      | 中文     | 版本计划，内部使用可不翻译         |

对于 1.0 正式发布，**英文文档是硬需求**：GitHub 的默认受众是英语用户，`curl | bash` 进来的用户需要能读懂设计意图、配置格式、安全模型、Lua API。

本版本将 `docs/` 和 `tutorial/` 按语言拆分为 `en/` 和 `zh/` 子目录，补齐英文翻译，并建立"改文档先改中文、再英译"的维护流程，保证双语同步。

---

## 2 目标

1. 重组目录结构：`docs/` → `docs/en/` + `docs/zh/`，`tutorial/` → `tutorial/en/` + `tutorial/zh/`。
2. 补齐所有 `docs/` 文件的英文翻译。
3. 补齐所有 `tutorial/` 文件的英文翻译。
4. 更新所有交叉引用（README、CLAUDE.md、源码注释中的文档路径）。
5. 建立文档维护规则（哪个是权威语言、翻译流程、CI 检查）。

---

## 3 目录结构设计

### 3.1 重组前 → 重组后

**当前**：

```
docs/
```

```
  zh/

    @cache.md
    @safety.md
tutorial/

  zh/
    (tutorial files)
```

### 3.2 路径约定

- 跨语言的交叉引用使用**相对路径 `../en/` / `../zh/`**，保证在 GitHub 上直接可点击。
- README 中的文档链接默认指向 `docs/en/`（英文 README）/ `docs/zh/`（中文 README），并在页面顶部提供"Read this in English / 阅读中文版"互链。
- CLAUDE.md 中的路径引用指向 `docs/en/`（AI 工具以英文为主）。

### 3.3 不翻译的内容

| 内容                  | 原因                                       |
| --------------------- | ------------------------------------------ |
| `plans/*.md`          | 内部版本计划，开发团队中文交流             |
| 源码注释              | 已有英文注释为主；零星中文注释保持即可     |
| `locale/en.json`      | 已存在，是 i18n 的一部分，不在本版本范围   |
| `locale/zh.json`      | 同上                                       |

## 4 翻译策略

### 4.1 权威语言

**中文为权威语言（source of truth）**。原因：
- 现有文档均为中文撰写，质量经过多轮审阅。
- 核心维护者以中文为主要工作语言。
- 英文翻译从中文出发，而非另写一套独立内容。

### 4.2 翻译流程

1. 中文文档（`zh/`）更新 → 合并到 main。
2. 同步更新英文翻译（`en/`）→ 单独 PR 或同一 PR 的第二 commit。
3. CI（§7）检查 `en/` 与 `zh/` 的文件一一对应（不存在"只有一边有"的文件）。

### 4.3 翻译质量要求

- **技术准确性优先于文学性**：API 名称、字段名、命令名、错误信息原文保留（如 `ezmk.toml`、`pkg install`、`I18nKey`）。
- **术语一致性**：建立术语表（§6.2），中→英映射统一。
- **代码块不翻译**：TOML 示例、shell 命令、Lua 代码保持原样。
- Markdown frontmatter（若有）仅翻译 `title`/`description` 字段。

## 5 各文档翻译要点

### 5.1 `docs/` 文件

| 文件               | 内容概要                         | 翻译注意事项                                                                   |
| ------------------ | -------------------------------- | ----------------------------------------------------------------------------- |
| `cli.md`           | CLI 命令参考表 + 详细说明        | 命令名/flag 保留原样；示例输出需更新为英文 locale 下的实际输出                |
| `config_file.md`   | `ezmk.toml` 完整格式规范         | 字段名保留；类型/默认值/说明翻译；TOML 示例共用                               |
| `pkg.md`           | 包管理设计文档                   | 概念翻译 + 命令示例保留；安装路径的说明翻译                                   |
| `repo.md`          | 仓库系统设计文档                 | `index.toml` 结构说明翻译；字段名保留                                         |
| `utils.md`         | Lua API 参考（22 个函数）        | 函数签名保留；参数/返回值说明翻译；权限模型说明翻译                           |
| `@cache.md`        | 缓存机制（哈希算法、原子写入）   | 算法描述翻译；文件路径保留                                                    |
| `@safety.md`       | 安全模型（全局安装确认、sha256） | 安全策略翻译；命令保留                                                        |

### 5.2 `tutorial/` 文件

需先确认 0.9.0 的教程文件清单。预期结构：

```
tutorial/
  en/
    index.md          # 教程目录
    01-quickstart.md  # 快速开始
    02-project.md     # 项目管理
    03-package.md     # 包管理
    ...
  zh/
    index.md
    01-quickstart.md
    02-project.md
    03-package.md
    ...
```

翻译要点：
- 每步的 shell 命令保留（用户实际敲的命令不变）。
- 命令的预期输出需切换为英文 locale（`EZMK_LANG=en`）。
- 概念解释翻译为英文，保持教程的"手把手"语调。

## 6 术语表

### 6.1 中→英术语映射

为保持翻译一致性，核心术语统一如下：

| 中文         | 英文                       | 说明                                     |
| ------------ | -------------------------- | ---------------------------------------- |
| 包           | package                    | 与 `pkg` 命令族对应                      |
| 仓库         | repository (repo)          | 与 `repo` 命令族对应                     |
| 作用域       | scope                      | `-g`/`-u`/`-p`                           |
| 构建         | build                      |                                          |
| 编译         | compile / compilation      |                                          |
| 链接         | link / linking             |                                          |
| 缓存         | cache                      |                                          |
| 钩子         | hook                       | 构建生命周期钩子                         |
| 补全         | completion                  | shell 补全                               |
| 工具链       | toolchain                  | 编译器/链接器抽象                        |
| 归档         | archive                    | `.zip`/`.tar.gz` 包文件                  |
| 工作区       | workspace                  |                                          |
| 依赖         | dependency                 |                                          |
| 可选依赖     | optional dependency        | `want.lib`                               |
| 宏           | macro                      | `compile.macros`                         |
| 权限         | permission                 | utils 白名单                             |
| 校验         | validation / verification  | sha256 / index.toml                      |
| 探测         | detection                  | 编译器自动检测                           |
| 沙箱         | sandbox                    | Lua 安全模型                             |

### 6.2 术语表存放

术语表放在 `docs/en/glossary.md`（英文版）和 `docs/zh/glossary.md`（中文版），供翻译者和贡献者参考。CI 不强制检查术语使用，但 PR review 以此为参考。

## 7 CI 检查

### 7.1 文件对应性检查

新增 CI 步骤（可并入 0.9.1 或后续 CI 计划，或临时用脚本）：

```bash
# 检查 docs/ 和 tutorial/ 下 en/ 与 zh/ 的文件一一对应
for dir in docs tutorial; do
  diff <(cd $dir/en  && find . -name '*.md' | sort) \
       <(cd $dir/zh  && find . -name '*.md' | sort)
done
```

存在差异 → CI 失败，提示缺失的文件。

### 7.2 链接有效性检查（可选）

用 `markdown-link-check` 或等价工具检查 `docs/en/` 与 `docs/zh/` 中内部链接均指向存在的文件。此项优先级低于 §7.1，可作为后续增强。

| `scripts/install.sh`   | 如有文档 URL 引用，更新                                        |

## 9 维护规则（写入 `CONTRIBUTING.md`）

在项目根（或 `docs/en/CONTRIBUTING.md`）中明确：

1. 新增/修改文档 → 先更新 `zh/` 版本（权威），再同步更新 `en/` 翻译。
2. 翻译 PR 应包含双语变更（或紧跟在中文 PR 之后提交）。
3. 术语参考 `docs/en/glossary.md`。
4. 代码示例中的命令输出需指定 `EZMK_LANG=en` 或 `EZMK_LANG=zh` 以匹配文档语言。

## 12 开放问题

1. **`plans/` 是否也需要翻译**：当前决定不译（§3.3），因为版本计划面向开发团队。但如果未来有外部贡献者参与版本设计，可能需要英文版。暂推迟。
2. **英文文档是否需要独立审阅**：当前维护者非英语母语，英文翻译可能有 Chinglish。是否需要找 native speaker 做一轮审阅？可在 1.0 前做一次 pass。
3. **文档格式是否统一为 MyST/CommonMark 子集**：当前各文档用的 Markdown 风格不完全一致（有的用 `>` blockquote、有的用表格、有的用 HTML）。是否借翻译机会统一格式？建议不在此版本做——翻译时保持原格式，格式统一另开版本。
4. **是否提供 PDF/HTML 构建**：将 Markdown 文档渲染为静态站点（GitBook / mdBook / Docusaurus）或 PDF 分发。当前 GitHub 渲染已够用，推迟到 post-1.0。
5. **翻译的"时滞"容忍度**：`zh/` 更新后，`en/` 允许滞后多久？建议宽松策略：中文 PR 合并时不强制英文同步，但每月检查一次 `en/` 与 `zh/` 的差异，集中补译。
