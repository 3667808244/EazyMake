# EazyMake 0.9.8 — CLI 改进、默认仓库扩充与文档检查

---

## 1 背景

0.9.7 大幅扩展了默认仓库生态（22 个新包，覆盖压缩/CLI/窗口/多媒体/YAML/ImGui 全家桶），0.9.6 完成了依赖版本锁定、构建进度显示等核心功能。本版本作为 1.0.0 之前最后一个功能版本，聚焦三个维度：

1. **CLI 输出一致性**：`ezmk pkg info`、`ezmk repo info`、`ezmk repo list` 等命令的输出缺少统一的 `[ezmk]` 前缀，与 `util::info()`/`util::warn()`/`util::error()` 系列函数的行为不统一
2. **CLI 用户体验**：使用简写命令（`ki`/`ri`/`rl` 等）时无提示，用户无法确认实际执行的完整命令
3. **默认仓库扩充**：继续扩展默认仓库包数量，降低新用户从零配置的门槛

---

## 2 目标

1. **补全缺失的 `[ezmk]` 前缀** — `ezmk pkg info`、`ezmk repo info`、`ezmk repo list` 输出添加统一的 `[ezmk]` 前缀
2. **`--verbose` 时输出简写命令全名** — 使用简写命令时，在 verbose 模式下显示展开后的完整命令
3. **默认仓库扩充** — 添加高频使用的 Header-Only 库（Boost 子集 + stb 系列）

---

## 3 详细设计

### 3.1 补全缺失的 `[ezmk]` 前缀

#### 3.1.1 现状分析

当前 EazyMake 有两套输出体系：

| 输出方式 | 前缀 | 目标流 | 用途 |
|---|---|---|---|
| `util::info()` / `util::warn()` / `util::error()` | `[ezmk]` / `[ezmk warn]` / `[ezmk error]` | `std::cerr` | 进度/状态/错误消息 |
| `std::cout` | 无 | `std::cout` | 结构化/信息性输出 |

问题在于 `pkg info`、`repo info`、`repo list` 三个命令全部使用 `std::cout` 直接输出，导致其输出缺少 `[ezmk]` 前缀，与其他命令（`pkg install`、`pkg list`、`repo add` 等）风格不统一。

**受影响位置**：

| 命令 | 文件 | 行号 | 当前行为 |
|---|---|---|---|
| `ezmk pkg info` | `src/pkg.cpp` | 652–762 | 全部字段用 `std::cout` 输出，无前缀 |
| `ezmk repo info` | `src/repo.cpp` | 508–592 | 全部详情行用 `std::cout` 输出，无前缀 |
| `ezmk repo list` | `src/repo.cpp` | 470–501 | 标题和条目用 `std::cout` 输出，无前缀 |

> **注**：`ezmk pkg list` 已使用 `util::info()` 输出（带前缀），但 `ezmk repo list` 的标题行使用了 `std::cout`，形成不一致。此外 `ezmk repo list` 的空列表提示走 `util::info()`（带前缀），与有数据时的 `std::cout` 输出混用。

#### 3.1.2 设计方案

**方案选择**：将 `pkg info`、`repo info`、`repo list` 的每行输出改为通过 `util::info()` 输出（或新增一个带前缀但无 `[ezmk]` 颜色标记的格式化辅助函数）。

**推荐做法**：为结构化信息输出新增一个轻量辅助函数 `util::info_line()`，行为与 `util::info()` 一致（带 `[ezmk]` 前缀、输出到 `stderr`），但不加颜色，适合多行结构化数据。

```cpp
// util.hpp 新增
void info_line(std::string_view msg);  // [ezmk] prefix, no color, stderr
```

**各命令改动**：

**`ezmk pkg info`（`src/pkg.cpp:652-762`）**：
- 每行 `std::cout <<` 替换为 `util::info_line()`
- 示例输出前后对比：

```
// 改前
Name: fmt
Version: 10.2.1
Type: static
Scope: user scope

// 改后
[ezmk] Name: fmt
[ezmk] Version: 10.2.1
[ezmk] Type: static
[ezmk] Scope: user scope
```

**`ezmk repo info`（`src/repo.cpp:508-592`）**：
- 同样将 `std::cout <<` 替换为 `util::info_line()`
- 包版本列表也逐行带前缀

**`ezmk repo list`（`src/repo.cpp:470-501`）**：
- 标题行和每条 repo 条目改为 `util::info_line()`
- 空列表的 `util::info()` 调用保持不变（已有前缀）

**不修改的部分**：
- `cli.cpp` 的 `print_usage()` 帮助输出 — 帮助文本是面向用户的格式化展示，不加前缀是合理的（对标 `git --help`、`cargo --help` 等工具的做法）
- `main.cpp` 的 `--version` 输出 — 版本信息保持简洁

#### 3.1.3 实现步骤

- [ ] 在 `include/ezmk/util.hpp` 中声明 `info_line(std::string_view msg)`
- [ ] 在 `src/util.cpp` 中实现 `info_line()`：`[ezmk] ` + msg，无颜色，输出到 `std::cerr`
- [ ] 改造 `src/pkg.cpp` 的 `pkg::info()`：`std::cout` → `util::info_line()`
- [ ] 改造 `src/repo.cpp` 的 `repo::info()`：`std::cout` → `util::info_line()`
- [ ] 改造 `src/repo.cpp` 的 `repo::list()`：`std::cout` → `util::info_line()`
- [ ] 更新 i18n 键值（若原有键值带 `[ezmk]` 前缀则去除，交由 `info_line()` 统一添加）

### 3.2 `--verbose` 时输出简写命令全名

#### 3.2.1 现状

简写命令（`pn`/`pb`/`ki`/`ri`/`rl` 等，共 18 个）在 `cli::parse()` 的最顶部（`src/cli.cpp:544-574`）被静默展开为完整命令：

```cpp
// 例：ezmk ri official → ezmk repo info official
static const std::map<std::string_view, ...> kAliases = { ... };
if (auto it = kAliases.find(toks[1]); it != kAliases.end()) {
    // 展开 toks[1] → it->second.first + it->second.second
}
```

展开后下游解析器只看到规范命令名，用户不知道简写被展开成了什么。这在日常使用中没问题，但调试时（`--verbose`）缺少透明度。

#### 3.2.2 设计

当 `--verbose` 标志存在时，在简写展开后输出一条信息：

```
[ezmk] shorthand: ri → repo info
```

**实现要点**：
- 展开逻辑在 `cli::parse()` 的最顶部，此时 `--verbose` 尚未被各子命令解析器处理
- 需要提前扫描 `argv` 中是否存在 `-v` / `--verbose`
- 或在展开后延迟输出（将简写信息存入 `ParsedArgs`，由后续流程在确认 verbose 后输出）

**推荐方案**：在 `ParsedArgs` 中新增字段记录简写展开信息，`main.cpp` 在分发命令前检查 `verbose` 并输出。

```cpp
// cli.hpp ParsedArgs 新增
struct ParsedArgs {
    // ...
    std::string shorthand_expansion;  // 非空表示使用了简写，"ri → repo info"
};
```

**输出示例**：

```
$ ezmk ri official -v
[ezmk] shorthand: ri → repo info
[ezmk] Repo info for 'official':
[ezmk]   Name: official
...

$ ezmk ki fmt -v
[ezmk] shorthand: ki → pkg install
[ezmk] Resolving dependencies...
```

#### 3.2.3 实现步骤

- [ ] `cli.hpp` 的 `ParsedArgs` 新增 `shorthand_expansion` 字段
- [ ] `cli.cpp` 简写展开时填充 `shorthand_expansion`（如 `"pb → project build"`）
- [ ] `main.cpp` 命令分发前检查 `verbose` + `!shorthand_expansion.empty()`，输出信息
- [ ] 新增 i18n 键 `shorthand_expansion`（en: `"shorthand: {}"`, zh: `"简写展开: {}"`）

### 3.3 默认仓库扩充

0.9.7 已添加 22 个预编译包（zlib/cli11/glfw/sdl2/yaml-cpp/imgui+16 后端）。本版本聚焦 **Header-Only 库**，这类库无需预编译二进制，仅需分发头文件，打包和安装成本极低。

#### 3.3.0 前置条件：Header-Only 包支持

**现状**：当前 `src/pkg.cpp` 的 `compile_package()` 函数始终编译源文件为 `.a` 静态库，没有 "header-only" 的概念。0.9.7 计划已识别此缺口（`cli11` 包建议 `pkg.toml` 声明 `header_only = true`），但未实现。

**需要实现**：

1. **`pkg.toml` 新增字段**：`header_only = true`（布尔值，默认 `false`）
2. **安装流程适配**（`src/pkg.cpp`）：
   - 解析 `pkg.toml` 后检查 `header_only` 字段
   - `header_only = true` 时：跳过编译步骤，仅复制 `include/` 头文件到安装目录
   - 不生成 `.a` / `.lib` 产物
3. **`ezmk pkg info` 适配**：header-only 包的信息输出中标注 `Type: header-only`
4. **向后兼容**：未声明或 `header_only = false` 的包行为不变

**验证**：先在本地创建一个 header-only 测试包，验证 `ezmk pkg install --type file <test_hdr_pkg.zip>` 能正确安装。

#### 3.3.1 Boost Header-Only 子库

Boost 体量巨大（100+ 子库），全量打包不现实。本版本选择 **使用率最高、完全 Header-Only、无内部 Boost 依赖** 的子库：

| 包名 | Boost 子库 | 说明 | 许可证 |
|---|---|---|---|
| `boost-assert` | Boost.Assert | 轻量断言宏（`BOOST_ASSERT`） | BSL-1.0 |
| `boost-config` | Boost.Config | 编译器/平台特性检测 | BSL-1.0 |
| `boost-core` | Boost.Core | 核心工具（`addressof`、`ref`、`noncopyable` 等） | BSL-1.0 |
| `boost-static-assert` | Boost.StaticAssert | 编译期断言 | BSL-1.0 |
| `boost-throw-exception` | Boost.ThrowException | 异常抛出工具 | BSL-1.0 |
| `boost-lexical-cast` | Boost.LexicalCast | 字符串↔数值转换 | BSL-1.0 |
| `boost-algorithm` | Boost.Algorithm | 字符串算法（`trim`/`split`/`join` 等） | BSL-1.0 |
| `boost-optional` | Boost.Optional | `optional<T>` 类型 | BSL-1.0 |
| `boost-variant2` | Boost.Variant2 | 类型安全联合体 | BSL-1.0 |
| `boost-mp11` | Boost.Mp11 | C++11 元编程工具 | BSL-1.0 |

> **选型原则**：
> - 纯 header-only，无需编译（`boost-*` 包的 `pkg.toml` 声明 `header_only = true`）
> - 无内部 Boost 依赖或依赖已包含在本列表中的其他包
> - 社区使用广泛（GitHub star 数 / 项目引用数 / StackOverflow 讨论量）
> - 不包含需要编译的子库（如 `boost-asio`、`boost-filesystem`、`boost-regex` 等留待后续版本）
>
> **版本策略**：跟随最新 Boost 发布版（当前 1.87.0），各子库使用统一的 Boost 版本号，简化维护。

#### 3.3.2 stb 单文件库

[stb](https://github.com/nothings/stb) 是 Sean Barrett 维护的一系列单文件 C 库，采用 Public Domain / MIT 许可证，在游戏开发和图形学社区使用极其广泛：

| 包名 | 对应文件 | 说明 |
|---|---|---|
| `stb-image` | `stb_image.h` | 图像加载/写入（PNG/JPEG/BMP/TGA/HDR 等） |
| `stb-image-write` | `stb_image_write.h` | 图像写入 |
| `stb-image-resize` | `stb_image_resize.h` | 图像缩放 |
| `stb-truetype` | `stb_truetype.h` | TrueType 字体光栅化 |
| `stb-rect-pack` | `stb_rect_pack.h` | 矩形打包（纹理图集） |
| `stb-vorbis` | `stb_vorbis.c` | Ogg Vorbis 解码（⚠️ 非 header-only） |
| `stb-perlin` | `stb_perlin.h` | Perlin 噪声 |
| `stb-sprintf` | `stb_sprintf.h` | `sprintf` 替代实现 |
| `stb-ds` | `stb_ds.h` | 动态数组/哈希表（类型安全泛型宏） |
| `stb-textedit` | `stb_textedit.h` | 文本编辑控件 |

> **注意**：
> - 除 `stb-vorbis` 外均为 header-only 单文件
> - `stb-vorbis` 需要编译 `.c` 文件，需要提供预编译静态库（平台：Windows/MSYS2、Linux、macOS）
> - stb 库无稳定版本号（每个文件独立维护），采用 GitHub commit hash 作为版本标识
> - 每个 stb 包的 `pkg.toml` 中 `header_only = true`，`ezmk pkg install` 跳过编译步骤

#### 3.3.3 包验证标准

每个包上线前通过：

1. **头文件完整性**：在示例项目中 `#include` 主要头文件，编译通过
2. **版本信息准确**：`index.toml` 中版本号与实际一致
3. **包依赖正确**：Boost 包的依赖关系在 `index.toml` 中准确声明

> Header-only 包**无需**平台预编译验证（无二进制产物），验证成本显著低于 0.9.7 的预编译包。

## 6 跨版本关注点

- **依赖**: 无硬依赖（不依赖 0.9.7 的代码改动；默认仓库包扩展可与 0.9.7 的包共存）
- **向后兼容**: `pkg info`/`repo info`/`repo list` 的输出格式变更为唯一潜在的破坏性变更；`[ezmk]` 前缀新增使输出更易解析，概率极低的脚本依赖可通过 `--color=never` + 固定格式保证稳定性
- **1.0.0 准备**: 本版本是 1.0.0 之前的最后一个功能版本；完成后项目进入纯文档/质量打磨阶段
- **延后项**: 更多 Boost 子库（需编译的 `boost-filesystem`/`boost-asio` 等）、其他 header-only 库（`tinygltf`/`cereal`/`cxxopts` 等）留待 1.0.0 之后
