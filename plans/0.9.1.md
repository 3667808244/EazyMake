> 仓库已经克隆到`E:\claude_workspace\ezmk-repo`
> Gitee镜像(已配置自动同步Github仓库): `https://gitee.com/egglzh/ezmk-repo.git`

---

## 1 背景

EazyMake 的仓库系统（`docs/repo.md`、`src/repo.cpp`）已完整实现：`repo add/remove/update/list/info`、跨仓库取最高版本的 `search_package()`、本地仓库 `index.toml` 校验、`pkg install` 按名安装、sha256 校验。但**没有任何默认仓库**——用户必须先知道某个第三方仓库 URL 并 `ezmk repo add`，才能按名安装包。这让"开箱即用"的包分发无从谈起。

本版本创建并托管一个**官方默认仓库**，并决定 ezmk 如何让用户便捷地接入它。

> 复用的既有事实（`docs/repo.md` + `src/repo.cpp`）：
> - 仓库 = 一个 git 仓库，根含 `index.toml`（`[repo]` + 可重复的 `[[packages]]`）与 `packages/` 目录。
> - `[[packages]]` 字段：`name`、`version`、`file`（相对路径）、`sha256`（可选，公共仓库强烈建议）。
> - 每个"包"本身是一个 ezmk 项目（有自己的 `ezmk.toml`，`type` 为 `static`/`shared`/`utils`），归档为 `.zip` 或 `.tar.gz`。
> - `ezmk pkg install <name>` 触发 `repo::search_package()`：跨作用域、跨仓库收集同名包所有版本，取最高版本，从对应仓库 `packages/` 取归档；若含 `sha256` 则安装时强制校验。
> - `type = "local"` 仓库注册时会校验 `file` 存在性与 `sha256` 格式（`repo_validate_*`）。

## 2 目标

1. 在 GitHub 上创建并托管官方默认仓库（git 仓库，符合 `docs/repo.md` 结构）。
2. 建立**打包 → 计算 sha256 → 维护 `index.toml`**的可复现流程与 CI 校验。
3. 决定 ezmk 侧如何接入默认仓库（预注册策略），让用户"装完 ezmk 就能按名装包"。
4. 提供一小组**真实可用的初始包**与**贡献流程**，让生态可增长。

## 3 仓库选型与命名

| 决策项       | 方案                                                                    |
| ------------ | ----------------------------------------------------------------------- |
| 托管         | GitHub，与主项目同 owner：`https://github.com/3667808244/ezmk-repo.git` |
| 注册名       | `official`（`ezmk repo list` 中显示）                                   |
| 分支         | `main`                                                                  |
| 镜像（可选） | 后续可加 Gitee 镜像，缓解国内访问；`repo add` 已支持任意 git URL        |

**目录结构**（与 `docs/repo.md` 一致）：

```
ezmk-repo/
  index.toml                 # 仓库元数据 + 包索引（权威）
  packages/                  # 归档文件
    hello-lib-0.1.0.tar.gz
    ...
  sources/                   # 各包的源工程（含 ezmk.toml），用于可复现打包
    hello-lib/
      ezmk.toml
      src/ include/ ...
  scripts/
    pack.sh                  # 打包 + 计算 sha256 + 重写 index.toml
    validate.sh              # 本地/CI 校验
  .github/workflows/ci.yml   # PR 校验
  README.md                  # 仓库说明 + 贡献指南
  CONTRIBUTING.md
```

> `sources/` 与 `packages/` 分离：`sources/` 是可审计的源工程，`packages/` 是由 `pack.sh` 从 `sources/` 生成的归档产物。CI 校验二者一致（见 §6）。

## 5 打包与发布流程

ezmk 目前**没有** `pkg pack` 命令（打包由仓库侧脚本完成，避免为 0.9.1 引入新 CLI 功能）。

`scripts/pack.sh`（在 ezmk-repo 内）职责：

1. 遍历 `sources/<pkg>/`，读取其 `ezmk.toml` 的 `name`/`version`。
2. 归档为 `packages/<name>-<version>.tar.gz`（排除 `.ezmk/`、`build/` 等构建产物）。
3. 计算 `sha256`（可直接调用系统 `sha256sum`，或复用 ezmk 的 `crypto::sha256_file` 语义——保持与安装端一致）。
4. **重写 `index.toml`**：扫描 `packages/` 全部归档，生成对应 `[[packages]]` 条目（`name`/`version`/`file`/`sha256`）。同名多版本各生成一条（`pkg install` 取最高版本，`docs/repo.md` §集成已定义）。

> 决策：`index.toml` 由 `pack.sh` **生成**而非手写，杜绝 sha256/文件名/路径不一致。手写只保留 `[repo]` 头（`name`/`description`）。

**可选（未来）**：把 pack 能力做成 ezmk 内置 `ezmk pkg pack` 或一个官方 utils 工具，让第三方仓库维护者复用。0.9.1 暂用仓库侧脚本。

## 6 CI 校验

`.github/workflows/ci.yml` 在每个 PR 上运行 `scripts/validate.sh`，校验规则**对齐 ezmk 客户端的 `repo_validate_*` 语义**（`src/repo.cpp`），把问题挡在合入前：

- [ ] `index.toml` 可解析，`[repo]` 含 `name`
- [ ] 每个 `[[packages]].file` 指向的归档在 `packages/` 中**存在**（对应 `repo_validate_missing_file`）
- [ ] 每个 `sha256` 为 **64 位十六进制**且与归档实际哈希**匹配**（对应 `repo_validate_bad_sha256` + 内容一致性）
- [ ] `packages/` 中的每个归档都能由 `sources/` 对应工程可复现地重建（跑 `pack.sh` 后 `git diff` 为空）
- [ ] 每个 `sources/<pkg>` 能被 ezmk 成功 `pkg install`（拉取当前 ezmk release，端到端装一遍）

> 第 4、5 条让"源 → 归档 → 索引"三者强一致，避免有人只改归档不改源、或漏更新 sha256。

## 7 ezmk 侧：默认仓库接入策略

核心问题：ezmk 如何让用户不必手动记 URL 就能用默认仓库。三种方案权衡：

| 方案                              | 做法                                                                                                                     | 优点                                                     | 缺点                                                        |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------- | ----------------------------------------------------------- |
| **A 安装时预注册**（推荐）        | `install.sh` 执行 `ezmk repo add -u <official_url> --name official`（可 `EZMK_NO_DEFAULT_REPO=1` 跳过）                  | 透明（`repo list` 可见）、零改代码、用户可 `repo remove` | 仅覆盖走 `install.sh` 的用户；裸 Windows 需文档引导手动 add |
| **B 内置默认常量 + 兜底自动注册** | `repo.cpp` 内置 `DEFAULT_REPO_URL`；当所有作用域都无仓库且 `pkg install <name>` 找不到时，提示或自动 `repo add` 官方仓库 | 覆盖所有安装途径                                         | 隐式网络行为、需改代码、"自动 clone"有惊讶性，安全上需谨慎  |
| **C 便捷子命令**                  | 新增 `ezmk repo add-default` / `--official`                                                                              | 显式、可控                                               | 仍需用户主动跑一次                                          |

**推荐组合：A 为主 + C 为辅**：
- `install.sh` 默认预注册官方仓库到**用户作用域**（`-u`，避免写全局安装目录的权限问题），并首次 `repo update` 拉取索引；用 `EZMK_NO_DEFAULT_REPO=1` 可关闭。
- 提供显式的接入方式给非脚本安装用户：README/`docs/cli.md` 给出一行 `ezmk repo add -u https://github.com/3667808244/ezmk-repo.git --name official`。
- **暂不做 B**（隐式自动 clone），把"是否内置默认仓库常量 + 自动兜底"留作开放问题（§12），避免在 0.9.1 引入隐式网络行为。

> 预注册用**用户作用域**而非全局：`curl|bash` 装到 `$HOME/.local` 时用户作用域可写、无需 root，且 `repo list -u` 可见、可 `repo remove -u official` 撤销。

## 8 贡献流程

`ezmk-repo/CONTRIBUTING.md` 定义第三方提交包的流程：

1. Fork 仓库，在 `sources/<your-pkg>/` 放入完整 ezmk 工程（含 `ezmk.toml`，`type` 为 `static`/`shared`/`utils`）。
2. 本地跑 `scripts/pack.sh` 生成 `packages/<name>-<version>.tar.gz` 并更新 `index.toml`。
3. utils 包必须声明 `[utils.permissions]`（0.2.5 权限模型），PR 审阅重点检查权限是否最小化。
4. 提交 PR；CI（§6）自动校验；维护者审阅源码与权限声明后合入。
5. 版本升级 = 新增 `sources/<pkg>` 的新版本并重跑 `pack.sh`，`index.toml` 追加新 `[[packages]]`（保留旧版本条目，`pkg install` 默认取最高版本）。

**命名/版本约定**：包名唯一、小写短横线；版本用 SemVer；归档名 `<name>-<version>.tar.gz`（与 `docs/repo.md` 一致）。

## 9 安全

对齐 `docs/@safety.md`（0.9.0 §3.2 将集中化）：

| 场景                 | 措施                                                                                                                         |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| 公共仓库包完整性     | `index.toml` **必须**提供 `sha256`；CI 校验哈希匹配；ezmk 安装时强制校验（现有 `pkg install` 逻辑）                          |
| 恶意 utils 包        | 强制声明 `[utils.permissions]`；PR 人工审阅；用户安装前可 `pkg info` 预览权限                                                |
| 供应链（仓库被篡改） | 源工程 `sources/` 可审计；CI 保证"源→归档→哈希"一致；git 历史可追溯                                                          |
| 预注册的隐式性       | 预注册在**用户作用域且显式可见**（`repo list`），可 `EZMK_NO_DEFAULT_REPO=1` 关闭、可 `repo remove` 撤销——不做静默自动 clone |
| 全局安装             | 从默认仓库 `pkg install -g` 仍触发全局安装二次确认（现有机制）                                                               |

## 11 测试

- [ ] 端到端（对真实/临时官方仓库）：`ezmk repo add -u <url> --name official` → `repo update` → `repo info -u official`（显示包与版本）→ `pkg search hello-lib` → `pkg install -p hello-lib` → 在示例项目 `[depends]` 引用并 `build` 通过

## 12 开放问题

1. **是否内置默认仓库常量（方案 B）**：为覆盖所有安装途径（含裸 Windows Release），是否在 `repo.cpp` 内置 `DEFAULT_REPO_URL` 并在"无仓库且找不到包"时提示/兜底注册？需权衡隐式网络行为与开箱即用。0.9.1 先不做，收集反馈。
2. **`ezmk pkg pack` 内置命令**：把打包能力从仓库脚本提升为 ezmk 子命令，方便所有第三方仓库维护者。属新功能，宜放 0.3.x/1.x 而非 0.9.1。
3. **国内镜像**：GitHub 访问受限时的 Gitee 镜像与同步机制（CI 双推？），以及 ezmk 是否支持"同一仓库多 URL 回退"。
4. **索引规模与缓存**：仓库包数增长后 `search_package()` 每次读全量 `index.toml` 的 I/O（`plans/0.2.5.md` §8 已提及），是否需要客户端 TTL 缓存。
5. **签名**：sha256 保完整性但不防"仓库所有者本人作恶"。是否引入 GPG/minisign 对 `index.toml` 或归档签名？0.9.1 暂以 sha256 + 源可审计 + 人工审阅为准。
6. **包的许可证与元数据**：`index.toml` 是否增加 `license`/`homepage`/`description` 字段（需同步扩展 `docs/repo.md` 与 `repo.cpp` 解析）？利于 `pkg search`/`info` 展示。
